**后面会反复出现的几个词（先混个脸熟即可）**：
- **挖矿**：矿工按协议做一连串操作，向智能合约提交一份「证明」，通过后领取奖励；不是比特币那种算哈希挖矿，而是「证明我存了某块数据」。
- **nonce**：矿工自己选的一个随机数，用来参与算出「这次要去验证哪一块数据」；每次试不同的 nonce，相当于抽不同的签。
- **召回位置**：系统根据 nonce 和规则算出来的「第几号数据块」；矿工必须能当场出示那一块，相当于被「随机召回」到某个位置接受检查。
- **密封（sealing）**：用矿工独有的方式把原始数据变换一下再存；同一份数据不同矿工存的样子不同，防止直接拿别人的数据来交差。
- **Scratchpad**：按规则算出的一小块临时数据，矿工必须把「要验证的数据块」和它做一次混合再算哈希，用来把「算」和「读盘」绑在同一台机器上，防分布式挖矿。

## 关键概念

- **存储捐赠（Storage Endowment）**：用户一次性支付的永久存储费用
- **PoRA（Proof of Random Access）**：随机访问证明，证明矿工真的存储了数据
- **挖矿周期（Mining Epoch）**：一次挖矿的时间段
- **数据密封（Data Sealing）**：为每个矿工以不同方式加密数据
- **Scratchpad**：随机生成的暂存区，防止分布式挖矿
- **定价段（Pricing Segment）**：每 8 GB 数据一个段，管理奖励池

## 5.1 基本模型："一次付费，永久存储"

### 设计理念

**是什么**：用户为每个创建的数据条目支付一次性的存储捐赠（storage endowment），之后这个捐赠用于激励维护该数据条目的矿工。

**为什么这样设计**：
- 用户不需要持续支付费用
- 矿工通过维护数据获得持续奖励
- 实现"一次付费，永久存储"

**类比**：
- 就像在银行存一笔钱
- 这笔钱产生利息
- 利息支付给维护账户的银行员工
- 本金永远不动，利息持续产生

### 奖励分配机制

**关键特性**：

1. **按数据条目维护**：
   - 每个数据条目有独立的捐赠池
   - 矿工只能从他们有权访问的数据条目获得存储奖励

2. **与流行度无关**：
   - 数据条目的总存储奖励与流行度无关
   - **热门数据**：被很多矿工存储 → 频繁挖矿 → 奖励在矿工间分摊
   - **冷门数据**：很少被挖矿 → 奖励累积 → 存储冷门数据的矿工获得更高回报

**设计目的**：
- 鼓励矿工存储**稀有数据**
- 平衡数据副本数量
- 确保所有数据都有足够的副本

**具体例子**：
```
数据条目 A（热门）:
- 存储捐赠: 1000 ZG
- 存储的矿工: 100 个
- 挖矿频率: 每天 10 次
- 每次奖励: 1000 / (100 × 10) = 1 ZG/次

数据条目 B（冷门）:
- 存储捐赠: 1000 ZG
- 存储的矿工: 2 个
- 挖矿频率: 每天 1 次
- 每次奖励: 1000 / (2 × 1) = 500 ZG/次

→ 冷门数据奖励更高，鼓励存储
```

**接下来**：知道了奖励从哪来，还要知道「用户付多少钱、怎么算」——下一节讲存储请求的定价（费用 + 存储捐赠）。

## 5.2 存储请求定价

### 成本组成

**两部分**：

1. **费用（Fee）**：
   - 支付给主链矿工/验证者
   - 用于调用 0G Storage 合约处理存储请求
   - 定价与其他智能合约调用交易相同

2. **存储捐赠（Storage Endowment）**：
   - 支持对 0G Storage 矿工的永久奖励
   - 这是本节的重点

### 单位价格计算

**是什么**：单位价格 = 存储捐赠 / 数据大小

**公式**：
```
单位价格 = 存储捐赠 / 数据大小（以 256B sector 为单位）
```

**具体例子**：
```
数据大小: 1024 sectors = 256 KB
存储捐赠: 1000 ZG 代币
单位价格 = 1000 / 1024 ≈ 0.98 ZG/sector
```

### 最低价格要求

**规则**：
- 单位价格必须超过全局指定的**下限**
- 否则请求将** pending**（待处理）
  - **什么是 pending**：请求不会立即被处理，而是排队等待。系统会定期调整最低价格下限；当下限降低到你的单位价格以下时，请求才会被接受。
- 直到下限降低到单位价格以下

**用户选择**：
- 用户可以设置更高的单位价格
- 更高的价格 → 更多存储节点挖矿 → 更好的数据可用性

**市场机制**：
- 用户根据需求选择价格
- 高价值数据可以支付更高价格
- 获得更好的可靠性和可用性

**接下来**：定价讲完了，核心问题是「矿工怎么证明自己真的存了数据」——下一节详细讲 **PoRA（随机访问证明）**：从「一句话」和「六步总览」开始，再拆开每一步。

## 5.3 Proof of Random Access (PoRA)：随机访问证明

### 核心问题

**问题**：如何证明矿工真的存储了数据，而不是临时从其他地方获取？

**为什么这是问题**：
- 如果矿工不存储数据，就不需要存储成本
- 但矿工仍然可以通过临时查询获得奖励
- 这会导致数据没有足够的副本，系统不可靠

**解决方案**：随机查询 + 时间压力

**机制**：
- 要求矿工回答**随机生成**的查询
- 查询针对归档的数据块
- 矿工必须快速响应（证明数据在本地）
- 如果数据不在本地，响应会太慢

### PoRA 用一句话说

**一句话**：矿工在每轮挖矿里不断试不同的 **nonce**（可理解为「抽签号」），由 nonce 和规则算出要被检查的「第几号数据块」（**召回位置**）；矿工必须当场从本地读出那一块、和系统给的一小片临时数据（**scratchpad**）混合后算哈希；若哈希满足难度，就交「这块数据 + 证明它属于整份数据的 Merkle proof」上链，领取奖励。

读完下面「六步总览」再往下看各步细节，会轻松很多。

### 六步总览（建议先扫一眼再进细节）

| 步 | 在做什么 | 矿工/系统各干什么 |
|----|----------|-------------------|
| 1 | 注册 | 矿工在主链合约里登记自己的 ID，之后才能参与挖矿。 |
| 2 | 等周期 | 挖矿按「周期」进行，矿工等到本周期对应的主链区块高度再开始。 |
| 3 | 拿状态 | 矿工从链上读出本轮的 **挖矿状态**（区块哈希、Merkle root、数据长度、context digest 等），后面所有计算都依赖这些「不可改的上下文」。 |
| 4 | 算可挖矿块数 | 根据数据总长度算出「一共有多少个 256 KB 的块」可以参与本轮挖矿（以及自己选的 8 TB 范围内有多少块）。 |
| 5 | **迭代**（核心） | 选 nonce → 算召回位置 τ 和 scratchpad → 从本地加载偏移 τ 的那块密封数据 → 与 scratchpad 混合 → 分成 64 个 4 KB 算哈希 → 若有某个哈希满足难度则成功，否则换 nonce 重试。 |
| 6 | 提交证明 | 把 **nonce**、**中签的那块 4 KB 数据**、以及**该块对应的 Merkle proof** 交到合约；合约验证通过则发放本周期奖励。 |

**记住**：步骤 5 是循环——多数时候「试一次不中」就换 nonce 再试，直到某次命中难度为止。

### PoRA 挖矿过程：详细步骤

下面按步骤 1～6 拆开说：每一步**在整体里干什么**、**为什么需要**、以及**具体怎么做**。

#### 步骤 1：注册矿工

**是什么**：矿工首先需要在智能合约上注册自己的矿工 ID。

**怎么做**：
- 矿工调用智能合约的 `registerMiner()` 函数
- 提供矿工 ID（通常是矿工的地址）
- 合约记录矿工信息

**这一步在整体里做什么**：只有先登记身份，后续提交的 PoRA 才能和「哪个矿工」对应，奖励才能发对人。

**具体例子**：
```solidity
// 矿工注册
function registerMiner() public {
    minerId = msg.sender;
    miners[minerId] = MinerInfo(...);
}
```

**接下来**：注册完成后，矿工不能随时开挖，而要等系统规定的「挖矿周期」开始。

#### 步骤 2：等待挖矿周期开始

**是什么**：挖矿周期（mining epoch）从主链特定区块高度开始。

**为什么需要挖矿周期**：
- PoRA 会占用矿工的磁盘 I/O
- 如果持续挖矿，无法响应用户查询
- 通过挖矿周期，矿工可以在挖矿和服务用户之间切换

**怎么做**：
- 智能合约定义挖矿周期的开始区块高度
- 比如：区块高度 1000, 2000, 3000, ...
- 矿工等待到达开始区块高度

**具体例子**：
```
当前区块高度: 950
下一个挖矿周期开始: 1000
→ 矿工等待 50 个区块
```

**这一步在整体里做什么**：把全网的挖矿行为对齐到「同一轮、同一套规则」——大家用同一区块高度下的 Merkle root、数据长度等，避免有人用旧数据或错误上下文挖矿。

**接下来**：周期到了之后，首先要从链上拿到本轮的「挖矿状态」，后面所有计算都基于这份状态。

#### 步骤 3：获取挖矿状态

**是什么**：矿工从主链读取当前的挖矿状态。

**挖矿状态包含什么**：
- 当前区块的哈希值（block hash）
- Merkle root（所有数据的 Merkle root）
- 数据总长度（data length）
- 上下文摘要（**context digest**）
  - **可以理解为**：「本轮规则的指纹」——把区块哈希、Merkle root、数据长度等一起做哈希得到的一个摘要。后面算召回位置、scratchpad、密封都会用到它，保证所有人用同一套不可篡改的上下文，防止有人用旧数据或预计算作弊。

**怎么做**：
- 矿工查询智能合约，获取当前挖矿状态
- 这些信息用于后续计算

**具体例子**：
```
挖矿状态:
- blockHash: 0xabc123...
- merkleRoot: 0xdef456...
- dataLength: 1000000 sectors (250 GB)
- contextDigest: 0x789abc...
```

**这一步在整体里做什么**：为后续「抽签」和「验证」提供不可篡改的公共输入——谁算召回位置、谁验证明，用的都是同一份 context digest、Merkle root、数据长度，保证公平。

**接下来**：有了状态里的数据长度（和自选的挖矿范围），就能算出「这一轮里一共有多少块可以参与挖矿」。

#### 步骤 4：计算可挖矿条目数

**是什么**：计算有多少个可挖矿的数据块。

**公式**：
```
n = ⌊dataLength / 256KB⌋
```

**为什么除以 256KB**：
- 每个 chunk 是 256 KB
- 只能对完整的 chunk 进行挖矿

**具体例子**：
```
dataLength: 1000000 sectors = 250 GB
256 KB = 1024 sectors
n = 1000000 / 1024 = 976 chunks
```

**这一步在整体里做什么**：确定「分母」——后面算召回位置时用「第几块」是落在 1～n 之间的，n 就是这里算出来的可挖矿块数（在自选 8 TB 范围内）。

**接下来**：块数确定后，就进入真正的「抽签 + 验证」循环：不断试 nonce，直到某次算出的那块刚好满足难度。

#### 步骤 5：PoRA 迭代（核心过程）

**是什么**：矿工重复执行 PoRA 迭代，直到找到有效的证明。可以理解为：每次迭代 = 抽一次签（选 nonce）→ 算出要检查哪一块（召回位置）→ 读盘、混合、算哈希 → 看有没有「中奖」（满足难度）；不中就换 nonce 再抽一次。

**这一步在整体里做什么**：整个 PoRA 的「证明我存了」都发生在这里——系统用 nonce 随机点名一块，矿工必须能当场拿出那块并完成混合与哈希；拿不出或算不出，就得不到有效证明。

**每次迭代包含以下子步骤**（5.1～5.7 按顺序做一遍，相当于「抽一次签」）：

##### 步骤 5.1：选择随机 nonce

**是什么**：矿工选择一个随机的 32 字节 **nonce**。可以理解为「本轮的抽签号」：后面会用它确定性算出「抽查哪一块」（召回位置）和用于混合的 scratchpad；每次试不同 nonce，相当于抽不同的签。

**为什么需要**：
- nonce 用来算出**随机召回位置**（要检查的第几号块）
- 不同 nonce → 不同块 → 增加随机性，无法预测会被点到哪一块

**怎么做**：
- 矿工生成一个随机 32 字节数
- 比如：`0x1234567890abcdef...`

**与下一步的关系**：这个 nonce 会参与算出「召回位置」和「scratchpad」——不同 nonce，被点名的块不同，相当于每次试不同的签。

##### 步骤 5.2：决定挖矿范围

**是什么**：矿工决定在哪个数据范围内挖矿。

**规则**：
- 挖矿范围限制为 **8 TB 阈值**
- 如果数据总长度超过 8 TB，矿工必须指定 8 TB 的范围
- 如果数据总长度小于 8 TB，使用全部数据

**公式**：
```
mineLength = min(8TB, n × 256KB)
startPosition = 矿工选择的起始位置
```

**为什么限制 8 TB**：
- 保证小矿工的公平性
- 如果数据太大，小矿工无法存储全部数据
- 限制范围后，小矿工可以选择 8 TB 范围，公平竞争

**具体例子**：
```
数据总长度: 20 TB
挖矿范围限制: 8 TB
→ 矿工可以选择 offset 0-8TB, 或 8TB-16TB, 或 16TB-20TB
```

**与下一步的关系**：有了「从哪开始、多长范围」（start position, mine length），下一步就能在这个范围内用 nonce 算出「具体抽查哪一块」（召回位置 τ）和用于混合的 scratchpad。

##### 步骤 5.3：计算召回位置和 Scratchpad

**是什么**：用上一步的 nonce 和挖矿状态，**确定性地**算出两样东西：**召回位置 τ**（要检查的是第几块）和 **scratchpad**（一块 256 KB 的临时数据，后面要和「那一块」做混合）。

**召回位置（recall position）是什么**：
- 数据流里的偏移量（offset），单位是 256 KB（一个 chunk）
- 表示「本轮要抽查的是第 τ 块」——矿工必须能当场从本地读出这一块。可以理解为：系统随机「召回」你到第 τ 号位接受检查，拿得出才算真的存了。

**Scratchpad 是什么**：
- 由 nonce 和上下文**确定性地**算出的 256 KB 数据（对所有人规则相同，但不知道 nonce 就算不出）
- 后面步骤里，矿工必须把「第 τ 块」和这块 scratchpad 做 XOR 再算哈希。可以理解为：把「读盘」和「算哈希」绑在同一台机器上——若把计算放另一台机器，就要传 256 KB scratchpad，网络成为瓶颈，见 5.4.3 防分布式挖矿。

**怎么做**：

**算法（Figure 6）**：

**关于 Blake2b**：Blake2b 是一种密码学哈希函数（类似 SHA-256），速度快、安全性高，常用于 PoRA 中的确定性计算。

```
输入: miner id, nonce, context digest, start position, mine length
输出: Recall position τ 和 scratchpad s

1. h ← Blake2b(miner id, nonce, context digest, start position, mine length)
2. 初始化空数组 s，包含 4096 个 64 字节元素（总共 256 KB）
3. 对 i 从 0 到 4095:
   - h ← Blake2b(h)
   - s[i] ← h
4. r ← Keccak256(h)
   - **关于 Keccak256**：Keccak256 是另一种密码学哈希函数（以太坊也使用），用于从 h 推导出随机数 r，进而决定召回位置 τ。
5. n ← mine length / 256KB
6. τ ← start position + (r mod n) × 256KB
```

**具体例子**：
```
输入:
- miner id: 0x1234...
- nonce: 0xabcd...
- context digest: 0xef01...
- start position: 0
- mine length: 8 TB

计算:
1. h = Blake2b(...) = 0x2345...
2. 生成 scratchpad s (256 KB)
3. r = Keccak256(h) = 0x5678...
4. n = 8TB / 256KB = 32768
5. τ = 0 + (0x5678... mod 32768) × 256KB = offset 5000 × 256KB

结果:
- 召回位置: offset 5000 × 256KB
- Scratchpad: 256 KB 的随机数据
```

**与下一步的关系**：上一步得到了**召回位置 τ** 和 **scratchpad s**；接下来矿工要用 τ 从本地读出「第 τ 块」的密封数据，并在后面的步骤里把这块数据和 s 做混合。

##### 步骤 5.4：加载密封数据块

**是什么**：用上一步算出的**召回位置 τ**，从本地存储中加载「第 τ 块」的 256 KB 数据——这块数据必须是**密封后的**（sealed），不能是原始数据。

**密封（sealing）是什么**：
- 用矿工自己的方式（由矿工 ID + 上下文推导）对原始数据做确定性变换，同一份数据不同矿工存出来的样子不同。可以理解为：每个矿工有一把自己的「封印」，只有自己能在本地高效解封、参与 PoRA；直接拿别人的块来用，对不上自己的封印，无法通过验证。
- **为什么能防外包**：不存数据、只在外网查的话，每次被点到都要现做密封；密封是重计算，比从本地 SSD 读慢得多，经济上更划算的是本地存好并密封。

**怎么做**：
- 矿工从本地存储加载 offset τ 位置的 256 KB 数据块
- 这个数据块是"密封"的（sealed data chunk）
- 如果数据不在本地，需要从网络下载并密封（成本高）

**具体例子**：
```
召回位置: offset 5000 × 256KB
→ 加载 chunk_5000.bin (256 KB, 已密封)
→ 如果不在本地，需要下载并密封（慢且贵）
```

**与下一步的关系**：已经拿到了「第 τ 块」的 256 KB 密封数据；下一步必须把这块数据和步骤 5.3 算出的 **scratchpad** 做 XOR，再对结果算哈希——这样「读盘」和「算哈希」就绑在同一台机器上（见 5.4 节防分布式挖矿）。

##### 步骤 5.5：与 Scratchpad 混合

**是什么**：将加载的数据块与 scratchpad 进行 XOR 操作。

**为什么需要**：
- 防止分布式挖矿
- 如果计算和加载分离在不同机器，需要传输 scratchpad（256 KB）
- 网络带宽通常小于存储 I/O 带宽
- 因此，计算和加载必须在同一机器

**怎么做**：
```
w = sealed_data_chunk XOR scratchpad
```

**具体例子**：
```
sealed_data_chunk: 256 KB
scratchpad: 256 KB
w = sealed_data_chunk XOR scratchpad  → 256 KB
```

**与下一步的关系**：混合后的 256 KB 数据 w 已经准备好；下一步把它切成 64 段各 4 KB，分别算哈希，然后看有没有哪一段的哈希「中奖」（满足难度）——若有，就只把**那一块 4 KB** 和对应 Merkle proof 交上链，省掉 64 倍的上链体积。

##### 步骤 5.6：分成 64 个 4 KB 块并计算哈希

**是什么**：将混合后的数据分成 64 个 4 KB 块，分别计算哈希。

**为什么分成 64 块**：
- 如果直接提交 256 KB 到链上，交易费用太高
- 分成小块后，只提交匹配的 4 KB 块
- 降低交易费用

**怎么做**：
```
将 w (256 KB) 分成 64 个 4 KB 块: v[0], v[1], ..., v[63]

对每个块 v[i]:
  hash[i] = Blake2b(miner id, nonce, context digest, start position, mine length, v[i])
```

**具体例子**：
```
w (256 KB) → 64 个 4 KB 块
v[0] (4 KB) → hash[0] = Blake2b(...)
v[1] (4 KB) → hash[1] = Blake2b(...)
...
v[63] (4 KB) → hash[63] = Blake2b(...)
```

##### 步骤 5.7：检查是否满足难度

**是什么**：检查是否有哈希值满足目标难度。

**目标难度是什么**：
- 哈希值需要有足够的前导零（或数值小于某个目标）
- 比如：要求 hash 小于某个数，等价于哈希的二进制形式里前若干位为 0
- 难度越高，目标越小，满足条件的 hash 越难找到
- **类比**：类似比特币挖矿的"算难题"，这里是用"随机试 nonce + 检查 hash 是否达标"来代替纯粹算力

**怎么做**：
- 检查每个 hash[i] 是否小于目标值
- 如果有一个满足，找到有效 PoRA
- 如果没有，回到步骤 5.1，选择新的 nonce

**具体例子**：
```
目标难度: 前 16 位都是 0
hash[0] = 0x00001234...  → 满足！
hash[1] = 0x0001abcd...  → 不满足
...
hash[63] = 0x0002ef01...  → 不满足

结果: 找到有效 PoRA（使用 v[0]）
```

**与下一步的关系**：某次迭代里有一个 hash[i] 满足了难度，矿工就「中签」了；下一步要把这次迭代用到的 **nonce**、中签的那块 **4 KB 数据 v[i]**、以及**证明 v[i] 属于整份数据的 Merkle proof** 一起提交到合约，领奖。

#### 步骤 6：提交 PoRA 证明

**是什么**：矿工把「中签」那次迭代用到的 nonce、命中的那块 4 KB 数据、以及证明该块属于整份数据的 Merkle proof，一起提交到主链合约。

**这一步在整体里做什么**：链上只验「这块 4 KB + Merkle proof 能否拼回链上的 Merkle root」，不存整份数据；验证通过就把本挖矿周期的奖励发给该矿工，本轮结束。

**矿工到底交什么？——是「数据块 + Merkle proof」，不是「文件 + 哈希」**

- 矿工向智能合约提交的三样东西是：**nonce**、**匹配的那一块 4 KB 数据**（v[i]）、以及**该块对应的 Merkle proof**。
- **不是**「文件 + 哈希」：链上只有整份数据的 **Merkle root**，没有整份文件；合约要验证的是「你交的这块 4 KB 确实属于当初承诺的那份数据」。要完成这件事，必须用 **Merkle proof** 把「这一块」和「链上已有的 root」连起来——即证明「把这块算进去、再按 proof 的路径往上算，能得到链上的 root」。
- 只交「一块数据」或只交「这块的哈希」都不够：合约无法仅凭一个哈希判断它属于哪份数据、在 Merkle 树里的位置；必须用 **Merkle proof**（一路的兄弟节点哈希）才能从「这一块」推回 root 并与链上 root 比对。所以挖矿提交的是**数据块 + Merkle proof**。

**证明包含什么**：
- 选择的随机 nonce
- 匹配的 4 KB 数据块（v[i]）
- 该数据块正确性的证明（**Merkle proof**，用于证明该块属于链上 Merkle root 所承诺的数据）

**怎么做**：
- 矿工调用智能合约的 `submitPoRA()` 函数
- 提供 nonce、4 KB 数据块、**Merkle proof**（不是「该块的哈希」）
- 合约用「数据块 + Merkle proof + 链上 Merkle root」验证：由块算出叶哈希，再沿 proof 推到 root，与链上 root 一致则通过

**具体例子**：
```solidity
function submitPoRA(
    bytes32 nonce,
    bytes calldata dataChunk,  // 4 KB
    bytes32[] calldata merkleProof
) public {
    // 验证证明...
    // 如果有效，分配奖励
}
```

### 间歇挖矿

**是什么**：挖矿周期从开始到第一个有效 PoRA 提交时结束。

**为什么需要**：
- PoRA 会占用磁盘 I/O
- 如果持续挖矿，无法响应用户查询
- 通过间歇挖矿，矿工可以在挖矿和服务用户之间切换

**怎么做**：
- 挖矿周期开始：主链特定区块高度
- 挖矿周期结束：第一个有效 PoRA 提交
- 矿工可以在挖矿和服务用户之间切换

**小结**：上面是 PoRA 的**主流程**（注册 → 等周期 → 拿状态 → 算块数 → 迭代抽签+验证 → 提交证明）。下面 5.4 节说明为了**公平、防作弊**做的设计：密封、scratchpad、8 TB 限制、只交 4 KB 等，都是在保证「真的在存、真的在这台机器上算」。

## 5.4 PoRA 的公平性设计

**本节在讲什么**：5.3 已经把 PoRA 的六步流程说完了；这一节专门讲为了**公平、防作弊**做的几项设计：小矿工公平（8 TB 限制）、防外包（密封）、防分布式挖矿（scratchpad）、省链上费用（只交 4 KB）。读的时候可以对照 5.3 里「为什么需要密封」「为什么有 scratchpad」等，这里展开原理与效果。

### 5.4.1 小矿工的公平性

**问题**：
- 当存储网络大小显著超过单机容量时
- 单机几乎把所有时间都花在寻找可用的召回位置上
- PoRA 变成了工作量证明（Proof of Work）

**解决方案**：限制挖矿范围

**规则**：
- 挖矿范围限制为 **8 TB 阈值**
- 当归档数据块大小超过 8 TB 时
- 矿工必须指定数据流序列中 8 TB 大小的挖矿范围

**影响**：
- **小矿工**：可以选择 8 TB 范围，公平竞争
- **大矿工**：有足够机器存储所有数据，可以在不同数据范围上并发挖矿

**具体例子**：
```
数据总长度: 20 TB
小矿工 A: 选择 offset 0-8TB 范围，存储这 8 TB 数据
大矿工 B: 存储全部 20 TB，可以在多个 8 TB 范围上并发挖矿
→ 小矿工和大矿工在单个 8 TB 范围内公平竞争
```

### 5.4.2 防止存储外包

**问题**：
- 矿工可能不存储数据，而是从其他地方查询
- 这样矿工不需要存储成本，但仍能获得奖励

**解决方案**：数据密封（Data Sealing）

**是什么**：PoRA 机制为每个矿工以不同方式"密封"数据。

**为什么有效**：
- 如果矿工外包存储并在 PoRA 挖矿中查询数据
- 必须**密封**回答的数据来计算输出摘要
- 密封是**大成本**操作

**密封算法（Figure 5）**：

**什么是"密封"（Sealing）**：
- 用矿工专有的密钥（由矿工 ID + 上下文推导）对数据做确定性变换
- 同一份原始数据，不同矿工得到的"密封数据"不同
- **为什么能防外包**：若矿工不存数据、只在外网查，每次 PoRA 挑战都要现做密封。密封是重计算，且要处理大量数据，比直接从本地 SSD 读慢得多，因此经济上更划算的是本地存储。

**输入**：
- seal seed：32 字节，矿工 ID 和上下文数据的摘要
- unsealed data：4 KB 未密封数据

**输出**：4 KB 密封数据

**算法**：
```
1. 将 4 KB 未密封数据视为 32 字节元素的数组 d（128 个元素）
2. h ← seal seed
3. 对 i 从 0 到 127:
   - d[i] ← d[i] XOR h
   - h ← Keccak256(d[i])
4. 将 d 视为 4 KB 密封数据块
```

**具体例子**：
```
输入:
- seal seed: 0x1234... (32 字节)
- unsealed data: 4 KB

处理:
d[0] = d[0] XOR 0x1234...
h = Keccak256(d[0])
d[1] = d[1] XOR h
h = Keccak256(d[1])
...
d[127] = d[127] XOR h

输出: 4 KB 密封数据
```

**成本对比**：
- **存储数据**：购买磁盘 + 同步数据（一次性成本，比如 $100）
- **外包 + 密封**：每次挖矿都需要密封（持续成本，比如每次 $1）
- 如果每天挖矿 10 次，一年需要 $3650
- 只要密封成本超过存储成本，矿工倾向于存储数据

**性能数据**：
- 从 SSD 读取数据：可达 **7 GB/s**
- 数据密封：重任务，会显著降低挖矿速度
- 矿工可以单线程密封和解封数十 MB 数据块
- 仍能高效服务用户查询

### 5.4.3 防止分布式挖矿

**问题**：
- 挖矿农场可能使用分布式系统 + 内存文件系统
- DDR 内存带宽远大于 SSD
- 可以产生更多算力，但对数据存储贡献很小

**解决方案**：强制计算和加载在同一机器

**机制**：
- 在计算阶段生成随机 scratchpad（256 KB）
- 在计算加载数据块的哈希之前，数据块必须与 scratchpad 进行 XOR 操作
- 如果计算和加载分离在不同机器，需要传输 scratchpad（256 KB）
- 网络带宽通常小于存储 I/O 带宽

**工作原理**：

**场景 1：计算和加载在同一机器（推荐）**
```
计算阶段: 生成 scratchpad (256 KB) → 内存
加载阶段: 从 SSD 加载数据块 (256 KB) → 内存
混合: 数据块 XOR scratchpad → 内存
→ 全部在内存中，速度快（7 GB/s）
```

**场景 2：计算和加载分离在不同机器（不推荐）**
```
计算机器: 生成 scratchpad (256 KB)
↓ 网络传输 (256 KB)
加载机器: 接收 scratchpad
加载机器: 从 SSD 加载数据块 (256 KB)
混合: 数据块 XOR scratchpad
→ 受网络带宽限制（通常 < 1 GB/s，远小于 7 GB/s）
```

**结果**：
- 如果分离，算力受网络带宽限制
- 如果同一机器，可以利用快速 SSD 读取速度
- 因此，矿工倾向于在同一机器上完成计算和加载

### 5.4.4 批量数据加载

**问题**：
- 随机加载 256 KB 数据块达到顺序访问的 **80% 读取速度**
- 但提交 256 KB 交易到主链会消耗大量交易费用

**解决方案**：批量数据加载（Batched Data Loading）

**机制**：
- 每次迭代加载 256 KB 数据块
- 分成 64 个 4 KB 数据块
- 分别计算每个 4 KB 块的哈希
- 如果其中一个匹配目标质量 → 找到有效答案
- **只提交**贡献答案的 4 KB 数据块到主链

**具体例子**：
```
加载: 256 KB 数据块
↓ 分成 64 块
v[0] (4 KB) → hash[0]
v[1] (4 KB) → hash[1]
...
v[63] (4 KB) → hash[63]

检查: hash[5] 满足难度
→ 只提交 v[5] (4 KB) 到链上
→ 交易费用降低 64 倍
```

**优势**：
- 保持 256 KB 的读取效率（80% 顺序访问速度）
- 只提交 4 KB 到链上（降低交易费用）

**接下来**：防作弊设计讲完了；矿工通过 PoRA 拿到奖励，那「奖励从哪来、怎么算、什么时候发」——下一节讲挖矿奖励（定价段、奖励池、R_base + R_storage）。

## 5.5 挖矿奖励

### 奖励池设计

**定价段（Pricing Segment）**：
- **注意**：这里的"定价段"是按**数据量**划分的（每 8 GB 一块），和第四章按"存储单价"划分的定价段（Pricing Tier）是不同概念。
- 每 8 GB 数据创建一个定价段（Segment）
- 每个定价段关联：
  - **捐赠池（Endowment Pool）**：收集该段内所有数据条目的存储捐赠
  - **奖励池（Reward Pool）**：接收从捐赠池按比例释放的奖励

**奖励释放**：
- 捐赠池以固定比率向奖励池释放余额
- 释放速率：**4% 每年**
- 每秒释放
- **为什么是 4%**：设计成"永久释放、缓慢释放"，既让矿工有长期激励，又避免捐赠池很快被掏空，保证数据能一直被维护。

**具体例子**：
```
定价段 1 (0-8 GB):
- 捐赠池: 10000 ZG
- 奖励池: 400 ZG/年 = 0.0127 ZG/秒

定价段 2 (8-16 GB):
- 捐赠池: 5000 ZG
- 奖励池: 200 ZG/年 = 0.0063 ZG/秒
```

### 奖励组成

**两部分**：

1. **基础奖励（Base Reward, R_base）**：
   - 为每个接受的挖矿证明支付
   - 每个证明的基础奖励随时间递减
   - 鼓励早期参与

2. **存储奖励（Storage Reward, R_storage）**：
   - 来自存储数据的永久奖励
   - 当 PoRA 落在某个定价段时
   - 该段奖励池余额的**一半**被认领为存储奖励

**总奖励公式**：
```
R_total = R_base + R_storage
```

**具体例子**：
```
矿工提交 PoRA，落在定价段 1:
- R_base: 10 ZG（基础奖励）
- R_storage: 奖励池余额的一半 = 200 ZG
- R_total = 10 + 200 = 210 ZG
```

### 奖励分配时机

**条件**：
- 矿工提交第一个合法的 PoRA 到 0G Storage 合约
- 获得该挖矿周期的奖励

**机制**：
- 每个挖矿周期只有一个矿工获得奖励
- 第一个提交有效 PoRA 的矿工获胜
- 鼓励矿工快速响应

**接下来**：奖励怎么发已经清楚；那「数据副本太多或太少怎么办」——下面 5.6 讲如何用声明和激励**平衡副本数量**。

## 5.6 如何平衡副本数量？

### 问题：如何确保数据有足够的副本？

**挑战**：
- 热门数据可能被过度复制（浪费）
- 冷门数据可能副本不足（风险）

### 解决方案：透明声明 + 经济激励

**机制**：

1. **透明声明**：
   - 每个矿工必须明确声明维护的数据条目范围
   - 只能使用声明的数据挖矿
   - 不能从未声明的数据获得奖励
   - 如果无法回答声明范围内的挑战 → 挖矿输出减少

2. **结果**：
   - 矿工没有动机谎报数据范围
   - 可以轻松观察和估计每个数据片段的副本数量

3. **经济激励**：
   - 存储捐赠支付给维护对应数据的矿工
   - 矿工被激励存储**稀有数据**（奖励累积）
   - 自动平衡副本数量

**市场机制**：
- 热门数据：很多矿工存储 → 奖励分摊 → 吸引力降低
- 冷门数据：很少矿工存储 → 奖励累积 → 吸引力增加
- 最终达到平衡

**接下来**：副本数量可以靠激励自动平衡；但「存了数据的人愿不愿意分享给新节点」是另一件事——下面 5.7 讲 **Royalty（版税）** 机制，用「分享才分红」来鼓励数据分享。

## 5.7 如何激励数据分享？

### 问题：存储容易激励，分享难以激励

**为什么分享难以激励？**
- 数据分享发生在共识之外（链下）
- 难以验证分享是否发生
- 理性节点倾向于不分享（节省带宽和电力）

### 0G 的解决方案：Royalty 机制（版税机制）

**核心思想**：数据提供者只有在数据**真正被分享**后才获得奖励。

**什么是 Royalty（版税）**：在这里指"数据被他人使用时，原始存储者按比例获得的分成"。类比：音乐人把歌存到平台，别人下载或使用这首歌时，音乐人收到版税。

**工作原理**：

1. **创建版税记录**：
   - 数据请求者向 0G Storage 合约发送请求
   - 创建版税记录，包含：
     - 请求的数据条目范围
     - 指定的数据提供者
     - 版税期限长度

2. **数据提供者决定**：
   - 数据提供者观察链上的新请求
   - 自行决定是否发送请求的数据给请求者

3. **奖励分配**：
   - 当请求者提交挖矿证明 π 时
   - 合约检查证明中的每个 PoRA 挑战查询 Qi
   - 如果 Qi 落在某个有效版税记录的范围内
   - 数据提供者获得存储奖励的**一定比例**作为分红

4. **记录过期**：
   - 过期的版税记录会被移除

### 为什么 Royalty 机制有效？

**关键洞察**：切换到可观察的事件

**问题**：
- 链下分享行为难以验证
- 第三方难以验证节点 A 是否通过链下通道发送数据给节点 B
- 两个节点都可能撒谎

**解决方案**：
- 不直接验证分享行为
- 验证**分享的结果**：基于分享数据生成的挖矿证明
- 如果数据没有分享 → 请求者无法在挖矿证明中使用该数据 → 没有奖励
- 如果数据分享了 → 请求者可能使用该数据 → 提供者获得分红

**经济激励**：
- 数据提供者：有明确动机分享数据（获得分红）
- 数据请求者：即使扣除分红，使用分享数据仍有正收益
- 新节点：更容易启动（可以请求数据）

## 总结：0G 激励机制的优势

1. **一次付费，永久存储**：用户只需支付一次，数据永久可用
2. **公平竞争**：小矿工和大矿工都有公平机会
3. **防止作弊**：通过密封和 scratchpad 防止各种作弊行为
4. **自动平衡**：经济机制自动平衡数据副本数量
5. **鼓励分享**：Royalty 机制鼓励节点分享数据，帮助新节点启动

这些机制共同确保了 0G Storage 网络能够可靠、高效、公平地运行。
