# 2. 0G 的核心设计理念：如何实现无限可扩展？

## 本章学习目标

在阅读本章之前，假设你已经了解：
- 以太坊的基本运作原理（区块、交易、智能合约）
- Layer 2 的基本概念（为什么需要 Layer 2，如何与 Layer 1 交互）

本章将详细解释：
- **是什么**：0G 系统的各个组件是什么
- **为什么**：为什么需要这些组件
- **怎么做**：这些组件如何协同工作，实现数据可用性

## 本章先弄清三个概念

### 什么是「数据条目」？

**数据条目（Data Entry）** = 用户**一次存储请求**在系统中对应的那条记录。

- 一次上传（比如你提交的 1 MB）→ 对应**一个**数据条目
- 每条数据条目有：ID、Merkle root、大小、存储捐赠、时间戳、在日志中的位置（offset）等
- 数据块（chunk）是**一次上传**被擦除编码、分散存储后的单位；数据条目是「这次上传」在链上/日志里的**整体**代表

**小结**：1 个用户上传 = 1 个数据条目；数据块是该条目在存储与验证时的划分单位。

### 智能合约在什么链上？

本章（以及本教程中）提到的「智能合约」「0G Storage 合约」，如无特别说明，都指部署在**主链（Host Chain）**上的 0G Storage 智能合约。

- **主链**：0G 选用的、用于部署这份合约的区块链，常见的是**以太坊**
- 用户发起的「提交存储请求」「查数据条目」等交易，都是发到**这条主链**上、由该链的节点执行合约并收 gas
- 若 0G 支持多主链，则会在具体产品中说明「当前使用的链」；本章为简化表述，统一用「主链」或「以太坊」代指

### 「顺序」指什么顺序？

**顺序**指的是**数据条目的顺序**，即**每一整次上传**在全局日志里谁先谁后，而不是「单个数据块在一条数据内部的顺序」。

- 数据条目 1、2、3… 在**逻辑上**按时间先后「追加到日志里」，每条占一段连续的 offset（**「日志」是逻辑顺序与编址，不对应一块物理上连续存好所有数据的存储**；真实数据已按块打散到各节点）。
- 每个数据条目内部会被切成多个数据块（chunk）做存储和验证；块与块之间的相对顺序由「属于哪个数据条目 + 在该条目内的偏移」唯一确定，但**共识与日志里记录的「顺序」是条目维度的**：先到的是条目 A，后到的是条目 B

**小结**：链上/日志里说的「顺序」= 各次上传（数据条目）的先后；数据块顺序是每个条目内部的实现细节。所谓「日志」= 逻辑上的一整条按条目排序、按 offset 编址的存储视图，不是一块真实存在的、整条数据连续存放的物理文件。

## 2.1 从数据上传到存储：完整流程详解

让我们从一个最简单的场景开始：**用户想要在 0G 上存储一份数据**。

### 场景设定

假设你是一个 Layer 2 网络的开发者，需要存储 1 MB 的交易数据。让我们一步步看 0G 如何处理这个请求。

### 第一步：用户准备并上传数据

#### 用户需要做什么？

1. **准备数据**：你有 1 MB 的原始数据（比如 Layer 2 的交易数据）
2. **选择存储参数**：
   - 决定支付多少存储费用（存储捐赠）
   - 选择数据流类型
     - **主流（Main Stream）**：所有用户的数据按顺序追加到同一个日志中，适合大多数场景
     - **专用流（Dedicated Stream）**：为特定应用创建独立的日志，适合需要独立管理数据的应用（详细内容在第四章讲解）
3. **调用智能合约**：向**主链**（通常是以太坊）上的 0G Storage 智能合约发送交易

#### 0G 系统在后台做什么？

**步骤 1.1：擦除编码（Erasure Coding）**

**是什么**：擦除编码是一种数据保护技术，将原始数据分割成多个块，即使部分块丢失，也能恢复完整数据。

**为什么需要**：
- 如果数据只存储在一个地方，一旦那个节点故障，数据就丢失了
- 通过擦除编码，即使部分存储节点故障，数据仍然可以恢复
- 这提高了数据的可靠性和容错性

**怎么做**：
- 假设你的 1 MB 数据经过擦除编码后，被分割成 10 个数据块（chunks）
- 每个块 128 KB
- 即使其中 3 个块丢失，仍然可以从剩余的 7 个块恢复原始数据
- 0G 使用 GPU 加速这个过程，提高处理速度

**具体例子**：
```
原始数据：1 MB
↓ 擦除编码
数据块 1: 128 KB
数据块 2: 128 KB
...
数据块 10: 128 KB
```

---

**擦除编码从算法上为什么能「丢一部分块仍可恢复」？**

下面从原理上说明：为什么不需要 10 块全在，只要「够多块」（例如 7 块）就能还原整份数据。

**1. 参数 (n, k) 表示什么**

- 设原始数据被切分成 **k 个「数据块」**（真正承载信息的部分）。
- 擦除编码会再生成 **(n − k) 个「校验块」**（由这 k 块算出来的冗余），一共得到 **n 个块**。
- 编码方式会保证：**任意 k 个块**（可以是 k 个数据块，或一部分数据块 + 一部分校验块）都**足以唯一还原**原来的 k 个数据块，从而还原整份数据。

例如 **(n=10, k=7)**：  
原始等价于 7 个「逻辑数据块」，编码成 10 块（7 数据 + 3 校验）。任意 7 块就能恢复，所以**最多可以丢 3 块**。

**2. 一类常见做法：看成「多项式 + 多点取值」**

- 把 k 个数据块（或每块再细成若干「符号」）视作 k 个系数，决定一个**次数为 k−1 的多项式 P**。
- 在 n 个不同的「位置」上算 P 的值，得到 n 个数，就对应 n 个编码块（数据块 + 校验块都在这 n 个值里）。
- **数学结论**：次数 k−1 的多项式由**任意 k 个点**唯一决定（ Lagrange 插值）。  
  → 所以**任意 k 个编码块**（即 k 个「位置 + 取值」）都能反推出 P，再推出 k 个系数，即原来的 k 个数据块。  
  → 这就是「任意 k 块可恢复」的来源。

**3. 再抽象一层：线性代数视角**

- 把 k 个数据块排成一个向量 **d**，编码看成 **c = G · d**，其中 **G** 是 k×n 的「生成矩阵」。
- n 个编码块就是 **c** 的 n 个分量；每一块都是 d 的若干线性组合。
- 若 G 设计成「任意 k 列都线性无关」（即任意 k 列组成的 k×k 子矩阵可逆），则：  
  任意拿到 k 个块，就得到 k 个线性方程、k 个未知数（d），且系数矩阵可逆 → **d 唯一可解**，即原始数据可恢复。

以上两种说法（多项式插值 / 线性方程组）本质是一回事：都是通过「冗余的线性结构」，让**任意 k 个编码块**都带有「完整解出 k 个未知数」的信息。

**4. 直观类比**

- 7 个未知数，你设计出 10 个线性方程，且保证「任意 7 个方程都线性无关、可解出那 7 个未知数」。
- 那么对方程组来说，丢掉 3 个方程没关系，剩下任意 7 个仍能解出那 7 个未知数。  
擦除编码里的「10 块中任意 7 块可恢复」，对应的就是这类「10 个方程里任意 7 个都足够」的性质。

**5. 和 0G 里「1 MB → 10 块、丢 3 仍可恢复」的对应**

- 1 MB 先按某种方式看成 k 个「逻辑数据块」（例如 k=7），再通过上述编码得到 n=10 个块（含数据块 + 校验块）。
- 协议里说「10 块中只要有不少于 7 块正确即可恢复」，对应的就是 **(n=10, k=7)** 这类参数：**任意 7 块**在算法上都足以解出原来的 k 个逻辑块，从而还原整份 1 MB。

**小结**：擦除编码在算法上保证的是——**由编码方式本身**，让「任意 k 个编码块」都含有足以唯一还原 k 个数据块的信息；因此「丢 n−k 块」不会破坏可恢复性，这正是「部分块丢失仍可恢复」的来源。

---

**步骤 1.2：生成数据承诺（Commitment）**

**是什么**：数据承诺是一个密码学哈希值（Merkle root），用来证明数据的完整性和存在性，而不需要传输完整数据。

**为什么需要**：
- 智能合约需要知道"数据确实存在且没有被篡改"
- 但智能合约不能存储大量数据（太贵）
- 所以只存储一个小的哈希值作为"承诺"

**怎么做**：
- 0G 将所有数据块组织成 Merkle 树
- 计算 Merkle root（一个 32 字节的哈希值）
- 这个 Merkle root 就是数据的"承诺"

**具体例子**：
```
数据块 1 ──┐
数据块 2 ──┤
数据块 3 ──┼── Merkle Tree ──> Merkle Root (32 bytes)
数据块 4 ──┤
...        │
数据块 10 ─┘
```

**步骤 1.3：提交到智能合约**

**是什么**：用户向部署在以太坊（或其他主链）上的 0G Storage 智能合约发送交易。

**为什么需要**：
- 智能合约负责记录数据的顺序和承诺
- 保证数据不会被篡改或丢失
- 管理存储费用和奖励分配

**怎么做**：
- 用户构造一个交易，包含：
  - 数据大小（1 MB = 4096 sectors，每个 sector 256 字节）
    - **什么是 sector**：sector 是 0G 存储的最小单位，每个 sector 是 256 字节。1 MB = 1024 KB = 1024 × 1024 字节 = 1,048,576 字节。除以 256 = 4096 sectors。这样设计是为了方便计算存储费用和奖励。
  - Merkle root（数据承诺）
  - 存储捐赠金额（比如 1000 ZG 代币）
    - **为什么叫"存储捐赠"而不是"存储费用"**：这笔钱不是一次性支付给某个服务商，而是存入一个"捐赠池"（endowment pool），用于长期奖励存储节点。存储节点通过挖矿（PoRA）从这个池子中获得奖励。这是一种可持续的经济模型，确保数据能够长期被存储。
  - 支付给主链的 gas 费用
- 交易被提交到主链（如以太坊）
- 智能合约验证交易并创建数据条目

**具体例子**：
```solidity
// 用户发送的交易（简化版）
{
  dataSize: 4096 sectors,
  merkleRoot: "0xabc123...",
  storageEndowment: 1000 ZG,
  gasFee: 0.001 ETH
}
```

### 第二步：智能合约处理请求

这里的「智能合约」指**主链**（如以太坊）上的 0G Storage 合约：用户发起的存储请求会作为一笔交易上主链，由该链执行合约逻辑。

#### 智能合约存储什么？不存储什么？

**智能合约存储的内容**（很小，适合链上存储）：
- 数据条目（即本次上传）的 ID
- Merkle root（32 字节）
- 数据大小
- 存储捐赠金额
- 时间戳
- 数据在日志中的位置（offset）

**智能合约不存储的内容**（太大，不适合链上存储）：
- 实际的 1 MB 数据
- 10 个数据块的具体内容

**为什么这样设计**：
- 如果存储完整数据，成本极高（以太坊存储 1 MB 需要约 $140）
- 只存储承诺，成本极低（32 字节的哈希值）
- 实际数据存储在链下的存储网络中

#### 智能合约的具体工作流程

**步骤 2.1：验证交易**

智能合约检查：
- 用户是否支付了足够的存储捐赠？
- 单位价格是否超过最低要求？
- 交易格式是否正确？

**步骤 2.2：创建数据条目**

如果验证通过，智能合约：
- 分配一个唯一的数据条目 ID
- 记录 Merkle root 和时间戳
- 计算数据在日志中的位置（offset）
- 将存储捐赠存入对应的捐赠池

**步骤 2.3：记录顺序**

**是什么**：0G 使用**日志（log）**这种方式在**逻辑上**组织数据——每个**数据条目**在逻辑上「按顺序追加到日志末尾」。

**重要：日志是逻辑视图，不是一块物理上的大文件**

- **物理上**：每个数据条目的数据都被**打散**成多块，存到不同节点（块 1→A，块 2→B，…）。不存在「一整条物理日志」把所有人的数据按顺序、连续地存在一块存储里。
- **逻辑上**：「日志」指的是由链上/合约维护的**顺序 + 编址**：
  - **顺序**：数据条目 1、2、3… 谁先谁后（由写入时间/合约记录决定）
  - **编址**：每条在「逻辑空间」里的起始位置 offset、长度（大小）
- 整份**逻辑日志** = 「按条目标号/offset 排好序、可索引」的那一整套数据的**抽象视图**；真正的内容物理上分散在许多节点，通过 (条目 ID / offset、大小、Merkle root) 等在链上查到「去哪几个节点、取哪几块」再拼成一条。

**小结**：所谓「日志」**并不对应**一块真实存在的、存满整条数据的物理存储；它是**逻辑层面**的「一整个存储」——顺序和 offset 在链上，数据在打散后的各节点，按链上的索引去取、按顺序理解即可。

---

**这里的「顺序」指什么**：指的是**数据条目的顺序**，即「哪一次上传先、哪一次上传后」。  
- 不是指某个 1 MB 内部 10 个数据块之间的先后（那是实现细节）  
- 而是指：用户 A 的 1 MB、用户 B 的 500 KB、用户 C 的 2 MB… 在**逻辑日志**里依次排列的先后

**为什么需要**：
- 保证**各次上传**的先后可区分（知道谁先存、谁后存）
- 方便按 offset 查询和验证某一次上传对应的数据（即按「逻辑位置」去对应节点拿块、重组）
- 为上层系统（如 Key-Value 存储）提供「按顺序、按 offset 读整条」的基础

**怎么做**：
- 每个**数据条目**在逻辑上有一个唯一的偏移量（**offset**），表示「这条在整条逻辑日志中的起始位置」
- 第一个数据条目的 offset = 0
- 第二个数据条目的 offset = 第一个数据条目占据的大小（例如第一条 1 MB，则第二条从 1 MB 起）
- 以此类推  
- 这些 offset、大小、Merkle root 等都在**链上/合约**里；真实数据仍按块分散存在各节点

**具体例子**（均为**逻辑**位置，链上记录的 offset；真实数据已打散到各节点）：
```
逻辑日志中的位置（offset）：
Offset 0:    数据条目 1 (1 MB)   → 实际存于节点 A,B,C,… 的若干块
Offset 1MB:  数据条目 2 (500 KB) → 实际存于另一些节点的若干块
Offset 1.5MB: 数据条目 3 (2 MB)  → …
...
```

### 第三步：数据存储到网络

#### 存储节点是什么？

**是什么**：存储节点是运行 0G Storage 软件的计算机，负责实际存储数据块。

**为什么需要**：
- 智能合约只存储承诺，不存储实际数据
- 需要有人真正存储数据，让需要的人能够访问
- 存储节点通过存储数据获得奖励

**怎么做**：
- 任何人都可以运行存储节点
- 节点连接到 0G Storage 网络
- 节点可以选择存储哪些数据（基于经济激励）

#### 数据如何分散到存储节点？

**步骤 3.1：数据块分散（Disperse）**

**是什么**：将 10 个数据块发送到不同的存储节点。

**为什么需要**：
- 如果所有数据块都在一个节点，该节点故障会导致数据丢失
- 分散到多个节点，提高可靠性和容错性

**怎么做**：

**分散服务器（Disperse Server）是什么**：
- 分散服务器是 0G 网络中的一个协调节点
- 它负责接收用户上传的数据块，并将它们路由到不同的存储节点
- 可以是一个专门的服务器，也可以是多个节点组成的集群

**分散服务器如何工作**：
1. 用户客户端完成擦除编码后，将 10 个数据块发送到分散服务器
2. 分散服务器维护一个存储节点列表（知道哪些节点在线、有哪些存储空间）
3. 分散服务器根据负载均衡策略，将数据块分配到不同的存储节点
4. 分散服务器将数据块发送到选定的存储节点

**如何保证分散服务器的可靠性**：
- 分散服务器可以是去中心化的（多个节点）
- 如果某个分散服务器故障，其他节点可以接管
- 数据块分配信息也会记录在链上或共识网络中，便于恢复

**具体流程**：
```
用户客户端
  ↓ 发送 10 个数据块
分散服务器
  ↓ 路由分配
数据块 1 → 存储节点 A
数据块 2 → 存储节点 B
数据块 3 → 存储节点 C
...
数据块 10 → 存储节点 J
```

**为什么需要分散服务器**：
- 用户不知道哪些存储节点在线
- 需要智能的路由和负载均衡
- 确保数据块均匀分布到网络中

**具体例子**：
```
数据块 1 → 存储节点 A
数据块 2 → 存储节点 B
数据块 3 → 存储节点 C
...
数据块 10 → 存储节点 J
```

**步骤 3.2：数据复制（Replication）**

**是什么**：根据用户支付的存储费用，数据可能被复制到更多节点。

**为什么需要**：
- 更多副本 = 更高的数据可用性
- 用户可以通过支付更高费用获得更好的可靠性

**怎么做**：
- 如果用户支付了更高的存储费用
- 系统会将数据块复制到更多节点
- 比如：每个数据块复制 3 份，总共 30 个数据块存储在 30 个节点上

**具体例子**：
```
基础费用：每个数据块 1 份副本（10 个节点）
高费用：每个数据块 3 份副本（30 个节点）
```

#### 存储节点如何知道要存储什么？

**步骤 3.3：节点同步**

**是什么**：存储节点定期检查智能合约，看看有哪些新的数据条目需要存储。

**为什么需要**：
- 节点需要知道哪些数据需要存储
- 节点需要知道数据的 Merkle root，以便验证数据完整性

**怎么做**：
1. 节点连接到**主链**（如以太坊）
2. 节点监听**主链上**的 0G Storage 智能合约的事件
3. 当有新数据条目（即新的上传记录）创建时，节点收到通知
4. 节点决定是否存储该数据（基于经济激励）
5. 如果决定存储，节点从分散服务器下载对应的数据块
6. 节点验证数据块的完整性（使用 Merkle root）

**节点如何验证数据块的完整性**：

**问题**：节点下载了数据块，如何知道这个数据块是正确的、没有被篡改？

**解决方案：使用 Merkle Root 验证**

**步骤 1：节点获取 Merkle Root**
- 节点从智能合约中获取数据条目的 Merkle Root（存储在链上）
- 这是数据的"承诺"，所有人都可以验证

**步骤 2：节点计算数据块的哈希值**
- 节点对下载的数据块计算哈希值
- 比如：`hash(数据块1) = 0xdef456...`

**步骤 3：节点需要 Merkle Proof 来验证**
- 节点需要知道这个数据块在 Merkle 树中的位置
- 节点需要 Merkle Proof 来证明数据块属于这个 Merkle Root

**步骤 4：节点验证数据块**
- 节点使用数据块、Merkle Proof 和 Merkle Root 进行验证
- 验证方法与验证者验证数据块的方法相同（见 2.2 节）
- 如果验证通过，节点存储数据块；如果验证失败，节点拒绝存储

**为什么节点也需要验证**：
- 防止存储错误或被篡改的数据
- 确保存储的数据与链上承诺一致
- 如果节点存储了错误数据，后续挖矿时会失败（见第五章）

**具体例子**：
```
节点 A 的验证过程：
1. 从智能合约获取：Merkle Root = 0xabc123...
2. 从分散服务器下载：数据块 1（128 KB）
3. 从分散服务器或网络获取：数据块 1 的 Merkle Proof
4. 验证：
   - 计算 hash(数据块1)
   - 使用 Merkle Proof 重建 Merkle Root
   - 对比：计算出的 Root = 0xabc123... ✓ 匹配！
5. 验证通过 → 存储数据块 1 到本地
```

**具体例子**：
```
节点 A 的决策过程：
1. 看到新数据条目：1 MB，存储捐赠 1000 ZG
2. 计算：单位价格 = 1000 / 4096 ≈ 0.24 ZG/sector
3. 判断：这个价格是否值得存储？
4. 如果值得：下载数据块 1，存储到本地
5. 如果值得：定期参与挖矿，获得奖励
```

## 2.2 共识网络：记录和验证的核心

### 共识网络是什么？

**是什么**：共识网络是一组验证者（validators）通过共识算法达成一致，决定数据的顺序和状态。

**类比**：就像以太坊的验证者网络，但专门用于数据可用性。

### 单共识网络如何工作？

让我们先理解单个共识网络的工作方式，然后再看多个共识网络。

#### 验证者如何工作？

**是什么**：验证者是运行共识软件的节点，负责验证交易和创建区块。

**验证者与存储节点的关系**：
- **验证者**：专门负责验证数据可用性和达成共识，不存储实际数据
- **存储节点**：专门负责存储数据块，不参与共识投票
- 它们是**不同的角色**，但可以运行在同一台机器上（一个节点可以同时是验证者和存储节点）

**为什么需要**：
- 需要有人验证数据的承诺是否正确
- 需要有人记录数据的顺序
- 需要有人验证存储节点的挖矿证明

**怎么做**：

**步骤 1：验证者接收数据条目**

- 用户提交数据条目到智能合约
- 验证者监听智能合约事件
- 验证者看到新的数据条目

**步骤 2：验证者验证数据可用性**

**是什么**：验证者需要确认"数据确实被存储到网络中，且可以被访问"。

**为什么需要**：
- 如果数据没有被存储，承诺就没有意义
- 需要确保数据可用性属性
- 共识层只传输 Merkle root（32 字节），不传输实际数据（1 MB），所以验证者需要验证数据真的存在

**怎么做**：

验证者进行"数据可用性采样"（Data Availability Sampling），这是一个多步骤的过程：

**步骤 2.1：验证者获取数据条目的元数据**

验证者从智能合约中获取：
- Merkle root（32 字节，存储在链上）
- 数据大小（比如 1 MB = 4096 sectors）
- 数据条目的 ID

**为什么只需要这些**：
- 共识层只传输这些小的元数据，不传输实际数据
- 验证者用这些信息来验证数据块

**步骤 2.2：验证者随机选择要验证的数据块**

验证者使用随机算法选择要验证的数据块：
- 假设有 10 个数据块（每个 128 KB）
- 验证者随机选择其中几个（比如数据块 3、7、9）
- 每个验证者选择不同的数据块，增加验证覆盖率

**为什么随机选择**：
- 不需要验证所有数据块（太耗时）
- 随机采样可以高效地检测数据是否可用
- 如果大部分随机样本都可用，说明整体数据很可能可用

**步骤 2.3：验证者从「存有该块」的存储节点获取数据块和 Merkle Proof**

这是**最关键的一步**，验证者如何验证数据块。

**重要前提：单个存储节点没有该数据的所有块**

- 数据是**分散存储**的：数据块 1 → 节点 A，数据块 2 → 节点 B，数据块 3 → 节点 C，…（见 2.1 节「数据块分散」）
- 所以**任何一个存储节点**都只存**分给它的那一块或几块**，不可能存整份数据的所有块
- 验证者若要验证「数据块 3」，必须向**存了数据块 3 的节点**（例如节点 C）请求，而不能向任意一个节点要「整份数据」或「数据块 3」

**验证者如何知道「谁存了数据块 3」？**

- 分散服务器在分配时会产生「数据块分配信息」：数据条目 X 的 数据块 1 → 节点 A，数据块 2 → 节点 B，数据块 3 → 节点 C，…
- 这类分配信息会记录在链上或共识网络中，便于恢复和查询（前文已提及）
- 验证者从链上/共识网络（或依赖分散服务器提供的索引）查出：**数据条目 X 的数据块 3 存在节点 C**
- 再向**节点 C** 请求：数据块 3 + 该块对应的 Merkle Proof

**Merkle Proof 从哪来？**

- 在数据分散到节点时，每个节点除了拿到自己的数据块，还会拿到「该块在整棵 Merkle 树中的 Merkle Proof」（由分散服务器或客户端在构建 Merkle 树时一并计算并随块下发）
- 节点把「数据块 + Merkle Proof」一起存下来；验证者向该节点请求时，节点一并返回，验证者即可用链上的 Merkle Root 做验证

**问题**：验证者只有 Merkle root（32 字节），如何验证某个数据块（128 KB）是正确的？

**关键理解**：
- 验证者**不需要下载完整数据**（1 MB），只需要向**存有该块的节点**要「这一块（128 KB）+ 它的 Merkle Proof（几百字节）」
- 通过 Merkle Proof，验证者可以用链上的 Merkle Root 验证这一块的正确性
- 这就像"抽样检查"：随机挑几块，每块只问「存这一块的那几个节点」要，就能判断整体是否可用

**解决方案：Merkle Proof（Merkle 证明）**

**Merkle Proof 是什么**：
- Merkle Proof 是一组哈希值，用来证明某个数据块确实属于 Merkle 树
- 它包含从数据块到 Merkle root 路径上的所有"兄弟节点"的哈希值
- 大小很小（通常几百字节），比数据块本身小得多

**Merkle Proof 如何工作**：

假设 Merkle 树结构如下：
```
        Merkle Root (存储在链上)
           /        \
      Hash(1-5)   Hash(6-10)
      /     \      /      \
  Hash(1-2) Hash(3-5) Hash(6-7) Hash(8-10)
  /   \    /   \    /   \    /   \
块1  块2  块3  块4  块5  块6  块7  块8  块9  块10
```

如果验证者想验证"数据块 3"：
1. 验证者先查「谁存了数据块 3」→ 得到节点 C；再向**节点 C**（即存有数据块 3 的存储节点）请求：数据块 3 + Merkle Proof
2. 节点 C 返回（因为它在分散时存了「块 3 + 该块的 Merkle Proof」）：
   - 数据块 3 的内容（128 KB）
   - Merkle Proof：包含路径上的兄弟节点哈希值
     - 块 4 的哈希值（块 3 的兄弟）
     - Hash(1-2)（Hash(3-5) 的兄弟）
     - Hash(6-10)（Hash(1-5) 的兄弟）

**步骤 2.4：验证者使用 Merkle Proof 验证数据块**

验证者收到数据块和 Merkle Proof 后，进行验证：

1. **计算数据块的哈希值**：
   - 对收到的数据块 3 计算哈希：`hash(块3)`

2. **使用 Merkle Proof 重建路径**：
   - 将 `hash(块3)` 和 `hash(块4)`（来自 Proof）组合，计算 `Hash(3-5)`
   - 将 `Hash(3-5)` 和 `Hash(1-2)`（来自 Proof）组合，计算 `Hash(1-5)`
   - 将 `Hash(1-5)` 和 `Hash(6-10)`（来自 Proof）组合，计算最终的 Merkle Root

3. **对比 Merkle Root**：
   - 将计算出的 Merkle Root 与链上存储的 Merkle Root 对比
   - 如果相同 → 数据块正确
   - 如果不同 → 数据块被篡改或错误

**为什么这样有效**：
- 如果数据块被篡改，计算出的 Merkle Root 会不同
- 如果 Merkle Proof 错误，计算出的 Merkle Root 也会不同
- 只有数据块和 Merkle Proof 都正确时，才能匹配链上的 Merkle Root

**步骤 2.5：验证者检查「存有该块的」节点是否能够提供数据**

验证者不仅验证数据块的正确性，还验证数据是否真的被存储：

1. 验证者向**存有该数据块的那一（些）节点**请求该数据块和 Merkle Proof
2. 若该节点能快速返回 → 说明这块数据确实被存了且可访问 → 数据可用
3. 若该节点无法提供或响应很慢 → 这块可能丢失或不可用 → 数据可能不可用

**步骤 2.6：验证者投票**

每个验证者完成采样后：
- 如果验证通过 → 投票"数据可用"
- 如果验证失败 → 投票"数据不可用"

如果大多数验证者（如 2/3）确认数据可用，数据就被标记为"可用"。

**完整例子**：

```
数据条目：1 MB，Merkle Root = 0xabc123...

验证者 A 的验证过程：
1. 从智能合约获取：Merkle Root = 0xabc123...，数据大小 = 1 MB
2. 随机选择：验证数据块 3
3. 查分配信息知「数据块 3 存在节点 C」→ 向节点 C 请求：数据块 3 + Merkle Proof
4. 节点 C 返回（它只存了块 3，不存其它块）：
   - 数据块 3（128 KB）
   - Merkle Proof：[hash(块4), Hash(1-2), Hash(6-10)]
5. 验证者 A 验证：
   - 计算 hash(块3)
   - 使用 Proof 重建 Merkle Root
   - 对比：计算出的 Root = 0xabc123... ✓ 匹配！
6. 验证者 A 投票："数据可用"

验证者 B 的验证过程：
1. 随机选择：验证数据块 7
2. 查知数据块 7 在节点 G → 向节点 G 请求：数据块 7 + Merkle Proof
3. 验证通过 → 投票"数据可用"

验证者 C 的验证过程：
1. 随机选择：验证数据块 1
2. 查知数据块 1 在节点 A → 向节点 A 请求：数据块 1 + Merkle Proof
3. 验证通过 → 投票"数据可用"

结果：3/3 验证者确认数据可用 → 数据标记为"可用"
```

**关键要点**：
- **每个存储节点只存一部分块**，不会存该数据条目的所有块；验证者要验「块 k」时，必须向**存了块 k 的节点**请求，不能向任意节点要整份数据
- 验证者通过链上/共识网络/分散服务器的**分配信息**查出「块 k 在谁那儿」，再向该节点要「块 k + Merkle Proof」
- 验证者**不需要下载完整数据**（1 MB），只需向存有该块的节点要「这一块（128 KB）+ 它的 Merkle Proof（几百字节）」
- 通过 Merkle Proof，验证者可以**用链上的 Merkle Root 验证这一块的正确性**
- 随机采样确保即使只验证部分数据块，也能高效检测数据可用性

---

**常见疑问：验证者只抽样一部分块，怎么保证「整份数据」都被验证过？**

你举的情况是：10 个块里 9 个正确、1 个错误，若按「多数验证者通过」就判整份可用，投票率可能很高，但有一块错，整份文件岂不是仍不可用、不可信？

回答分两层：

**1. 擦除编码：整份「可用」不要求 10 块全对**

- 0G 用了擦除编码：10 块中**只要有不少于某个阈值的块正确**（例如 7 块），就能恢复出整份数据。
- 所以「9 块对、1 块错」时，仍可用那 9 块恢复整份数据 → **整份数据在实际使用时是可用的**。真正危险的是：**坏/丢的块超过阈值**（例如 4 块以上坏或丢），那时才无法恢复。
- 「数据可用」的含义是：**将来有人要取这份数据时，能恢复出完整内容**。在擦除编码下，不需要「10 块全对」，只需要「足够多的块对」。

**2. 抽样 + 概率：怎么保证「坏块超过阈值」时不会被打成「可用」？**

- 验证者**随机**抽样，每人验的是不同子集；**验证者数量**和**每人抽的块数**足够大时，只要坏/丢的块**超过阈值**，就很容易有人抽到坏块或发现取不到块，从而投「不可用」。
- 粗略直觉：若 10 块里有 4 块坏，每个验证者随机抽几块，抽到至少一块坏块的概率不低；很多验证者一起抽，**集体覆盖**就会很高，只要参数（验证者数 × 每人抽样数）设计合理，就可以把「坏块超过阈值、却被多数验证者判为可用」的概率压到极低（概率性安全）。
- 协议层面会要求：**参与抽样的验证者足够多、每人抽的块数足够多**，使得在「超过阈值的块坏/丢」时，期望下有足够多验证者会抽到坏块或取不到块，从而投「不可用」，不会出现「90% 通过但整份其实不可用」还被视为通过。

**小结**：
- 「整份可用」在 0G 里靠**擦除编码**：不需要 10 块全对，只需「够多块对」即可恢复。
- 「只抽一部分块」仍能保护整份，靠的是：**随机抽样 + 足够多的验证者和抽样数**，让「坏块超过恢复阈值却仍被多数判为可用」的概率足够小；若真的超过阈值，恢复时会失败，和「标记为不可用」一致。

**步骤 3：验证者达成共识**

**是什么**：验证者通过共识算法（如 POS，权益证明）达成一致，决定哪些数据条目被接受。

**为什么需要**：
- 不同验证者可能看到不同的数据（网络延迟、节点故障等原因）
- 需要达成一致，决定数据的最终顺序
- 确保所有验证者对数据状态有相同的认知

**与步骤 2 的关系**：
- 步骤 2 是验证"数据是否可用"（数据可用性验证）
- 步骤 3 是验证"数据条目是否被接受"（共识投票）
- 只有通过步骤 2 验证的数据，才会在步骤 3 中被投票

**怎么做**：
- 验证者投票决定是否接受数据条目
- 投票权重基于质押的代币数量
- 如果大多数验证者（如 2/3）同意，数据条目被接受
- 数据条目的顺序被记录在区块中

**具体例子**：
```
验证者 A（质押 1000 代币）：投票"接受"
验证者 B（质押 2000 代币）：投票"接受"
验证者 C（质押 500 代币）：投票"拒绝"
...
总质押：3500 代币
同意质押：3000 代币（> 2/3）
结果：数据条目被接受
```

#### 单共识网络的瓶颈在哪里？

**吞吐量限制**和**带宽限制**不是一回事，但都会卡住单共识网络，需要分开看：

---

**吞吐量限制（条/秒）**

**是什么**：共识网络**每秒能处理多少笔/条**——受验证、投票、共识轮次等**逻辑与计算**限制，不直接看「传了多少字节」。

**为什么是瓶颈**：
- 每笔都要验证、都要参与共识，验证者需要时间
- 条数一多，排队变长，**条数/秒**上不去

**例子**：
```
共识网络每秒只能处理 100 个数据条目（不管每条多大）
1000 个请求 → 至少 10 秒才能处理完
→ 吞吐量 = 100 条/秒
```

---

**带宽限制（字节/秒）**

**是什么**：共识网络**每秒能传多少数据**——受网络物理带宽（MB/s、GB/s）限制。若每笔都带很大数据，传不动就会成为瓶颈。

**为什么是瓶颈**：
- 若「数据条目」要把整份 1 MB 数据在共识网络里广播，每条 1 MB、100 条就是 100 MB
- 验证者要收、要验，网络要传，**字节/秒**不够就卡住

**例子**：
```
若每条在共识网络里传 1 MB：100 条 = 100 MB → 带宽很快打满
0G 只在共识网络里传 Merkle root（32 字节）：100 条 ≈ 3.2 KB → 带宽压力小很多
```

---

**二者区别小结**

| | 吞吐量限制 | 带宽限制 |
|---|---|---|
| **量纲** | 条/秒（或笔/秒） | 字节/秒（MB/s、GB/s） |
| **瓶颈来自** | 验证、共识、投票等**处理能力** | **网络传输能力** |
| **典型现象** | 「条数太多，处理不过来」 | 「单条或总数据太大，传不过来」 |
| **0G 的应对** | 多共识网络并行，提高「条/秒」 | 共识层只传承诺（小数据），降低「字节/秒」需求 |

可以出现「吞吐量够但带宽不够」（能处理很多条，但每条太大传不动），或「带宽够但吞吐量不够」（传得动，但共识/验证慢）。0G 用「只传小数据」主要缓解**带宽**问题，用「多共识网络」主要提高**吞吐量**。

## 2.3 多共识网络：如何实现无限扩展？

### 为什么需要多个共识网络？

**问题**：单共识网络的吞吐量有限，无法满足大规模需求。

**解决方案**：创建多个独立的共识网络，并行处理数据请求。

### 如何创建多个共识网络？

#### 步骤 1：创建新的共识网络

**是什么**：0G 可以创建多个共识网络（Chain 0, Chain 1, Chain 2, ...），各自处理不同分区的数据请求，但**共用同一组验证者**。

**为什么可以**：
- 每个网络处理的数据请求不同（分工不同）
- 它们可以并行运行，互不干扰
- **验证者不是「每网一套」**：所有网络由**同一组验证者**参与共识（具体机制见 2.4 节「共享质押」）

**怎么做**：
- 0G 部署多个共识网络
- **所有网络共享同一组验证者**（即不是「每个网络各有一套验证者」）
- 每个网络有自己的区块和状态（处理的数据请求不同）

**具体例子**：
```
同一组验证者 V 同时参与：
Chain 0：验证者 V，处理数据请求 A, B, C
Chain 1：验证者 V，处理数据请求 D, E, F
Chain 2：验证者 V，处理数据请求 G, H, I
→ 验证者只有一套（V），网络有三条，分工不同
```

#### 步骤 2：存储网络分区

**是什么**：存储网络被分成多个分区（Partition），每个分区连接到不同的共识网络。

**什么是分区**：
- 分区是存储网络的逻辑划分，类似于数据库的分片（sharding）
- 每个分区包含一组存储节点，这些节点存储该分区的数据
- 不同分区的数据可以独立处理，互不干扰

**为什么需要**：
- 不同分区的数据可以并行处理
- 避免所有数据都通过同一个共识网络
- 提高系统的可扩展性和吞吐量

**怎么做**：
- 存储网络被分成多个分区（Partition A, B, C, ...）
- 每个分区连接到特定的共识网络（比如分区 A 连接到 Chain 0）
- 用户可以选择将数据存储到哪个分区，或者系统自动分配

**具体例子**：
```
分区 A → Chain 0
分区 B → Chain 1
分区 C → Chain 2
```

#### 步骤 3：数据路由

**是什么**：系统决定将用户的数据请求路由到哪个共识网络。

**为什么需要**：
- 需要平衡不同网络的负载
- 需要确保数据被正确路由

**怎么做**：
- 用户可以选择将数据存储到特定分区
- 或者系统自动分配（基于负载均衡）
- 数据请求被路由到对应的共识网络

**具体例子**：
```
用户 1 的数据 → 分区 A → Chain 0
用户 2 的数据 → 分区 B → Chain 1
用户 3 的数据 → 分区 C → Chain 2
→ 三个请求并行处理，互不阻塞
```

### 多共识网络的工作流程

让我们看一个完整的例子：

**场景**：三个用户同时上传数据

**用户 A**：
1. 上传 1 MB 数据
2. 系统路由到分区 A
3. 分区 A 连接到 Chain 0
4. Chain 0 的验证者处理这个请求
5. 数据存储到分区 A 的存储节点

**用户 B**：
1. 上传 2 MB 数据
2. 系统路由到分区 B
3. 分区 B 连接到 Chain 1
4. Chain 1 的验证者处理这个请求
5. 数据存储到分区 B 的存储节点

**用户 C**：
1. 上传 500 KB 数据
2. 系统路由到分区 C
3. 分区 C 连接到 Chain 2
4. Chain 2 的验证者处理这个请求
5. 数据存储到分区 C 的存储节点

**结果**：三个请求**同时处理**，互不阻塞，总吞吐量 = 单网络吞吐量 × 网络数量

### 关键问题：如何保证安全性？

**若采用「每网各自一套验证者」**：每个共识网络都有自己的验证者集合（V0、V1、V2…），则每个网络都要单独拉验证者、单独质押，成本高，且小网络的验证者集合小、安全性弱。

**0G 的做法**：不采用「每网各自验证者」，而是让**所有共识网络共用同一组验证者**（上文的「同一组验证者 V」）。这样只需一套质押，即可为所有网络提供共识，安全性与主网一致。具体如何实现「共享验证者」和「统一质押」，见下节**共享质押（Shared Staking）**。

## 2.4 共享质押：如何保证所有网络的安全性？

### 共享质押是什么？

**是什么**：所有共识网络共享同一组验证者，使用相同的质押状态。

**类比**：就像一张高速公路通行证，可以在多个收费站使用，通行证的价值（安全性）在所有收费站都相同。

### 为什么需要共享质押？

**问题**：如果每个共识网络都需要**独立的**验证者和质押，会带来「成本极高」和「小网络不安全」两个后果。

---

**为什么成本高？**

- **质押要重复锁**：同一批验证者若想参与多条链，就要在每条链上各质押一份。例如参与 Chain 0、1、2，就要锁三份资产（如各 1000 ETH），而不能用一份质押同时为三条链背书。
- **运营和激励成倍增加**：每条链都要单独招募验证者、单独建质押池、单独发奖励，网络越多，要维护的「验证者集合 + 质押池」越多，总成本和运营复杂度成倍上升。
- **对小验证者不友好**：资本有限的验证者只能选少量网络参与，若每条链都要求最低质押，多链参与门槛会被推得很高。

**结论**：每网一套验证者、一套质押 → 总锁仓和总运营成本随网络数量线性甚至更快增长，成本极高。

---

**为什么不安全？**

- **单网验证者集合小**：若 Chain 1 只有「自己的」那几百个验证者，攻击者只需收买或攻破这一小批人，就能在该链上作恶（双花、审查等）。  
- **安全性与质押规模相关**：以太坊之所以难攻破，是因为有几十万验证者、几千亿美金级质押；若每条小链只有少量验证者和少量质押，攻击该链的成本会低很多。
- **长尾链更弱**：越晚建、越小的链，越难拉到足够多验证者和足够大质押，安全性越差，形成「小链更不安全 → 更难吸引验证者」的负循环。

**结论**：每网一套验证者 → 小网络的验证者集合小、质押少，安全性弱；攻击成本低。

---

**解决方案**：所有网络**共享**同一组验证者和同一套质押（共享质押）。同一批验证者、同一笔质押，同时为多条链提供共识 → **成本只付一份**，**安全性等于主网（或主链）级别**。

### 共享质押如何实现？

#### 步骤 1：选择一个主网络（C0）

**是什么**：选择一个共识网络作为"主网络"，维护共享质押状态。

**为什么需要**：
- 需要一个地方统一管理质押状态
- 避免每个网络都维护自己的质押状态

**怎么做**：
- 0G 可以选择 Chain 0 作为主网络
- 或者使用外部区块链（如以太坊）作为主网络
  - **为什么选择以太坊**：以太坊有最高的安全性和最大的验证者集合，作为主网络最可靠
  - **如何选择**：0G 团队在部署时决定，通常选择以太坊作为主网络
- 主网络维护一个智能合约，记录所有验证者的质押状态

**具体例子**：
```
主网络：以太坊（C0）
智能合约：0G Staking Contract
记录：验证者 A 质押 1000 ETH，验证者 B 质押 2000 ETH，...
```

#### 步骤 2：代币映射机制

**是什么**：当其他共识网络（Ci）产生代币时，通过跨链桥映射到主网络的代币。

**为什么需要**：
- 每个共识网络可能产生自己的代币（$0G0, $0G1, $0G2, ...）
- 但质押需要统一使用主网络的代币（T0）
- 所以需要将其他网络的代币映射到主网络

**怎么做**：

**详细步骤**：

1. **网络 Ci 产生代币 Ti**
   - 比如 Chain 1 通过挖矿奖励产生 $0G1 代币
   - 这些代币在 Chain 1 上流通

2. **销毁 Ti，铸造 T0**
   - 验证者想要在 Chain 1 上质押
   - 通过跨链桥：在 Chain 1 上销毁 $0G1
   - 在以太坊（C0）上铸造对应的 T0（ERC20 代币）
   - 这个 T0 被存入质押合约

3. **跨链桥如何保证安全？**
   - 使用安全的跨链桥协议（如 LayerZero、Wormhole 等成熟的跨链桥）
   - 验证跨链交易的有效性（通过多签、中继链等方式）
   - 确保"销毁"和"铸造"是原子操作（要么都成功，要么都失败）
   - **什么是原子操作**：指一系列操作要么全部成功，要么全部失败，不会出现部分成功的情况。这保证了跨链操作的安全性。

**具体例子**：
```
Chain 1 产生 1000 $0G1
↓ 跨链桥
在 Chain 1 上销毁 1000 $0G1
在以太坊上铸造 1000 T0（ERC20）
T0 存入质押合约
```

#### 步骤 3：统一投票权

**是什么**：验证者在任何共识网络上投票时，都使用在主网络（C0）上的质押数量作为投票权。

**为什么需要**：
- 保证所有网络使用相同的安全级别
- 避免不同网络有不同的安全性

**怎么做**：

**详细流程**：

1. **验证者质押**
   - 验证者在以太坊上质押 1000 T0
   - 质押状态记录在以太坊的智能合约中

2. **验证者参与 Chain 1 的共识**
   - 验证者运行 Chain 1 的验证者软件
   - Chain 1 的共识算法查询以太坊的质押合约
   - 获取验证者在以太坊上的质押数量（1000 T0）
   - 使用这个数量作为在 Chain 1 上的投票权

3. **验证者参与 Chain 2 的共识**
   - 同样的验证者也可以参与 Chain 2
   - Chain 2 也查询以太坊的质押合约
   - 使用相同的质押数量（1000 T0）作为投票权

**具体例子**：
```
验证者 V：
- 在以太坊上质押：1000 T0
- 在 Chain 1 上投票权：1000 T0（从以太坊查询）
- 在 Chain 2 上投票权：1000 T0（从以太坊查询）
- 在 Chain 3 上投票权：1000 T0（从以太坊查询）

→ 所有网络共享相同的安全级别
```

### 使用以太坊作为主网络的优势

**优势 1：标准化**
- T0 可以是标准 ERC20 代币
- 兼容现有的钱包和工具

**优势 2：高安全性**
- 继承以太坊的安全性
- 以太坊有最大的验证者集合和最长的运行历史

**优势 3：集成现有系统**
- 可以与 EigenLayer 等再质押框架集成
- 可以使用 BTC、ETH 等高价值代币作为质押来源

**结果**：0G 的所有共识网络可以达到以太坊级别的安全性，同时实现无限扩展。

## 2.5 数据发布通道 vs 数据存储通道：如何分离？

### 为什么需要分离两个通道？

**问题**：如果所有数据都要通过共识网络，共识网络会成为瓶颈。

**解决方案**：将数据发布（小数据）和数据存储（大数据）分离到两个不同的通道。

### 数据发布通道是什么？

**是什么**：数据发布通道负责在共识网络上记录数据的承诺和验证数据可用性。

**传输什么**：
- Merkle root（32 字节）
- 数据条目的元数据（大小、时间戳等）
- 存储节点的聚合签名（证明数据被存储）

**为什么数据量小**：
- 只传输承诺和签名，不传输实际数据
- 数据量极小（几 KB），不会造成瓶颈

**具体例子**：
```
用户上传 1 MB 数据
↓ 数据发布通道
传输到共识网络：
- Merkle root: 32 字节
- 元数据: 100 字节
- 聚合签名: 64 字节
总计: ~200 字节（而不是 1 MB）
```

### 数据存储通道是什么？

**是什么**：数据存储通道负责在存储网络中传输和存储实际数据。

**传输什么**：
- 实际的数据块（每个 128 KB 或 256 KB）
- 数据块的副本

**为什么数据量大**：
- 传输的是实际数据，数据量很大
- 但不需要通过共识网络，直接通过存储网络传输

**具体例子**：
```
用户上传 1 MB 数据
↓ 数据存储通道
传输到存储网络：
- 数据块 1: 128 KB → 存储节点 A
- 数据块 2: 128 KB → 存储节点 B
...
- 数据块 10: 128 KB → 存储节点 J
总计: 1 MB（通过存储网络，不经过共识网络）
```

### 两个通道如何协同工作？

让我们看一个完整的例子：

**用户上传 1 MB 数据**

**步骤 1：数据准备**
- 擦除编码：1 MB → 10 个 128 KB 数据块
- 生成 Merkle root

**步骤 2：数据发布通道（共识网络）**
- 用户向智能合约提交交易
- 包含：Merkle root（32 字节）、数据大小、存储捐赠
- 智能合约创建数据条目
- 验证者验证数据可用性
- **传输数据量：~200 字节**

**步骤 3：数据存储通道（存储网络）**
- 分散服务器将 10 个数据块发送到 10 个存储节点
- 存储节点存储数据块
- **传输数据量：1 MB**

**步骤 4：验证一致性**

**是什么**：验证者需要确保数据发布通道（链上的承诺）和数据存储通道（实际存储的数据）是一致的。

**为什么需要**：
- 链上只有 Merkle Root，没有实际数据
- 需要确保实际存储的数据与链上的承诺匹配
- 防止存储节点存储错误数据或撒谎

**怎么做**：

**步骤 4.1：验证者进行数据可用性采样**
- 验证者随机选择一些数据块进行验证
- 验证者根据分配信息找到「存有该块的节点」，向该节点请求该数据块和 Merkle Proof（而不是向任意节点要整份数据；见 2.2 节）

**步骤 4.2：验证者使用 Merkle Root 验证数据块**
- 验证者使用链上的 Merkle Root 验证数据块（详细方法见 2.2 节）
- 如果验证通过，说明数据块正确
- 如果验证失败，说明数据被篡改或错误

**步骤 4.3：验证者检查数据是否真的被存储**
- 验证者检查**存有该块的**存储节点是否能够快速提供该数据块（及 Merkle Proof）
- 若该节点无法提供或响应很慢，说明这块可能未存或不可用

**步骤 4.4：验证者投票**
- 如果验证通过，验证者投票"数据可用"
- 如果大多数验证者确认数据可用，数据被标记为"可用"

**具体例子**：
```
数据发布通道（共识网络）：
- 链上记录：Merkle Root = 0xabc123...，数据大小 = 1 MB

数据存储通道（存储网络）：
- 存储节点 A 存储：数据块 1（128 KB）
- 存储节点 B 存储：数据块 2（128 KB）
...
- 存储节点 J 存储：数据块 10（128 KB）

验证一致性：
1. 验证者随机选择：验证数据块 3
2. 查知数据块 3 在节点 C → 向节点 C 请求：数据块 3 + Merkle Proof
3. 使用链上的 Merkle Root (0xabc123...) 验证数据块 3
4. 验证通过 → 数据块 3 与链上承诺一致 ✓
5. 验证者投票："数据可用"

结果：数据发布通道和存储通道一致，数据可用
```

**结果**：
- 共识网络只处理小数据（承诺），不会成为瓶颈
- 存储网络处理大数据（实际数据），可以水平扩展
- 两个通道协同工作，实现高效的数据可用性

### 为什么这样设计？

**优势 1：避免共识瓶颈**
- 共识网络只处理小数据，不会因为大数据而阻塞
- 可以处理更多的数据请求

**优势 2：存储网络可扩展**
- 存储网络可以水平扩展（添加更多节点）
- 不受共识网络的限制

**优势 3：效率最大化**
- 大数据走存储网络（快速、便宜）
- 小数据走共识网络（安全、可验证）

## 2.6 数据可用性 Quorum：如何防止串通？

### Quorum 是什么？

**是什么**：Quorum 是一组存储节点的集合，负责验证特定数据的可用性。

**为什么需要**：
- 需要一组节点来验证数据是否真的被存储
- 不能只依赖单个节点（可能不可靠）

### 如何构建 Quorum？

#### 问题：随意构建的风险

**如果让节点自己选择加入哪个 Quorum**：
- 恶意客户端可能与节点串通
- 客户端选择"友好"的节点组成 Quorum
- 这些节点可能撒谎，说数据可用，但实际上没有存储

**具体例子**：
```
恶意客户端 + 恶意节点串通：
客户端：选择节点 A, B, C（都是我的朋友）
节点 A, B, C：即使没有存储数据，也说"数据可用"
结果：系统被欺骗，数据实际上不可用
```

#### 0G 的解决方案：VRF 随机构建

**是什么**：使用可验证随机函数（VRF）随机选择节点组成 Quorum。

**什么是 VRF（可验证随机函数）**：
- VRF 是一种密码学函数，可以生成一个随机数，同时提供证明
- 任何人都可以验证这个随机数确实是通过 VRF 计算得出的，不是伪造的
- 类似于"抛硬币"，但结果是可验证的、不可预测的

**为什么有效**：
- 客户端无法选择节点（随机选择）
- 节点无法预测自己是否会被选中
- 防止串通

**怎么做**：

**详细步骤**：

1. **生成随机种子**
   - 使用当前区块的哈希值作为随机种子
   - 这个种子是公开的，所有人都可以验证

2. **VRF 计算**
   - 对每个存储节点，使用 VRF 计算一个随机数
   - VRF 保证：随机数看起来是随机的，但可以验证是正确的

3. **选择节点**
   - 根据随机数选择节点组成 Quorum
   - 比如：随机数最小的 10 个节点组成 Quorum

4. **验证随机性**
   - 任何人都可以验证 VRF 计算是否正确
   - 确保 Quorum 真的是随机选择的

**具体例子**：
```
区块哈希：0xabc123...
↓ VRF 计算
节点 A: 随机数 0.123
节点 B: 随机数 0.456
节点 C: 随机数 0.789
...
节点 Z: 随机数 0.234

选择随机数最小的 10 个节点 → Quorum
→ 客户端无法预测或控制哪些节点被选中
```

### VRF Quorum 的优势

**优势 1：防止串通**
- 客户端无法选择"友好"节点
- 节点无法预测是否会被选中

**优势 2：共享安全性**
- Quorum 是随机选择的，诚实节点的分布与整个网络相同
- 如果整个网络有 2/3 诚实节点，Quorum 也有约 2/3 诚实节点

**优势 3：高效验证**
- 聚合签名提交到 0G 共识网络验证
- 比提交到以太坊快几个数量级

### Quorum 如何验证数据可用性？

**问题**：Quorum 中的节点如何证明数据可用？如何防止节点撒谎？

**0G 的解决方案：聚合签名（Aggregated Signature）**

**步骤 1：Quorum 中的节点验证数据**

**重要**：Quorum 中的节点**不是存储节点**，而是**验证节点**（可以是验证者或专门的验证节点）。

每个被选中的节点：
1. 根据分配信息找到存有目标数据块的节点，向该节点获取该数据块和 Merkle Proof（同一份数据的各块分散在不同节点，所以是「向存了某块的那几个节点」分别要对应块）
2. 使用 Merkle Root 验证数据块（方法与验证者相同，见 2.2 节）
3. 如果验证通过，节点生成一个签名，表示"我确认数据可用"

**步骤 2：节点生成签名**

节点使用自己的私钥对以下内容签名：
- 数据条目的 ID
- Merkle Root
- 时间戳
- 其他元数据

签名内容：`sign(节点私钥, 数据ID + MerkleRoot + 时间戳)`

**为什么需要签名**：
- 签名证明节点确实验证了数据
- 签名不可伪造（只有拥有私钥的节点才能生成）
- 如果节点撒谎，签名可以作为证据

**步骤 3：聚合签名**

**是什么**：将多个节点的签名聚合成一个签名，减少验证成本。

**为什么需要**：
- 如果 Quorum 有 10 个节点，提交 10 个独立签名，验证成本高
- 聚合签名将 10 个签名合并成 1 个，验证成本低

**怎么做**：
- 使用 BLS（Boneh-Lynn-Shacham）聚合签名算法
  - **什么是 BLS 聚合签名**：BLS 是一种特殊的签名算法，可以将多个签名"合并"成一个签名
  - **为什么可以合并**：BLS 签名的数学特性允许将多个签名相加，结果仍然是一个有效签名
  - **优势**：验证者只需要验证一个聚合签名，就能确认所有节点都验证了数据，大大降低验证成本
- 将多个签名聚合成一个签名
- 验证者只需要验证一个聚合签名，就能确认所有节点都验证了数据

**步骤 4：提交到共识网络**

Quorum 将聚合签名提交到 0G 共识网络：
- 包含：数据条目 ID、聚合签名、验证结果
- 共识网络的验证者验证聚合签名
- 如果签名有效，数据被标记为"可用"

**具体例子**：
```
数据条目：1 MB，Merkle Root = 0xabc123...

VRF 选择 Quorum：
- 节点 A（随机数 0.123）
- 节点 B（随机数 0.234）
- 节点 C（随机数 0.345）
...
- 节点 J（随机数 0.456）

每个节点验证数据：
1. 节点 A：验证数据块 3 → 通过 → 生成签名 Sig_A
2. 节点 B：验证数据块 7 → 通过 → 生成签名 Sig_B
3. 节点 C：验证数据块 1 → 通过 → 生成签名 Sig_C
...
4. 节点 J：验证数据块 5 → 通过 → 生成签名 Sig_J

聚合签名：
- 将 Sig_A, Sig_B, Sig_C, ..., Sig_J 聚合成一个签名 AggSig
- 提交到共识网络：{数据ID, AggSig, "数据可用"}

共识网络验证：
- 验证聚合签名 AggSig
- 如果有效 → 数据标记为"可用"
```

**如何防止节点撒谎**：

**机制 1：随机选择**
- 节点无法预测是否会被选中
- 恶意节点无法提前准备

**机制 2：签名验证**
- 节点必须提供有效签名
- 如果节点撒谎（说数据可用但实际不可用），后续验证会失败

**机制 3：经济惩罚**
- 如果节点提供错误验证，会被惩罚（见第五章）
- 节点有经济激励诚实验证

**关键要点**：
- Quorum 通过聚合签名高效地证明数据可用性
- 随机选择防止串通
- 签名机制确保验证的可信度

## 2.7 GPU 加速的擦除编码：为什么重要？

### 擦除编码的性能瓶颈

**是什么**：擦除编码是将数据分割成多个块的过程，计算量很大。

**为什么是瓶颈**：
- 对于大文件，擦除编码可能需要很长时间
- 如果用户上传 1 GB 数据，擦除编码可能需要几分钟
- 这限制了单个客户端的吞吐量

**具体例子**：
```
用户上传 1 GB 数据
↓ CPU 擦除编码
处理时间：5 分钟
→ 用户需要等待 5 分钟才能完成上传
```

### GPU 加速如何工作？

**是什么**：使用 GPU（图形处理器）并行计算，加速擦除编码过程。

**为什么 GPU 更快**：
- GPU 有数千个核心，可以并行处理
- CPU 只有几个核心，串行处理
- 对于擦除编码这种可以并行化的任务，GPU 快得多

**怎么做**：
- 0G 提供 GPU 加速的擦除编码库
- 用户可以使用 GPU 加速处理
- 处理速度提升 10-100 倍

**具体例子**：
```
用户上传 1 GB 数据
↓ GPU 擦除编码
处理时间：5 秒（而不是 5 分钟）
→ 用户几乎立即完成上传
```

### 影响

**对用户**：
- 上传速度更快
- 更好的用户体验

**对系统**：
- 提高整体吞吐量
- 支持更大规模的数据上传

## 2.8 通用存储系统：支持多种应用

### 为什么需要通用存储？

**问题**：不同的应用需要不同的存储特性。

**例子**：
- Layer 2 网络：需要永久存储交易数据
- 去中心化社交网络：需要可变数据（可以更新、删除）
- 协作编辑：需要事务处理（并发控制）

**解决方案**：0G Storage 提供分层设计，支持多种存储需求。

### 三层架构概览

**Log 层（基础层）**：
- 追加写入，永久存储
- 适合：数据归档、历史记录

**Key-Value 层**：
- 可变数据，访问控制
- 适合：社交网络、文件系统

**事务处理层**：
- 并发控制，ACID 特性
- 适合：协作编辑、数据库

## 总结：0G 的核心设计优势

通过本章的学习，你应该理解：

1. **0G 如何工作**：
   - 数据上传 → 擦除编码 → 生成承诺 → 提交到**主链**上的 0G Storage 智能合约
   - 合约为每次上传创建一个**数据条目**，并记录承诺以及**数据条目的先后顺序**
   - 数据块分散到存储节点
   - 存储节点参与挖矿获得奖励

2. **为什么需要多共识网络**：
   - 单共识网络有吞吐量瓶颈
   - 多共识网络可以并行处理，实现无限扩展

3. **如何保证安全性**：
   - 共享质押机制，所有网络共享相同的安全级别
   - VRF 随机选择 Quorum，防止串通

4. **如何提高效率**：
   - 分离数据发布和数据存储通道
   - GPU 加速擦除编码

这些设计理念共同构成了 0G 实现"无限可扩展且可靠"数据可用性系统的基础。
