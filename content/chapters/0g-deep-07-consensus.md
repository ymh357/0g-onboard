# 7. 多网络共识与 Shared Staking

## 本章导读

0G 的创新架构不是单一区块链，而是多个专用网络的协同。Shared Staking（共享质押）机制让验证者可以同时为多个网络提供安全性。本章将深入剖析这一独特设计。

**学习目标**：
- 理解多网络架构的必要性
- 掌握 Shared Staking 的工作原理
- 分析跨网络安全性保证
- 理解验证者的激励与惩罚机制

### 7.1.2 为什么需要多网络？

**单一网络的局限**：

```
问题：一条链试图做所有事情

┌──────────────────────────────────────────────────────────┐
│           单一链的性能冲突                                │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  交易处理（Chain）：                                     │
│    需求：低延迟、高吞吐、EVM 兼容                       │
│                                                          │
│  数据存储（Storage）：                                   │
│    需求：大容量、持久性、随机访问                       │
│                                                          │
│  数据可用性（DA）：                                      │
│    需求：极高吞吐、短期存储、并行验证                   │
│                                                          │
│  AI 计算（Compute）：                                    │
│    需求：GPU 支持、TEE 验证、异步执行                   │
│                                                          │
│  → 这些需求相互冲突！                                   │
│  → 单一链难以同时优化                                   │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**多网络的优势**：

```
┌──────────────────────────────────────────────────────────┐
│           专用网络架构优势                                │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  1. 性能专化（Specialization）                           │
│     每个网络针对特定任务优化                             │
│                                                          │
│  2. 独立扩展（Independent Scaling）                      │
│     Storage 繁忙 → 增加 Storage 节点                    │
│     不影响 Chain 性能                                    │
│                                                          │
│  3. 故障隔离（Fault Isolation）                          │
│     DA 网络故障 → 不影响 Chain 交易                     │
│                                                          │
│  4. 灵活升级（Flexible Upgrades）                        │
│     可以单独升级某个专网                                 │
│     无需整体硬分叉                                       │
│                                                          │
│  5. 经济效率（Economic Efficiency）                      │
│     用户只支付需要的服务                                 │
│     不必为未使用的功能付费                               │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

## 7.2 Shared Staking 机制

### 7.2.1 什么是 Shared Staking？

```
┌──────────────────────────────────────────────────────────┐
│              Shared Staking 核心思想                      │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  传统多链模型：                                          │
│  ┌────────────────────────────────────────┐             │
│  │  Chain A: 质押 10M tokens              │             │
│  │  Chain B: 质押 10M tokens              │             │
│  │  Chain C: 质押 10M tokens              │             │
│  │                                        │             │
│  │  总质押：30M tokens                     │             │
│  │  安全性：每条链独立                     │             │
│  └────────────────────────────────────────┘             │
│                                                          │
│  Shared Staking 模型：                                   │
│  ┌────────────────────────────────────────┐             │
│  │  验证者质押：15M tokens（一次质押）     │             │
│  │  ↓                                     │             │
│  │  同时为多个网络提供安全                 │             │
│  │  ├─ Chain: 15M 安全性                  │             │
│  │  ├─ Storage: 15M 安全性                │             │
│  │  ├─ DA: 15M 安全性                     │             │
│  │  └─ Compute: 15M 安全性                │             │
│  │                                        │             │
│  │  总质押：15M tokens                     │             │
│  │  安全性：所有网络共享                   │             │
│  └────────────────────────────────────────┘             │
│                                                          │
│  优势：                                                  │
│  • 资本效率提升 2x（15M vs 30M）                        │
│  • 验证者收益增加（多个网络的奖励）                     │
│  • 网络安全性提升（共享更大的质押池）                   │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 7.2.2 Shared Staking 工作流程

```
┌──────────────────────────────────────────────────────────┐
│          验证者参与 Shared Staking 流程                   │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  Step 1: 验证者质押到主网络（0G Chain）                  │
│    stake(amount: 100,000 0G)                             │
│    → 成为 0G Chain 验证者                                │
│                                                          │
│  Step 2: 选择参与的专网                                  │
│    registerForNetwork(networks: [Storage, DA, Compute])  │
│    → 声明愿意为这些网络提供服务                          │
│                                                          │
│  Step 3: 专网通过 VRF 选择验证者                         │
│    • Storage 网络：随机选择 500 个验证者                │
│    • DA 网络：选择 300 个验证者                         │
│    • Compute 网络：选择 200 个验证者                    │
│    （有重叠，同一验证者可能参与多个）                   │
│                                                          │
│  Step 4: 验证者执行多网络任务                            │
│    • 同时运行多个节点软件                               │
│    • Chain: 验证交易、出块                              │
│    • Storage: PoRA 挖矿                                 │
│    • DA: Quorum 验证                                    │
│    • Compute: AI 推理/训练                              │
│                                                          │
│  Step 5: 多网络奖励聚合                                  │
│    totalReward = chainReward + storageReward +           │
│                 daReward + computeReward                 │
│    → 质押收益率提升                                     │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 7.2.3 跨网络安全性

**关键问题：一个恶意验证者能攻击多个网络吗？**

**防御机制**：

```
1. 独立验证集
   • 每个专网有独立的验证者子集
   • 使用 VRF 随机选择
   • 子集有较大重叠，但不完全相同

2. 独立惩罚
   • 在 Storage 网络作恶 → 只削减 Storage 相关质押
   • 不影响其他网络的参与资格

3. 分层惩罚
   • 轻微违规：警告 + 小额罚款
   • 严重违规：大额罚款 + 临时暂停
   • 极端违规：永久封禁 + 没收全部质押

4. 声誉系统
   • 跟踪验证者在各网络的表现
   • 差评多的验证者被降权
   • 优秀验证者获得奖励加成
```

**攻击成本分析**：

```
假设攻击者想攻击 Storage 网络：

需要：
  • 控制 1/3+ 的 Storage 验证者
  • 假设 Storage 有 500 个验证者
  • 需要控制 167 个

成本：
  • 每个验证者质押 100,000 0G
  • 总成本：167 × 100,000 = 16,700,000 0G
  • 假设 0G = $0.1，成本 = $1,670,000

攻击成功后：
  • 质押被没收（损失 $1,670,000）
  • 被永久封禁
  • 0G 代币价格暴跌（攻击者持有的代币贬值）

结论：攻击成本 >> 潜在收益，不经济
```

## 7.3 网络间协调

### 7.3.1 跨网络通信

```
场景：Rollup 需要同时使用 Chain 和 DA

┌──────────────────────────────────────────────────────────┐
│          跨网络交互流程                                   │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  Rollup 提交交易批次：                                   │
│                                                          │
│  Step 1: 提交数据到 DA 网络                              │
│    blob = { transactions: [...], stateRoot: ... }        │
│    da_proof = DA_Network.submit(blob)                    │
│    → 获得 DA Proof (聚合签名)                            │
│                                                          │
│  Step 2: 提交 DA Proof 到 Chain                          │
│    tx = Chain.submitDAProof(da_proof)                    │
│    → Chain 验证 DA Proof                                 │
│    → 记录 Rollup 状态更新                                │
│                                                          │
│  Step 3: Chain 触发 Storage（可选）                      │
│    IF 数据需要永久存储：                                 │
│      Chain 发出事件                                      │
│      → Storage 网络监听                                  │
│      → 自动将 Blob 永久存储                              │
│                                                          │
│  关键：                                                  │
│  • 各网络独立运行                                       │
│  • 通过 Chain（主网络）协调                             │
│  • 事件驱动的松耦合                                     │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 7.3.2 冲突解决

```
问题：如果不同网络给出冲突的信息怎么办？

例子：
  • DA 网络说："数据已提交"
  • Storage 网络说："数据不存在"

解决机制：

1. 主网络仲裁
   • 0G Chain 作为"真相源"
   • 链上记录的 DA Proof 为准
   • Storage 网络必须同步

2. 挑战-响应
   • 任何人可以发起挑战
   • 被挑战方必须提供证明
   • 无法提供证明 → 惩罚

3. 多数投票
   • 多个验证者投票
   • 2/3 多数裁定真相

4. 超时机制
   • 如果专网长期不响应
   • 自动降级（临时停用）
   • 等待恢复或替换
```

### 7.4.2 运营成本

```
验证者成本分析：

  • 服务器：$5,000（CPU、内存、主板）
  • GPU：$50,000（A100）
  • 存储：$2,000（16TB SSD）
  • 网络：$500/月（10 Gbps）
  • 总硬件：$57,000

软件：
  • 节点软件：免费（开源）
  • 运维工具：$100/月

人力：
  • 兼职运维：$1,000/月

电费：
  • 24/7 运行：$800/月

总成本：
  • 初始：$57,000
  • 月度：$2,400

收益分析：
  • 月收入：6500 0G × $0.1 = $650
  • 月成本：$2,400
  • 月净利润：-$1,750（亏损）

问题？
  → 代币价格假设过低（$0.1）
  → 如果 0G = $1，月收入 = $6,500
  → 月净利润 = $4,100
  → 回本周期 = 14 个月
```
