## 6.1 Key-Value 运行时架构

### 基本接口

**标准 API**：
- `Put(key, value)`：更新或创建键值对
- `Get(key)`：获取键对应的值
- 接受任何应用特定结构的序列化键值对

### 工作原理

#### 状态同步机制

**本地状态维护**：
- Key-Value 节点在本地维护最新的键值状态
- 通过 `Put()` API 更新键值
- 更新操作：组成包含更新键值对的日志条目，追加到日志系统

**日志回放**：
- 运行时持续监控日志中的新条目
- 获取新条目并更新本地键值状态
- 多个 Key-Value 节点通过共享的去中心化日志同步

**一致性保证**：
- 所有节点播放相同的日志
- 得到相同的状态快照
- 保证最终一致性
  - **什么是最终一致性**：不要求每一刻所有节点都完全一致，但要求"只要不再有新更新，经过一段时间同步后，所有节点都会收敛到同一个状态"。适合分布式、高可用的场景；若要强一致性，需要额外机制（如事务+冲突检测）。

**类比**：
- 就像多个数据库副本
- 都从同一个事务日志恢复
- 最终状态一致

#### 反序列化

**用户定义函数**：
- 使用用户定义的函数反序列化日志条目中的原始内容
- 转换为应用特定的键值结构
- 允许不同应用使用不同的数据结构

#### 批量更新

**优化**：
- `Put()` 允许批量更新多个键值对
- 提高小键值对更新的效率
- 减少日志条目数量

### 访问控制

**所有权管理**：
- 每个键的所有权信息可以维护在日志条目中
- 所有诚实的 Key-Value 节点遵循基于所有权的相同更新规则
- 实现状态一致性

**应用场景**：
- **去中心化社交网络**：只有消息所有者可以修改/删除
- **去中心化文件系统**：只有文件所有者可以更新
- **任何需要权限管理的应用**

### 新节点加入

**启动过程**：
1. 新 Key-Value 节点连接到日志层
2. 从头部到尾部播放日志条目
3. 构建键值存储的最新状态

**优化**：
- 应用特定的 Key-Value 节点可以跳过不相关的日志条目
- 只处理包含关心的 stream ID 的条目
- 加快启动速度

## 6.2 事务处理：乐观并发控制

### 为什么需要事务处理？

**问题场景**：
- 多个用户同时更新同一个键
- 需要保证数据一致性
- 需要处理并发冲突

**例子**：Google Docs 协作编辑
- 用户 A 和用户 B 同时编辑同一段落
- 需要检测冲突
- 需要解决冲突

### 乐观并发控制机制

#### 事务生命周期

**1. 开始事务**：`BeginTx()`
- 通知运行时：事务将基于当前状态快照工作
- 状态快照通过播放日志到当前尾部构建
- 记录事务开始的日志位置

**2. 执行操作**：
- 在服务器本地更新键值
- **不暴露更新到日志**
- 所有操作都在本地缓冲区
- 其他节点看不到这些更新

**3. 提交事务**：`EndTx()`
- 运行时组成提交记录，包含：
  - 事务开始的日志位置
  - 事务的读写集（read-write set）
- 提交记录追加到日志

**4. 冲突检测**：
- 其他节点播放日志时，识别冲突窗口
- 冲突窗口 = 事务开始位置到提交位置之间的所有日志条目
- 检查并发操作是否更新了事务读取的键
- 如果有冲突 → 事务中止
- 如果没有冲突 → 事务提交

### 冲突窗口示例

**场景**：
```
日志位置: 1    2    3    4    5
          ↓    ↓    ↓    ↓    ↓
操作:    A    B    C    D    E

事务开始位置: #1
事务提交位置: #4
冲突窗口: 位置 1-4 的所有操作（A, B, C, D）
```

**冲突检测**：
- 如果事务的读集中包含的键在冲突窗口中被更新
- 事务中止（因为读取的数据已过期）
- 否则事务提交

### 安全假设

#### 协作模型

**假设**：
- 事务参与者是协作的
- 会诚实地在提交记录中填写正确的读写集

**适用场景**：
- **Google Docs**：用户通常信任共享访问的其他人
- **协作应用**：参与者通常是可信的
- **内部系统**：组织内部的应用

#### 不可信场景

**解决方案**：

1. **存储事务代码**：
   - 将事务代码存储在 0G 日志中
   - 所有人都可以看到代码
   - 可以验证执行是否正确

2. **可验证计算**：
   - 使用零知识证明（ZKP）验证事务执行
     - **什么是 ZKP**：零知识证明允许一方向另一方证明"某个陈述为真"，而不泄露陈述之外的信息。在这里可用于证明"事务按规则执行且读写集正确"，而不暴露全部数据。
   - 或使用可信执行环境（TEE）硬件
     - **什么是 TEE**：可信执行环境（如 Intel SGX、ARM TrustZone）在硬件里划出一块隔离区域，代码和数据在其中执行，外界难以窃取或篡改，可用于证明"在某段受保护环境中正确执行了事务"。
   - 事务执行者可以检测提交记录的有效性

**应用场景**：
- 需要强安全保证的应用
- 不可信环境
- 高价值交易

## 6.3 实际应用场景

### 场景 1：去中心化社交网络

**需求**：
- 用户可以发布消息
- 用户可以修改/删除自己的消息
- 需要维护消息所有权
- 需要时间线功能

**实现**：
- 使用 Key-Value 层
- 键 = 消息 ID
- 值 = 消息内容 + 元数据（作者、时间戳等）
- 访问控制确保只有所有者可以更新
- 时间线通过日志顺序实现

**优势**：
- 数据所有权明确
- 支持修改和删除
- 历史记录可追溯

### 场景 2：协作编辑（Google Docs）

**需求**：
- 多个用户同时编辑文档
- 需要检测和处理冲突
- 保证最终一致性
- 支持撤销/重做

**实现**：
- 使用事务处理层
- 每个编辑操作是一个事务
- 系统自动检测冲突
- 冲突时事务中止，用户可以重试

**工作流程**：
1. 用户 A 开始事务，读取文档
2. 用户 A 修改文档（本地）
3. 用户 A 提交事务
4. 如果用户 B 在冲突窗口内修改了相同部分 → 事务中止
5. 用户 A 可以重新读取并重试

### 场景 3：去中心化数据库

**需求**：
- 支持复杂的数据库操作
- ACID 特性（原子性、一致性、隔离性、持久性）
- 支持事务
- 支持查询

**实现**：
- 使用事务处理层
- 数据库操作作为事务执行
- 利用日志的总顺序保证一致性
- 支持复杂的查询和更新

**优势**：
- 去中心化
- 可验证
- 可扩展

### 场景 4：去中心化文件系统

**需求**：
- 文件存储和检索
- 文件元数据管理
- 访问控制
- 版本控制

**实现**：
- Log 层：存储文件内容
- Key-Value 层：管理文件元数据和索引
- 事务处理层：支持文件操作的原子性

**优势**：
- 永久存储
- 版本历史
- 访问控制

## 6.4 性能考虑

### 瓶颈分析

**Key-Value 层瓶颈**：
- 吞吐量和延迟受 Log 系统限制
- Log 层的效率对整个系统性能至关重要

**优化策略**：
- 批量更新：减少日志条目数量
- 选择性同步：跳过不相关的日志条目
- 本地缓存：减少日志回放时间

### 事务处理性能

**乐观并发控制的优势**：
- 无冲突时性能高（不需要锁定）
- 冲突时性能低（需要重试）

**适用场景**：
- 冲突较少的应用（如协作编辑）
- 可以容忍重试的应用

**不适用场景**：
- 冲突频繁的应用
- 需要强一致性的应用

## 总结

0G Storage 的高级功能提供了：

1. **Key-Value 存储**：
   - 可变数据管理
   - 访问控制
   - 状态同步

2. **事务处理**：
   - 乐观并发控制
   - 冲突检测
   - ACID 特性

3. **应用支持**：
   - 从简单到复杂
   - 从存储到数据库
   - 从单用户到多用户协作

这些功能使得 0G Storage 能够支持各种复杂的去中心化应用，同时保持高性能和可扩展性。
