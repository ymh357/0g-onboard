# 2. 0G Storage 系统架构深度解析

## 本章导读

0G Storage 是整个生态的核心组件之一，它不仅仅是一个去中心化存储系统，更是一个**分层的、可组合的数据基础设施**。本章将深入剖析 Log、KV、Transaction 三层架构的设计哲学与技术实现。

**学习目标**：
- 理解为什么需要三层架构，而不是单一存储层
- 掌握 Log 层的 Sector/Chunk、Merkle树、Erasure Coding 实现细节
- 理解 KV 层如何通过日志回放实现状态同步
- 深入理解 Transaction 层的乐观并发控制（OCC）机制
- 分析整个架构的设计权衡与适用场景

**前置知识**：建议先阅读第二部分教程的第3章（0G 存储系统：分层设计）

## 2.2 为什么：分层架构设计哲学

### 2.2.1 为什么需要分层？

**单层架构的问题**：

```
假设只有一个"通用存储层"：

问题 1：性能冲突
  - 大文件存储：需要高吞吐、顺序写入
  - 小键值更新：需要随机访问、低延迟
  → 单一架构难以同时优化两者

问题 2：一致性冲突
  - 不可变数据：不需要锁，可并行访问
  - 可变数据：需要并发控制，可能需要锁
  → 统一处理会引入不必要的开销

问题 3：应用适配性差
  - AI 训练：需要不可变的大数据集
  - 协作编辑：需要可变的文档状态
  - 数据库应用：需要事务保证
  → 单层难以满足多样化需求
```

**分层架构的优势**：

```
┌───────────────────────────────────────────────────────┐
│              分层架构的设计优势                        │
├───────────────────────────────────────────────────────┤
│                                                       │
│  1. 关注点分离 (Separation of Concerns)               │
│     每层专注于解决特定问题                            │
│                                                       │
│  2. 渐进式复杂度 (Progressive Complexity)             │
│     底层简单稳定，上层提供高级功能                    │
│                                                       │
│  3. 可组合性 (Composability)                          │
│     应用可选择需要的层，不必全部使用                  │
│                                                       │
│  4. 独立优化 (Independent Optimization)               │
│     每层可以独立优化，不影响其他层                    │
│                                                       │
│  5. 清晰的信任模型 (Clear Trust Model)                │
│     底层是"真相源"，上层是"派生状态"                 │
│                                                       │
└───────────────────────────────────────────────────────┘
```

### 2.2.2 为什么 Log 层是基础？

**Log 作为"真相源"（Source of Truth）**

```
核心思想：所有数据的最终真相都在 Log 中

类比：区块链的 Ledger
  - 区块链：交易日志 → 账户余额（派生）
  - 0G Storage：操作日志 → KV 状态（派生）

优势：
  ✓ 不可变：历史记录永久保留
  ✓ 有序：总顺序保证因果关系
  ✓ 可重放：任何状态都可以从日志重建
  ✓ 可审计：所有操作都有记录
```

**为什么是 Append-Only（只追加）？**

```
Append-Only 的好处：

1. 性能
   • 顺序写入极快（磁盘、SSD 最优场景）
   • 无需寻址开销
   • 可并行写入（不同 Chunk）

2. 简化设计
   • 无需处理"删除"、"更新"
   • 无需垃圾回收（或简化很多）
   • 无锁并发（写入不冲突）

3. 数据持久性
   • 永久保存（符合 AI 数据集需求）
   • 历史可追溯（审计、调试）
   • 灾难恢复（完整历史）

代价：
  ✗ 存储空间增长（但可以通过归档/压缩缓解）
  ✗ 删除困难（需要上层逻辑处理）
```

**Log 层的哲学**：

```
"Log 是数据的不可变历史，而非当前状态的快照"

这个设计受到启发于：
  - Apache Kafka（分布式日志）
  - Event Sourcing（事件溯源）
  - Blockchain（区块链账本）
```

### 2.2.3 为什么需要 KV 层？

**Log 层的局限**：

```
问题 1：查询效率低
  Log: "文件 X 在位置 12345"
  要找文件 X，需要扫描整个 Log（O(n)）

问题 2：无法表达"当前状态"
  Log: "Set A=1" → "Set A=2" → "Set A=3"
  用户想知道 A 的当前值，需要重放所有操作

问题 3：不支持复杂查询
  Log: 线性结构，不支持索引、范围查询等
```

**KV 层的价值**：

```
┌────────────────────────────────────────────────────┐
│           Key-Value 层解决的问题                   │
├────────────────────────────────────────────────────┤
│                                                    │
│  1. 快速查询 (O(1) 或 O(log n))                   │
│     key → value 直接查找                          │
│                                                    │
│  2. 当前状态维护                                   │
│     自动合并历史操作，只保留最新值                 │
│                                                    │
│  3. 索引与元数据                                   │
│     文件名 → 位置、权限、属性                     │
│                                                    │
│  4. 访问控制                                       │
│     key → owner → 权限检查                        │
│                                                    │
│  5. 应用特定逻辑                                   │
│     可以反序列化为应用需要的数据结构               │
│                                                    │
└────────────────────────────────────────────────────┘
```

**Log + KV 的关系**：

```
Log 层（真相源）
    ↓ 日志回放（Log Replay）
KV 层（物化视图 Materialized View）

类比数据库：
  - Log = Write-Ahead Log (WAL)
  - KV = Table/Index

好处：
  • KV 可以随时从 Log 重建（故障恢复）
  • 多个 KV 节点可以并行回放（扩展性）
  • KV 可以根据应用需求定制（灵活性）
```

### 2.2.4 为什么需要 Transaction 层？

**KV 层的局限**：

```
问题 1：并发冲突
  用户 A: Read(X)=5 → Write(X=6)
  用户 B: Read(X)=5 → Write(X=7)
  最终 X=? (可能是 6 或 7，丢失更新)

问题 2：原子性缺失
  操作："从账户 A 扣 100，给账户 B 加 100"
  如果中间失败，可能出现：
    A 扣了 100，但 B 没加（钱消失了）

问题 3：复杂业务逻辑
  例子：Google Docs 协作编辑
    - 多人同时编辑同一段落
    - 需要检测冲突并提示用户
```

**Transaction 层的价值**：

```
┌────────────────────────────────────────────────────┐
│            Transaction 层提供的保证                │
├────────────────────────────────────────────────────┤
│                                                    │
│  ACID 特性：                                       │
│                                                    │
│  • Atomicity (原子性)                              │
│    事务内的操作要么全部成功，要么全部失败          │
│                                                    │
│  • Consistency (一致性)                            │
│    事务执行前后，数据保持一致性约束                │
│                                                    │
│  • Isolation (隔离性)                              │
│    并发事务之间互不干扰                            │
│                                                    │
│  • Durability (持久性)                             │
│    事务一旦提交，结果永久保存                      │
│                                                    │
└────────────────────────────────────────────────────┘
```

**为什么选择乐观并发控制（OCC）？**

```
并发控制的两种策略：

1. 悲观锁 (Pessimistic Locking)
   策略：先获取锁 → 执行操作 → 释放锁
   优点：冲突时能保证正确性
   缺点：
     - 性能差（等待锁）
     - 可能死锁
     - 去中心化环境难以实现（谁持有锁？）

2. 乐观并发控制 (Optimistic Concurrency Control)
   策略：先执行 → 提交时检测冲突 → 有冲突则中止
   优点：
     ✓ 无冲突时性能高（无锁开销）
     ✓ 适合去中心化（无需中心化锁管理器）
     ✓ 适合冲突率低的场景
   缺点：
     ✗ 冲突时需要重试（浪费计算）
     ✗ 不适合高冲突场景

0G 的选择：OCC
  原因：大多数应用冲突率低（例如协作编辑的不同段落）
```

## 2.3 怎么做：Log 层深度剖析

### 2.3.1 Sector 与 Chunk：物理存储单元

#### 基本概念

```
┌───────────────────────────────────────────────────────┐
│            0G Storage 存储单元层级                     │
├───────────────────────────────────────────────────────┤
│                                                       │
│  Sector (扇区)                                        │
│    └─ 大小：256 Bytes                                │
│    └─ 作用：PoRA 挖矿的基本单位                       │
│                                                       │
│  Chunk (块)                                           │
│    └─ 大小：256 KB = 1024 Sectors                    │
│    └─ 作用：数据传输与存储的基本单位                  │
│                                                       │
│  File (文件)                                          │
│    └─ 大小：任意                                      │
│    └─ 分解为多个 Chunks                               │
│                                                       │
└───────────────────────────────────────────────────────┘
```

**为什么 Sector = 256B？**

```
设计考量：

1. PoRA 挖矿效率
   • 随机访问 256B 数据
   • 太小：IO 次数过多
   • 太大：读取浪费（只需要一个 Sector）
   • 256B 是 SSD 访问的合理粒度

2. 内存友好
   • 100MB Scratchpad = 409,600 Sectors
   • 便于缓存热点数据

3. 哈希计算
   • 256B 数据的哈希计算快速
   • 便于构建 Merkle 树
```

**为什么 Chunk = 256KB？**

```
设计考量：

1. 网络传输
   • 256KB 是 HTTP 请求的合理大小
   • 太小：请求次数过多，协议开销大
   • 太大：单次传输慢，失败重传代价高

2. 纠删码效率
   • Erasure Coding 在 Chunk 级别进行
   • 256KB 编码速度快，编码后大小可控

3. Merkle 树深度
   • 1GB 文件 = 4096 Chunks
   • Merkle 树深度 = log2(4096) = 12
   • 证明大小 = 12 * 32 bytes = 384 bytes（合理）
```

### 2.3.2 Merkle 树：数据完整性证明

#### Merkle 树结构

```
┌──────────────────────────────────────────────────────────┐
│          一个 Chunk 的 Merkle 树（简化版）                │
├──────────────────────────────────────────────────────────┤
│                                                          │
│                      Root                                │
│                  (Chunk Hash)                            │
│                       │                                  │
│            ┌──────────┴──────────┐                      │
│            │                     │                      │
│         H(0-511)              H(512-1023)               │
│            │                     │                      │
│      ┌─────┴─────┐         ┌─────┴─────┐              │
│      │           │         │           │              │
│   H(0-255)   H(256-511) H(512-767) H(768-1023)        │
│      │           │         │           │              │
│    ┌─┴─┐       ┌─┴─┐     ┌─┴─┐       ┌─┴─┐          │
│    │   │       │   │     │   │       │   │          │
│   ... ...     ... ...   ... ...     ... ...          │
│    │   │       │   │     │   │       │   │          │
│   S0  S1      S2  S3    S4  S5      S6  S7  ...     │
│  (Sector Hashes)                                      │
│                                                          │
│  深度 = log2(1024) = 10 层                              │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

#### Merkle 证明

**问题**：如何证明 Sector 5 确实属于这个 Chunk？

**Merkle Proof（Merkle 证明）**：

```
要证明 S5 (Sector 5) 在 Chunk 中，需要：

1. S5 的哈希：H(S5)
2. 从叶子到根的路径上的"兄弟节点"哈希：

   Siblings:
   ├─ H(S4)          (S5 的兄弟)
   ├─ H(S6-S7)       (父节点的兄弟)
   ├─ H(S0-S3)       (祖父节点的兄弟)
   ├─ H(S8-S15)      (曾祖父节点的兄弟)
   └─ ...            (直到根)

验证过程：
  Step 1: 计算 H(S5)
  Step 2: 结合 H(S4)，计算 H(S4-S5)
  Step 3: 结合 H(S6-S7)，计算 H(S4-S7)
  Step 4: 结合 H(S0-S3)，计算 H(S0-S7)
  ...
  Step 10: 得到 Root

如果计算出的 Root 等于已知的 Chunk Root，则证明 S5 确实在 Chunk 中。
```

**证明大小**：
```
证明大小 = 树深度 × 哈希大小
        = 10 × 32 bytes
        = 320 bytes

相比之下，传输整个 Chunk (256KB) 来验证一个 Sector，
Merkle Proof 只需要 320 bytes！
```

#### 完整文件的 Merkle 树

```
┌──────────────────────────────────────────────────────────┐
│              文件级别的 Merkle 树                         │
├──────────────────────────────────────────────────────────┤
│                                                          │
│                   File Root                              │
│                       │                                  │
│            ┌──────────┴──────────┐                      │
│            │                     │                      │
│       Chunk 0 Root          Chunk 1 Root                │
│            │                     │                      │
│       (256KB Chunk)         (256KB Chunk)               │
│            │                     │                      │
│      ┌─────┴─────┐         ┌─────┴─────┐              │
│   Sector      Sector     Sector      Sector            │
│   Merkle      Merkle     Merkle      Merkle            │
│   Tree        Tree       Tree        Tree              │
│                                                          │
│  双层 Merkle 树：                                        │
│    第一层：Sector → Chunk Root                          │
│    第二层：Chunk Root → File Root                       │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**验证任意 Sector**：
```
需要两个证明：
  1. Sector → Chunk Root 的 Merkle Proof (~320 bytes)
  2. Chunk Root → File Root 的 Merkle Proof (~384 bytes for 1GB file)

总计：~700 bytes 即可验证 1GB 文件中的任意 256B 数据！
```

### 2.3.3 Erasure Coding（纠删码）：冗余与容错

#### 什么是 Erasure Coding？

**核心思想**：将数据编码成 n 份，任意 k 份即可恢复原始数据（k < n）。

```
┌──────────────────────────────────────────────────────┐
│            Erasure Coding 原理                       │
├──────────────────────────────────────────────────────┤
│                                                      │
│  原始数据 (k 个 Chunk)                               │
│  ┌────┬────┬────┬────┐                             │
│  │ C0 │ C1 │ C2 │ C3 │ ...  (假设 k=10)            │
│  └────┴────┴────┴────┘                             │
│       │                                             │
│       ▼ Encoding                                    │
│  编码后 (n 个 Chunk)                                │
│  ┌────┬────┬────┬────┬────┬────┬────┬────┐        │
│  │ C0 │ C1 │ C2 │ C3 │ P0 │ P1 │ P2 │ P3 │ ...    │
│  └────┴────┴────┴────┴────┴────┴────┴────┘        │
│   Data Chunks       Parity Chunks  (假设 n=14)    │
│                                                      │
│  容错能力：可以丢失 n-k = 4 个 Chunk                 │
│                                                      │
│  任意 k=10 个 Chunk → 可以恢复原始数据               │
│                                                      │
└──────────────────────────────────────────────────────┘
```

#### Reed-Solomon 编码

0G Storage 使用 **Reed-Solomon 编码**，这是最经典的纠删码算法。

**数学原理（简化）**：

```
将数据视为多项式的系数：
  原始数据: [d0, d1, d2, ..., d(k-1)]
  构造多项式: P(x) = d0 + d1*x + d2*x^2 + ... + d(k-1)*x^(k-1)

编码：
  选择 n 个不同的点 x0, x1, ..., x(n-1)
  计算 P(x0), P(x1), ..., P(x(n-1))
  得到 n 个编码后的 Chunk

解码：
  给定任意 k 个点 (xi, P(xi))
  使用拉格朗日插值恢复多项式 P(x)
  从而恢复原始数据 [d0, d1, ..., d(k-1)]

关键性质：
  k 个点唯一确定一个 k-1 次多项式
  因此 n 个点中任意 k 个都可以恢复 P(x)
```

**实例**：

```
假设 k=2, n=4，原始数据 [3, 5]

构造多项式：P(x) = 3 + 5x

编码（选择 x=0,1,2,3）：
  P(0) = 3
  P(1) = 8
  P(2) = 13
  P(3) = 18

编码后：[3, 8, 13, 18]

丢失情况：假设丢失 P(1) 和 P(3)，只有 P(0)=3 和 P(2)=13

恢复：
  用 (0,3) 和 (2,13) 两点恢复直线
  P(x) = 3 + 5x
  得到原始数据 [3, 5]
```

#### 0G Storage 的编码参数

```
典型配置（示例）：
  k = 10 (原始 Chunks)
  n = 14 (编码后 Chunks)

冗余度 = n/k = 1.4 (140%)

对比：
  • 简单复制：3 副本 = 300% 冗余
  • 0G Erasure Coding：140% 冗余，但容错能力相同

存储效率：
  Erasure Coding 节省 (300% - 140%) / 300% ≈ 53% 的存储空间！
```

#### 编码与解码流程

**编码流程**（上传文件时）：

```
1. 文件分块
   File → [Chunk_0, Chunk_1, ..., Chunk_(k-1)]

2. 纠删码编码
   [C_0, ..., C_(k-1)] → Encode → [C_0, ..., C_(k-1), P_0, ..., P_(n-k-1)]

3. 分发存储
   将 n 个 Chunk 分发到不同的 Storage 节点
   每个节点存储部分 Chunk

4. Merkle 树构建（双层结构）
   • 第一层：每个 Chunk 内部构建 Merkle 树
     - 1024 个 Sectors → Chunk Root
   • 第二层：所有 Chunk Roots 构建 Merkle 树
     - N 个 Chunk Roots → File Root
```

**解码流程**（下载文件时）：

```
1. 查询元数据
   从 KV 层获取：哪些节点有哪些 Chunk

2. 并行请求
   向多个节点请求 Chunk
   目标：获取至少 k 个 Chunk

3. 验证完整性
   使用 Merkle Proof 验证每个 Chunk

4. 纠删码解码
   从 k 个 Chunk 恢复原始数据

5. 返回完整文件
```

### 2.3.4 数据分发机制

0G Storage 有两种数据分发流程：**Main Flow** 和 **Specialized Flow**。

#### Main Flow（主流程）

**适用场景**：大文件、AI 数据集、公共数据

**流程**：

```
┌──────────────────────────────────────────────────────────┐
│                   Main Flow 详解                          │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  1. 用户上传                                             │
│     ↓                                                    │
│  ┌────────────────┐                                     │
│  │  Sequencer     │ (接收数据，分配 Log Position)        │
│  │  (排序节点)    │                                     │
│  └──────┬─────────┘                                     │
│         │                                                │
│  2. 广播到 Quorum                                        │
│     ↓                                                    │
│  ┌────────────────┐                                     │
│  │  Quorum Nodes  │ (随机选择的验证节点子集)            │
│  │  (验证与见证)  │                                     │
│  └──────┬─────────┘                                     │
│         │ 验证数据有效性                                 │
│         │ BLS 签名                                       │
│         ▼                                                │
│  ┌────────────────┐                                     │
│  │  Aggregated    │ (聚合签名 = 多个节点的联合签名)      │
│  │  Signature     │                                     │
│  └──────┬─────────┘                                     │
│         │                                                │
│  3. 提交到 Chain                                         │
│     ↓                                                    │
│  ┌────────────────┐                                     │
│  │  0G Chain      │ (记录 Merkle Root + Signature)      │
│  │  (链上确认)    │                                     │
│  └──────┬─────────┘                                     │
│         │                                                │
│  4. 通知 Storage 节点                                    │
│     ↓                                                    │
│  ┌────────────────┐                                     │
│  │  Storage Nodes │ (从 Sequencer 或其他节点获取数据)   │
│  │  (永久存储)    │ (进行 PoRA 挖矿)                    │
│  └────────────────┘                                     │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**关键角色**：

```
Sequencer (排序节点)：
  • 负责接收上传数据
  • 分配 Log Position（全局递增的位置）
  • 协调数据分发

Quorum (验证子集)：
  • 通过 VRF 随机选择
  • 验证数据完整性
  • 提供 BLS 聚合签名（Data Availability 证明）

Storage Nodes (存储节点)：
  • 永久存储数据
  • 参与 PoRA 挖矿
  • 响应数据读取请求
```

#### Specialized Flow（专用流程）

**适用场景**：小文件、高频更新、需要快速确认的数据

**流程**：

```
┌──────────────────────────────────────────────────────────┐
│                Specialized Flow 详解                      │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  1. 用户直接提交到 Chain                                 │
│     ↓                                                    │
│  ┌────────────────┐                                     │
│  │  0G Chain      │ (智能合约调用)                       │
│  │  Tx 包含数据   │                                     │
│  └──────┬─────────┘                                     │
│         │                                                │
│  2. 链上确认                                             │
│     ↓                                                    │
│  ┌────────────────┐                                     │
│  │  Block         │ (数据包含在交易中)                   │
│  └──────┬─────────┘                                     │
│         │                                                │
│  3. Storage 节点监听事件                                 │
│     ↓                                                    │
│  ┌────────────────┐                                     │
│  │  Storage Nodes │ (从链上读取数据)                    │
│  │  提取并存储    │                                     │
│  └────────────────┘                                     │
│                                                          │
│  优势：                                                  │
│    • 快速确认（1 block = ~1秒）                         │
│    • 简化流程（无需 Sequencer）                         │
│                                                          │
│  劣势：                                                  │
│    • Gas 成本高（数据上链）                             │
│    • 受限于区块大小                                     │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

#### 两种流程对比

| 维度 | Main Flow | Specialized Flow |
|------|-----------|------------------|
| **适用数据** | 大文件 (>1MB) | 小数据 (<100KB) |
| **延迟** | 较高（需要 Quorum 验证） | 低（1 block） |
| **成本** | 低（链下存储） | 高（链上 Gas） |
| **可靠性** | 高（专门的存储网络） | 高（链上记录） |
| **典型场景** | AI 数据集、视频、备份 | 元数据、配置、日志 |

## 2.4 怎么做：KV 层深度剖析

### 2.4.1 状态同步机制

#### 日志回放（Log Replay）

**核心思想**：KV 层的状态是 Log 层的"物化视图"。

```
┌──────────────────────────────────────────────────────────┐
│                 日志回放机制                              │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  Log 层（不可变日志）                                    │
│  ┌────┬────┬────┬────┬────┬────┐                      │
│  │ E1 │ E2 │ E3 │ E4 │ E5 │ E6 │ ...                  │
│  └────┴────┴────┴────┴────┴────┘                      │
│    │    │    │    │    │    │                          │
│    │    │    │    │    │    │                          │
│    ▼    ▼    ▼    ▼    ▼    ▼                          │
│  回放（顺序执行）                                        │
│    │                                                    │
│    ▼                                                    │
│  KV 状态                                                 │
│  ┌──────────────────────────────────┐                 │
│  │  Key1 → Value_from_E3             │                 │
│  │  Key2 → Value_from_E6             │                 │
│  │  Key3 → Value_from_E1             │                 │
│  └──────────────────────────────────┘                 │
│                                                          │
│  规则：后面的操作覆盖前面的操作                          │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**日志条目（Log Entry）结构**：

```
每个 Log Entry 包含：
  • Stream ID：标识数据流（类似"主题"）
  • Data Payload：实际数据（可以是任意格式）
  • Timestamp：时间戳
  • User Signature：用户签名

示例（KV 更新）：
  Entry_5: {
    stream_id: "app_kvstore",
    data: {
      key: "user:alice:profile",
      value: {name: "Alice", age: 30},
      op: "PUT"
    },
    timestamp: 1700000000,
    signature: "0x..."
  }
```

**回放过程**：

```
KV 节点启动流程：

1. 连接到 Log 层
   ↓
2. 获取自己关心的 Stream ID 列表
   （例如：["app_kvstore", "app_metadata"]）
   ↓
3. 从头到尾遍历 Log
   ↓
4. 对于每个 Entry：
   IF Entry.stream_id 在关心的列表中：
     解析 Entry.data
     执行操作（PUT/DELETE）
     更新本地 KV 状态
   ELSE：
     跳过（不处理）
   ↓
5. 持续监听新 Entry
   增量更新状态
```

#### 并发回放优化

**挑战**：顺序回放慢（特别是初始同步）

**优化方案：并行回放**

```
观察：
  如果两个 Entry 操作不同的 Key，可以并行执行

┌──────────────────────────────────────────────────────────┐
│                  并行回放策略                             │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  Log 扫描                                                │
│  ┌────┬────┬────┬────┬────┬────┐                      │
│  │ E1 │ E2 │ E3 │ E4 │ E5 │ E6 │                      │
│  └────┴────┴────┴────┴────┴────┘                      │
│    │    │    │    │    │    │                          │
│    │    │    │    │    │    │                          │
│    ▼    ▼    ▼    ▼    ▼    ▼                          │
│  分析读写集 (Read-Write Set)                            │
│    E1: Write(Key_A)                                     │
│    E2: Write(Key_B)  ← 可与 E1 并行                    │
│    E3: Write(Key_A)  ← 必须在 E1 后                    │
│    E4: Write(Key_C)  ← 可与 E1,E2,E3 并行              │
│    ...                                                  │
│    │                                                    │
│    ▼                                                    │
│  构建依赖图 (Dependency Graph)                          │
│    E1 → E3                                              │
│    (E2, E4 独立)                                        │
│    │                                                    │
│    ▼                                                    │
│  并行执行无依赖的 Entry                                 │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐               │
│  │ Thread1 │  │ Thread2 │  │ Thread3 │               │
│  │  E1→E3  │  │   E2    │  │   E4    │               │
│  └─────────┘  └─────────┘  └─────────┘               │
│                                                          │
│  加速比：理想情况下接近线性                              │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**实现细节**：

```
1. 预处理阶段（扫描 Log）
   for each Entry in Log:
     analyze Entry.key
     record dependencies

2. 构建 DAG（有向无环图）
   nodes = Entries
   edges = dependencies (E_i must before E_j)

3. 拓扑排序 + 并行执行
   while DAG not empty:
     找到所有入度为 0 的节点（无依赖）
     并行执行这些节点
     从 DAG 中移除已执行节点
     更新其他节点的入度
```

#### 增量同步

**问题**：节点离线一段时间后重新上线，如何快速追赶？

**方案：增量同步（Incremental Sync）**

```
节点状态：
  • Last Synced Position: 12345

重新上线：
  1. 查询当前 Log Tail Position: 15000
  2. 只回放 [12346, 15000] 的 Entry
  3. 快速追上最新状态

优化：
  • Checkpoint（检查点）：
    定期保存 KV 状态快照
    新节点可以从最近的 Checkpoint 开始，而非从头回放

  例如：
    Checkpoint at Position 10000（包含完整 KV 状态）
    新节点：
      下载 Checkpoint → 只回放 [10001, current] → 追上
```

### 2.4.2 访问控制实现

#### 所有权模型

**核心概念**：每个 Key 有一个所有者（Owner）

```
┌──────────────────────────────────────────────────────────┐
│                  Key 的访问控制                           │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  Key 元数据结构：                                        │
│  {                                                       │
│    key: "user:alice:profile",                           │
│    value: { ... },                                      │
│    owner: "0xAlice_Address",  ← 所有者地址              │
│    permissions: {                                        │
│      read: ["public"],        ← 读权限：任何人          │
│      write: ["0xAlice_Address"] ← 写权限：仅所有者      │
│    },                                                    │
│    created_at: 1700000000,                               │
│    updated_at: 1700001000                                │
│  }                                                       │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

#### 权限验证流程

**写入验证**：

```
用户提交 PUT(key, value) 操作：

1. 日志条目包含签名
   Entry = {
     op: "PUT",
     key: "user:alice:profile",
     value: {...},
     signature: Sign(Entry, Alice_PrivateKey)
   }

2. KV 节点回放时验证
   ├─ 提取 signer_address = Verify(Entry.signature)
   ├─ 查询 key 的 owner
   │  IF key 不存在：
   │    owner = signer_address（首次创建，签名者成为所有者）
   │  ELSE：
   │    owner = existing_owner
   ├─ 检查权限
   │  IF signer_address == owner OR signer_address in permissions.write：
   │    允许操作
   │  ELSE：
   │    拒绝操作（忽略这个 Entry）
   └─ 更新状态（如果允许）

3. 所有诚实节点执行相同的验证逻辑
   → 状态一致
```

**读取权限**：

```
读取操作通常不需要严格验证（数据公开）

但可以实现"私密 KV"：
  • value 存储加密数据
  • 只有拥有密钥的用户能解密
  • KV 层仍能同步（但看不到明文）
```

#### 多级权限（扩展）

```
更复杂的权限模型（可选）：

permissions: {
  owner: "0xAlice",
  read: ["public"],               // 任何人可读
  write: ["0xAlice", "0xBob"],   // Alice 和 Bob 可写
  admin: ["0xAlice"]              // 只有 Alice 能修改权限
}

权限继承（路径前缀）：
  "folder/file1" 继承 "folder/" 的权限
```

#### Stream 过滤优化

**问题**：Log 中可能有很多不相关的数据（不同应用的 Stream）

**优化：Stream ID 过滤**

```
应用特定的 KV 节点：

配置：
  interested_streams = ["app_kvstore", "app_config"]

回放优化：
  FOR each Entry：
    IF Entry.stream_id NOT IN interested_streams：
      跳过（不下载、不处理）
    ELSE：
      正常处理

好处：
  • 减少网络带宽（不下载无关数据）
  • 减少存储空间（不存储无关状态）
  • 加速启动（处理更少的 Entry）
```

## 2.5 怎么做：Transaction 层深度剖析

### 2.5.1 乐观并发控制（OCC）完整实现

#### OCC 核心原理

**三阶段协议**：

```
┌──────────────────────────────────────────────────────────┐
│          乐观并发控制（OCC）三阶段                        │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  Phase 1: Read (读取)                                    │
│  ┌────────────────────────────────────────────┐         │
│  │  • 开始事务：BeginTx()                     │         │
│  │  • 在当前状态快照上工作                    │         │
│  │  • 记录开始时的 Log Position               │         │
│  │  • 执行读取操作，构建读集 (Read Set)       │         │
│  └────────────────────────────────────────────┘         │
│                                                          │
│  Phase 2: Validate (验证)                                │
│  ┌────────────────────────────────────────────┐         │
│  │  • 本地执行写操作（buffered）              │         │
│  │  • 记录写集 (Write Set)                    │         │
│  │  • 准备提交记录                            │         │
│  └────────────────────────────────────────────┘         │
│                                                          │
│  Phase 3: Write (写入)                                   │
│  ┌────────────────────────────────────────────┐         │
│  │  • 提交事务：EndTx()                       │         │
│  │  • 将事务记录追加到 Log                    │         │
│  │  • 其他节点回放时进行冲突检测              │         │
│  │  • 如果冲突 → 事务中止                     │         │
│  │  • 如果无冲突 → 事务提交                   │         │
│  └────────────────────────────────────────────┘         │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

#### 详细流程

**Step 1: BeginTx()**

```
用户代码：
  tx = BeginTx()
  // 事务开始

内部逻辑：
  {
    tx_id: generate_unique_id(),
    start_log_position: current_log_tail,  // 记录起始位置
    read_set: {},                           // 将要填充
    write_set: {},                          // 将要填充
    state_snapshot: clone(current_kv_state) // 本地快照（可选，视实现）
  }

关键点：
  • start_log_position 记录了事务"看到的世界"
  • 事务在这个快照上工作，不受后续更新影响
```

**Step 2: 执行操作（Read & Write）**

```
用户代码：
  value = tx.Get("key1")       // 读取
  tx.Put("key1", new_value)    // 写入
  tx.Put("key2", another_value)

内部逻辑：

tx.Get("key1")：
  • 从本地快照读取
  • 记录到 read_set: {"key1": old_value}

tx.Put("key1", new_value)：
  • 写入本地 buffer（不立即提交到 Log）
  • 记录到 write_set: {"key1": new_value}
  • 如果再次 Get("key1")，返回 new_value（读自己的写）

重要：所有操作都在本地，其他事务看不到！
```

**Step 3: EndTx() 提交**

```
用户代码：
  tx.Commit()  // 或 tx.EndTx()

内部逻辑：
  1. 组装事务记录
     tx_record = {
       tx_id: "tx_123",
       start_position: 10000,
       end_position: 10050,  // 当前 Log Tail（提交时）
       read_set: {"key1": hash(old_value)},  // 读了哪些 key
       write_set: {"key1": new_value, "key2": another_value},
       signature: sign(tx_record, user_private_key)
     }

  2. 追加到 Log
     Log.append(tx_record)

  3. 等待确认
     位置 = 10051（例如）

  4. 返回给用户
     "Transaction pending, position=10051"

此时事务还未真正提交！需要等待 KV 节点回放并验证。
```

#### 冲突检测

**核心问题**：两个事务是否冲突？

**冲突窗口（Conflict Window）**：

```
Tx1 的冲突窗口 = [start_position, end_position]

例子：
  Tx1: start=10000, end=10050
  冲突窗口 = [10000, 10050]

检测规则：
  如果 Tx1 的 read_set 中的 key 在冲突窗口内被其他事务修改，
  则 Tx1 冲突，必须中止。
```

**验证算法**（KV 节点回放时执行）：

```
当 KV 节点回放到 tx_record：

FUNCTION validate_transaction(tx):
  conflict_window = [tx.start_position, tx.end_position]

  FOR each key IN tx.read_set:
    // 检查冲突窗口内是否有其他写操作修改了这个 key
    FOR each entry IN Log[conflict_window]:
      IF entry 是另一个事务的提交记录:
        IF key IN entry.write_set:
          RETURN ABORT  // 冲突！读的数据已过期

      IF entry 是普通 KV 更新（非事务）:
        IF entry.key == key:
          RETURN ABORT  // 冲突！

  // 无冲突，提交事务
  FOR each (key, value) IN tx.write_set:
    KV_state[key] = value

  RETURN COMMIT
```

**示例：冲突情况**

```
时间线：

Position 10000: Tx1 开始，读取 key="X"，X=5
Position 10010: Tx2 提交，写入 key="X"，X=10
Position 10020: Tx1 提交

冲突检测（在 Position 10020）：
  Tx1.start = 10000
  Tx1.end = 10020
  Tx1.read_set = {"X": 5}

  检查冲突窗口 [10000, 10020]:
    Position 10010: Tx2 写入了 X=10

  结论：Tx1 读取的 X=5 已经过期（被 Tx2 改成 10）
        → Tx1 ABORT

用户需要重试：
  重新 BeginTx()
  此时 X=10（读到最新值）
  重新执行业务逻辑
  再次 Commit
```

**示例：无冲突情况**

```
Position 10000: Tx1 开始，读取 key="X"，X=5，写入 key="Y"
Position 10010: Tx2 开始，读取 key="Z"，Z=3，写入 key="W"
Position 10020: Tx1 提交
Position 10030: Tx2 提交

Tx1 冲突检测（Position 10020）：
  冲突窗口 [10000, 10020]
  Tx1.read_set = {"X"}
  窗口内无其他事务修改 X
  → Tx1 COMMIT

Tx2 冲突检测（Position 10030）：
  冲突窗口 [10010, 10030]
  Tx2.read_set = {"Z"}
  窗口内无修改 Z（Tx1 只修改了 Y）
  → Tx2 COMMIT

两个事务都成功！因为它们操作的是不同的 key。
```

### 2.5.2 读写集（Read-Write Set）

**Read Set**：事务读取过的所有 key 及其哈希

```
目的：检测"读取的数据是否被修改"

结构：
  read_set = {
    "key1": hash(value1),
    "key2": hash(value2),
    ...
  }

为什么存哈希而不是完整值？
  • 减少事务记录大小
  • 隐私保护（不暴露读取的具体值）
```

**Write Set**：事务写入的所有 key 及其新值

```
目的：提交后应用这些更新

结构：
  write_set = {
    "key1": new_value1,
    "key2": new_value2,
    ...
  }

注意：Write Set 包含完整的新值（需要更新状态）
```

#### 优化：Bloom Filter

**问题**：冲突检测需要逐个检查 read_set 中的每个 key

**优化**：使用 Bloom Filter 快速判断"一定不冲突"

```
Bloom Filter:
  • 概率型数据结构
  • 可能误报（False Positive），但不会漏报（No False Negative）

应用：
  为 read_set 构建 Bloom Filter

  冲突检测：
    FOR each entry IN conflict_window:
      IF entry.key NOT IN bloom_filter:
        跳过（一定不冲突）
      ELSE:
        详细检查（可能冲突）

  好处：大部分情况下快速跳过无关 Entry
```

### 2.5.4 安全假设与信任模型

#### 核心问题：为什么需要信任模型？

**OCC 的隐藏假设**：

```
乐观并发控制（OCC）依赖一个关键假设：
事务提交的 read_set 是诚实的

如果用户可以伪造 read_set，整个系统的一致性就会被破坏！
```

#### 攻击场景：伪造 read_set 的危害

**具体例子：银行转账**

```
┌──────────────────────────────────────────────────────────┐
│              恶意事务：隐瞒 read_set                      │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  初始状态：                                              │
│    Alice 账户余额：balance_alice = 100                   │
│    Bob 账户余额：balance_bob = 50                        │
│                                                          │
│  恶意事务 Tx1（由黑客提交）：                            │
│    实际执行的代码：                                      │
│      if balance_alice > 50:   // 🔴 读取了 balance_alice│
│          balance_bob += 1000  // 🔴 写入 balance_bob    │
│                                                          │
│    提交的事务记录（伪造）：                              │
│      read_set = {}            // ❌ 声称没有读任何 key  │
│      write_set = {                                       │
│        "balance_bob": 1050    // 只声称写入 Bob 余额    │
│      }                                                   │
│                                                          │
│  问题：                                                  │
│    • 实际上事务读取了 balance_alice                      │
│    • 但 read_set 中没有声明                             │
│    • 如果此时另一个事务修改了 balance_alice，           │
│      Tx1 应该被中止（因为读到了过期数据）               │
│    • 但因为 read_set = {}，冲突检测无法发现！           │
│                                                          │
│  结果：                                                  │
│    ❌ Bob 的余额凭空增加 1000（违反一致性）             │
│    ❌ 系统无法检测到这个不一致                          │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**为什么这个攻击有效？**

```
1. OCC 的冲突检测只检查 read_set 中声明的 key
2. 如果事务隐瞒了实际读取的 key，冲突检测就会漏掉真实的冲突
3. 恶意事务可以基于"过期数据"做决策，但系统认为"无冲突"
4. 最终导致数据库状态不一致（违反隔离性 Isolation）
```

#### 防御方案对比

##### 方案 1：重新执行事务代码（Code Re-execution）

**核心思想**：验证者重新执行事务代码，验证 read_set 是否匹配

```
┌──────────────────────────────────────────────────────────┐
│                 代码重新执行流程                          │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  【用户端】                                              │
│  1. 编写事务代码（WebAssembly）                          │
│     function transfer(from, to, amount) {                │
│         balance_from = kv_get(from)     // 读取 from 余额│
│         if balance_from >= amount:                       │
│             kv_put(from, balance_from - amount)          │
│             kv_put(to, kv_get(to) + amount)              │
│         else:                                            │
│             abort()                                      │
│     }                                                    │
│                                                          │
│  2. 执行代码，记录 read_set                              │
│     执行 transfer("alice", "bob", 50)                    │
│     实际 read_set = {"alice": hash(100), "bob": hash(50)}│
│                                                          │
│  3. 提交到 Log                                           │
│     tx_record = {                                        │
│         code_hash: hash(transfer),  // 代码哈希         │
│         inputs: ["alice", "bob", 50],                    │
│         read_set: {"alice": hash(100), "bob": hash(50)}, │
│         write_set: {"alice": 50, "bob": 100}             │
│     }                                                    │
│                                                          │
│  【验证者端】                                            │
│  4. 下载代码（通过 code_hash）                           │
│     code = fetch_from_storage(code_hash)                 │
│                                                          │
│  5. 重新执行                                             │
│     // 在相同的起始状态下重新运行                        │
│     snapshot = get_state_at(tx.start_position)           │
│     actual_result = execute(code, inputs, snapshot)      │
│                                                          │
│  6. 验证 read_set                                        │
│     IF actual_result.read_set == tx.read_set:            │
│         接受事务 ✓                                       │
│     ELSE:                                                │
│         拒绝事务 ✗（read_set 造假）                     │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**优点与缺点**：

```
✓ 优点：
  • 完全可验证（任何人都能重新执行检查）
  • 不依赖硬件（纯软件方案）
  • 透明性高（代码公开）

✗ 缺点：
  • 计算成本高：每个验证者都要重新执行
    例如：100 个验证者 = 重复执行 100 次
  • 确定性要求：代码必须是确定性的
    不能用 random()、time() 等非确定性函数
  • 隐私问题：代码和数据都是公开的
```

**适用场景**：

```
• 简单事务（执行快）
• 低频交易（不需要每秒处理大量事务）
• 对透明性要求高的场景（公共服务）
```

##### 方案 2：零知识证明（Zero-Knowledge Proof, ZKP）

**核心思想**：用数学证明"我诚实执行了，但不告诉你细节"

```
┌──────────────────────────────────────────────────────────┐
│                 零知识证明流程                            │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  【用户端】                                              │
│  1. 执行事务（正常流程）                                 │
│     balance_alice = kv_get("alice")  // = 100            │
│     balance_bob = kv_get("bob")      // = 50             │
│     kv_put("alice", 50)                                  │
│     kv_put("bob", 100)                                   │
│                                                          │
│  2. 生成零知识证明                                       │
│     证明声明：                                           │
│       "我执行了代码 C，读取了 keys [alice, bob]，        │
│        它们的哈希分别是 [H1, H2]，                      │
│        并且写入了新值 [50, 100]，                        │
│        这一切都是正确执行的结果"                         │
│                                                          │
│     proof = ZKP_prove(                                   │
│         code: transfer,                                  │
│         inputs: ["alice", "bob", 50],                    │
│         read_set: {"alice": H1, "bob": H2},              │
│         write_set: {"alice": 50, "bob": 100}             │
│     )                                                    │
│                                                          │
│  3. 提交 proof（大小约 1KB，远小于完整代码）             │
│                                                          │
│  【验证者端】                                            │
│  4. 验证 proof（只需几毫秒）                             │
│     IF ZKP_verify(proof, public_inputs) == TRUE:         │
│         接受事务 ✓                                       │
│         "相信 read_set 是诚实的，无需重新执行"           │
│     ELSE:                                                │
│         拒绝事务 ✗                                       │
│                                                          │
│  关键：验证者不需要知道执行细节，只需验证证明有效        │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**类比理解**：

```
类比：证明"我知道密码"，但不告诉你密码是什么

传统方式：
  你："密码是什么？"
  我："是 'abc123'"
  你验证："对，是这个密码"
  → 你知道了密码（隐私泄露）

零知识证明：
  我："我生成一个 proof，证明我知道密码"
  你验证 proof："这个 proof 有效，所以他确实知道密码"
  → 你不知道密码是什么（隐私保护）

应用到事务：
  我："我生成 proof，证明 read_set 是诚实的"
  验证者："proof 有效，所以 read_set 是诚实的"
  → 验证者不需要重新执行事务
```

**优点与缺点**：

```
✓ 优点：
  • 验证极快（几毫秒）
    → 100 个验证者也只需验证 100 次 proof（每次几毫秒）
  • 隐私保护（不暴露代码和数据细节）
  • Proof 体积小（~1KB），传输高效

✗ 缺点：
  • 生成 proof 很慢（用户端可能需要几秒到几分钟）
  • 技术复杂度极高（需要深厚的密码学知识）
  • 硬件要求高（需要强大 CPU 或专用硬件加速）
```

**适用场景**：

```
• 高价值交易（生成 proof 的成本可接受）
• 隐私要求高的场景（医疗、金融）
• 高频验证（验证快 → 支持大量验证者）
```

---

##### 方案 3：可信执行环境（Trusted Execution Environment, TEE）

**核心思想**：在硬件隔离的"黑盒"里执行，硬件保证诚实

```
┌──────────────────────────────────────────────────────────┐
│                 TEE 执行流程                              │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  【硬件层：Intel SGX / ARM TrustZone】                   │
│                                                          │
│  ┌────────────────────────────────────┐                 │
│  │  Enclave（飞地）                   │                 │
│  │  ┌──────────────────────────────┐  │                 │
│  │  │  执行区域（硬件隔离）        │  │                 │
│  │  │  • CPU 强制加密内存          │  │                 │
│  │  │  • 操作系统无法访问          │  │                 │
│  │  │  • 恶意软件无法读取          │  │                 │
│  │  │                              │  │                 │
│  │  │  【事务代码在这里运行】      │  │                 │
│  │  │  balance = read("alice")     │  │                 │
│  │  │  write("bob", new_value)     │  │                 │
│  │  │                              │  │                 │
│  │  └──────────────────────────────┘  │                 │
│  │                                    │                 │
│  │  输出：                             │                 │
│  │  • read_set                        │                 │
│  │  • write_set                       │                 │
│  │  • 数字签名（由 CPU 的私钥签名）    │                 │
│  │                                    │                 │
│  └────────────────────────────────────┘                 │
│                                                          │
│  【验证者】                                              │
│  1. 收到事务 + TEE 签名                                  │
│  2. 验证签名：                                           │
│     • 签名来自真实的 Intel SGX CPU？                    │
│     • 代码哈希匹配？                                     │
│     • 输入输出一致？                                     │
│  3. 如果签名有效 → 信任 read_set                        │
│                                                          │
│  核心：硬件保证"代码在隔离环境中正确执行"                │
│        验证者信任硬件，而非信任用户                      │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**远程证明（Remote Attestation）详解**：

```
为什么验证者要相信 TEE 的签名？

1. CPU 内置私钥
   • Intel/ARM 在制造 CPU 时烧录私钥
   • 私钥存储在 CPU 内部，无法提取
   • 对应的公钥由 Intel/ARM 官方认证

2. 远程证明流程
   TEE 输出：
   {
       code_hash: "0xabc...",     // 运行的代码哈希
       read_set: {...},
       write_set: {...},
       signature: Sign(以上内容, CPU_PrivateKey)
   }

   验证者：
   • 用 Intel/ARM 的公钥验证签名
   • 签名有效 → 证明输出确实来自真实的 SGX/TrustZone
   • 并且代码是 code_hash 对应的代码
   • 并且没有被篡改

3. 信任链
   验证者信任 Intel/ARM 的硬件 → 信任 TEE 的输出
```

**优点与缺点**：

```
✓ 优点：
  • 执行速度快（接近原生性能，无需 ZKP 生成开销）
  • 用户体验好（无需等待 proof 生成）
  • 验证快速（只验证签名，几毫秒）

✗ 缺点：
  • 硬件依赖（必须有 SGX/TrustZone）
    → 限制了用户硬件选择
  • 中心化风险：信任 Intel/ARM 公司
  • 硬件漏洞：SGX 曾被攻破（Spectre、Foreshadow 等）
  • 成本：SGX 服务器比普通服务器贵
```

**适用场景**：

```
• 对性能要求极高的场景（高频交易）
• 企业环境（可以强制用户使用 TEE 硬件）
• 混合方案：TEE + 抽查验证（降低硬件依赖风险）
```

---

#### 三种方案对比总结

| 维度 | 代码重新执行 | 零知识证明（ZKP） | 可信执行环境（TEE） |
|------|------------|-----------------|-------------------|
| **验证速度** | 慢（需重新执行） | 快（几毫秒） | 快（验签） |
| **用户生成成本** | 低（正常执行） | 高（生成 proof） | 低（TEE 执行） |
| **硬件要求** | 无 | 高（CPU 密集） | 必须（SGX/TrustZone） |
| **隐私保护** | 差（代码公开） | 优秀 | 良好 |
| **信任假设** | 无（纯数学） | 无（密码学） | 信任硬件厂商 |
| **去中心化** | 优秀 | 优秀 | 中等（硬件中心化） |
| **适合场景** | 低频简单事务 | 高价值隐私交易 | 高频企业应用 |

```
┌──────────────────────────────────────────────────────────┐

├──────────────────────────────────────────────────────────┤
│                                                          │

│                                                          │

```
