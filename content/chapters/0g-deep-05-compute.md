# 5. 0G Compute Network 深度解析

## 本章导读

0G Compute Network 是生态中的"计算大脑"，为 AI 推理和训练提供去中心化的算力服务。本章将深入剖析如何通过 TEE（可信执行环境）和 Serving Broker 实现可信、高效的 AI 计算网络。

**学习目标**：
- 理解去中心化 AI 计算的核心挑战
- 掌握 TEE（可信执行环境）的工作原理
- 深入理解 Serving Broker 的请求路由与负载均衡
- 分析推理服务和微调服务的完整流程
- 对比 0G Compute 与 Akash、Render、Gensyn 的差异

---

### 5.0.9 模型在 0G Compute 中的角色

```
┌──────────────────────────────────────────────────────────┐
│            0G Compute 中的"模型"                          │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  推理服务：                                              │
│    • Provider 加载"GPT-4 模型"                           │

│    • 用户发送输入 → Provider 用模型计算 → 返回输出      │
│                                                          │
│  微调服务：                                              │
│    • 用户提供"基础模型"（如 LLaMA-3-8B）                │
│    • 用户提供训练数据                                   │
│    • Provider 训练 → 生成"微调后的模型"                 │
│    • 新模型保存到 0G Storage                             │
│                                                          │
│  模型即资产：                                            │
│    • 模型是有价值的文件                                 │
│    • 训练成本高（几万美元到几百万美元）                 │
│    • 需要安全存储（0G Storage）                          │
│    • 需要可信计算（0G Compute + TEE）                    │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**核心总结**：

> **模型 = 一个经过训练的"智能程序"，能把输入转换为有用的输出。0G Compute 提供去中心化的基础设施来运行和训练这些模型。**

## 5.2 为什么：去中心化计算的挑战与方案

### 5.2.1 为什么 AI 计算难以去中心化？

**核心挑战**：

```
挑战 1：性能要求高
  • AI 推理：需要 GPU
  • 训练：需要大量 GPU 协同
  • 去中心化网络延迟高

挑战 2：结果可验证性
  • 如何证明 Provider 诚实执行？
  • 如何防止 Provider 返回错误结果？

挑战 3：模型隐私
  • 用户不想暴露模型参数
  • Provider 不想泄露推理数据
```

### 5.2.2 为什么使用 TEE？

**TEE（Trusted Execution Environment）** = 可信执行环境

```
┌──────────────────────────────────────────────────────────┐
│              TEE 核心原理                                 │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  TEE = 硬件隔离的安全区域                                │
│                                                          │
│  ┌────────────────────────────────────┐                │
│  │       普通操作系统 (Untrusted)      │                │
│  │                                    │                │
│  │  ┌──────────────────────────┐     │                │
│  │  │   TEE (Trusted Zone)     │     │                │
│  │  │  • 隔离的内存            │     │                │
│  │  │  • 加密执行              │     │                │
│  │  │  • 防篡改                │     │                │
│  │  └──────────────────────────┘     │                │
│  └────────────────────────────────────┘                │
│                                                          │
│  特性：                                                  │
│  • 外部无法窃取 TEE 内的数据                            │
│  • 外部无法篡改 TEE 内的执行                            │
│  • TEE 可以生成"远程证明"（Remote Attestation）         │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**常见 TEE 技术**：

```
Intel SGX（Software Guard Extensions）：
  • CPU 级别隔离
  • 内存加密
  • Enclave（安全飞地）

AMD SEV（Secure Encrypted Virtualization）：
  • 虚拟机级别隔离
  • 内存加密

ARM TrustZone：
  • 硬件分区
  • 移动设备常用

NVIDIA Confidential Computing：
  • GPU TEE
  • 保护 AI 计算
```

**远程证明（Remote Attestation）**：

```
┌──────────────────────────────────────────────────────────┐
│            远程证明流程                                   │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  用户想验证："Provider 真的在 TEE 中执行了我的任务"      │
│                                                          │
│  Step 1: Provider 生成证明                               │
│    attestation = TEE_attest({                            │
│      code_hash: hash(推理代码),                          │
│      input_hash: hash(用户输入),                         │
│      output: 推理结果,                                   │
│      TEE_signature: TEE 私钥签名                         │
│    })                                                    │
│                                                          │
│  Step 2: 用户验证证明                                    │
│    • 检查 TEE_signature（确保来自真实 TEE）             │
│    • 检查 code_hash（确保运行了正确的代码）             │
│    • 检查 input_hash（确保输入未被篡改）                │
│    • 信任 output（因为 TEE 保证了执行正确性）           │
│                                                          │
│  安全保证：                                              │
│  • Provider 无法伪造 attestation                        │
│  • 外部攻击者无法篡改 TEE 内执行                        │
│  • 用户获得密码学级别的信任                             │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 5.2.3 为什么需要 Serving Broker？

**问题：用户如何找到合适的 Provider？**

```
无 Broker 的情况：
  • 用户需要手动发现 Provider
  • 不知道哪个 Provider 可用、性能如何
  • 每个用户都要管理 Provider 列表

有 Broker 的优势：
  • 统一入口
  • 智能路由（根据负载、性能、价格）
  • 负载均衡
  • 故障转移
  • 计费管理
```

**Serving Broker 架构**：

```
┌──────────────────────────────────────────────────────────┐
│            Serving Broker 职责                            │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  用户 → Broker → Provider                                │
│                                                          │
│  Broker 功能：                                           │
│                                                          │
│  1. 服务发现（Service Discovery）                        │
│     • 维护 Provider 列表                                │
│     • 监控 Provider 状态（在线/离线）                   │
│                                                          │
│  2. 请求路由（Request Routing）                          │
│     • 根据模型类型选择 Provider                         │
│     • 根据负载选择最空闲的 Provider                     │
│                                                          │
│  3. 负载均衡（Load Balancing）                           │
│     • 轮询（Round-Robin）                               │
│     • 最少连接（Least Connections）                     │
│     • 加权路由（Weighted Routing）                      │
│                                                          │
│  4. 计费管理（Billing）                                  │
│     • 记录每次请求的计算量                              │
│     • 计算费用                                          │
│     • 与链上结算                                        │
│                                                          │
│  5. 质量监控（Quality Monitoring）                       │
│     • 跟踪 Provider 性能（延迟、成功率）                │
│     • 声誉评分                                          │
│     • 差评 Provider 降权                                │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

  • 硬件：$50,000（A100 GPU）

ROI 估算（粗略）：
  假设每天处理 10,000 次推理
  收入 = 10,000 × $0.01 × 1.2（奖金）= $120/天 = $3,600/月
  净利润 = $3,600 - $600 = $3,000/月
  回本周期 ≈ 17 个月

### 5.4.2 分布式训练协调

在深入分布式训练前，我们先理解训练中的关键概念。

#### 超参数（Hyperparameters）详解

**超参数 = 训练过程的"配置参数"**（不是模型学到的参数）

```
┌──────────────────────────────────────────────────────────┐
│            超参数：训练的控制旋钮                          │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  超参数 vs 模型参数：                                    │
│  ─────────────────                                       │
│                                                          │
│  模型参数（Model Parameters）：                          │
│    • 模型"学到"的权重和偏置                             │
│    • 例如：神经网络的权重矩阵 W、偏置 b                 │
│    • 数量：数百万到数十亿（如 GPT-3 有 1750 亿参数）    │
│    • 由训练过程自动学习                                 │
│                                                          │
│  超参数（Hyperparameters）：                             │
│    • 训练前人工设定的"配置"                             │
│    • 控制训练的速度、方式、时长                         │
│    • 不被训练过程改变                                   │
│    • 需要人工调优（或自动调参）                         │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**常见超参数详解**：

```
┌──────────────────────────────────────────────────────────┐
│            1. Learning Rate（学习率）                     │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  定义：每次更新模型参数时的"步长"                        │
│                                                          │
│  类比：                                                  │
│    想象你在山上找最低点（最优解）                        │
│    • 学习率太大 = 步子太大，可能跳过最优点              │
│    • 学习率太小 = 步子太小，训练太慢                    │
│    • 合适的学习率 = 稳定下降，快速收敛                  │
│                                                          │
│  常见取值：                                              │
│    • 1e-5 = 0.00001（微调大模型）                       │
│    • 1e-4 = 0.0001（常规训练）                          │
│    • 1e-3 = 0.001（小模型）                             │
│                                                          │
│  示例：                                                  │
│    new_weight = old_weight - learning_rate × gradient    │
│                                                          │
│    如果 learning_rate = 0.01，gradient = 0.5：           │
│    new_weight = old_weight - 0.01 × 0.5 = old_weight - 0.005│
│                                                          │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│            2. Epochs（训练轮数）                          │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  定义：完整遍历训练数据集的次数                          │
│                                                          │
│  例如：                                                  │
│    训练数据有 10,000 条                                  │
│    epochs = 3                                            │
│    → 模型会看到每条数据 3 次                            │
│                                                          │
│  常见取值：                                              │
│    • 3-5 epochs（微调）                                 │
│    • 10-50 epochs（常规训练）                           │
│    • 100+ epochs（小数据集）                            │
│                                                          │
│  为什么不一直训练？                                      │
│    过多 epochs → 过拟合（模型记住训练数据，泛化差）     │
│                                                          │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│            3. Batch Size（批次大小）                      │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  定义：每次训练更新参数时使用的样本数量                  │
│                                                          │
│  例如：                                                  │
│    训练数据 10,000 条                                    │
│    batch_size = 32                                       │
│    → 每次拿 32 条数据训练，更新一次参数                 │
│    → 一个 epoch 需要 10,000 / 32 ≈ 313 次更新           │
│                                                          │
│  常见取值：                                              │
│    • 16-64（大模型，显存受限）                          │
│    • 128-256（中等模型）                                │
│    • 512+（小模型）                                     │
│                                                          │
│  权衡：                                                  │
│    • batch 大 → 训练快，但占显存多，可能收敛差          │
│    • batch 小 → 显存占用少，收敛稳定，但训练慢          │
│                                                          │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│            其他重要超参数                                 │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  Optimizer（优化器）：                                   │
│    • Adam（最常用）：自适应学习率                       │
│    • SGD：经典随机梯度下降                              │
│    • AdamW：Adam + 权重衰减                             │
│                                                          │
│  Weight Decay（权重衰减）：                              │
│    • 正则化技术，防止过拟合                             │
│    • 常见值：0.01, 0.001                                │
│                                                          │
│  Warmup Steps（预热步数）：                              │
│    • 开始训练时逐渐增加学习率                           │
│    • 防止初期梯度爆炸                                   │
│                                                          │
│  Dropout Rate（丢弃率）：                                │
│    • 训练时随机关闭部分神经元                           │
│    • 防止过拟合，常见值：0.1, 0.2                       │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**超参数调优策略**：

```
1. 网格搜索（Grid Search）：
   • 穷举所有组合
   • 例如：learning_rate = [1e-5, 1e-4, 1e-3]
           batch_size = [16, 32, 64]
   • 组合：3 × 3 = 9 次训练
   • 优点：全面
   • 缺点：慢

2. 随机搜索（Random Search）：
   • 随机采样超参数
   • 比网格搜索更高效

3. 贝叶斯优化（Bayesian Optimization）：
   • 根据之前的结果智能选择下一组超参数
   • 最高效，但实现复杂
```

#### Code Hash 详解

┌──────────────────────────────────────────────────────────┐
│            什么是 Code Hash？                             │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  定义：执行代码的哈希值（SHA-256）                       │
│                                                          │
│  目的：证明"Provider 运行了正确的代码"                   │
│                                                          │
│  类比：                                                  │
│    你下载了一个软件                                      │
│    官网提供 MD5 哈希：3a5f8c2d...                        │
│    你计算下载文件的 MD5                                  │
│    如果一致 → 文件未被篡改                              │
│                                                          │
│  在 0G Compute 中：                                      │
│    code_hash 证明 Provider 运行了"官方推理代码"         │
│    而不是恶意代码                                        │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**Code Hash 的生成与验证**：

```
┌──────────────────────────────────────────────────────────┐
│            Code Hash 完整流程                             │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  Step 1: 官方发布推理代码                                │
│  ─────────────────────────                               │
│    0G 团队发布官方推理代码：                             │
│    ```python                                             │
│    def inference(model, input):                          │
│        output = model.forward(input)                     │
│        return output                                     │
│    ```                                                   │
│                                                          │
│    计算代码哈希：                                        │
│    code_hash = SHA256(code)                              │
│             = "a3f5c8d9b2e1..."                          │
│                                                          │
│    公布：                                                │
│    • 发布在 GitHub                                      │
│    • 记录在 0G Chain 上                                 │
│                                                          │
│  Step 2: Provider 加载代码到 TEE                         │
│  ────────────────────────────────                        │
│    Provider 在 TEE 中：                                  │
│    1. 加载推理代码                                       │
│    2. TEE 自动计算 code_hash                             │
│    3. TEE 生成 attestation                               │
│                                                          │
│    attestation 包含：                                    │
│    {                                                     │
│      code_hash: "a3f5c8d9b2e1...",  // TEE 测量的       │
│      input_hash: "b4e2f3a1...",     // 用户输入哈希     │
│      output: "...",                 // 推理结果         │
│      TEE_signature: "..."           // TEE 签名         │
│    }                                                     │
│                                                          │
│  Step 3: 用户验证                                        │
│  ───────────────                                         │
│    用户收到 attestation 后：                             │
│    1. 检查 code_hash == "a3f5c8d9b2e1..."（官方哈希）   │
│       ✓ 一致 → Provider 运行了正确代码                  │
│       ✗ 不一致 → Provider 可能作恶，拒绝结果            │
│                                                          │
│    2. 验证 TEE_signature                                 │
│       ✓ 确保 attestation 来自真实 TEE                   │
│                                                          │
│    3. 信任结果                                           │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**为什么需要 Code Hash？**

```
没有 code_hash 的风险：
  • Provider 可以运行恶意代码
  • 例如：
    - 偷取用户输入（私密数据）
    - 返回错误结果
    - 跳过计算（省电费）

有 code_hash 的保证：
  • 用户确认 Provider 运行了"官方代码"
  • TEE 确保代码未被篡改
  • 结果可信
```

**实际例子**：

```
场景：用户要求推理 GPT-4 模型

恶意 Provider 尝试：
  • 跳过推理，返回随机文本（省算力）
  • 修改代码：
    def inference(model, input):
        return "random text"  # 作弊

  • 生成的 code_hash = "xyz123..."（不同于官方）

用户验证：
  • 检查 code_hash
  • 发现 "xyz123..." ≠ "a3f5c8d9..."（官方）
  • 拒绝结果，举报 Provider
  • Provider 失去质押

结果：
  • 恶意 Provider 无法作恶
  • 经济惩罚（质押损失）
```

#### 分布式训练协调

**多 GPU 训练架构**：

```
┌──────────────────────────────────────────────────────────┐
│            数据并行（Data Parallelism）                   │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  原理：每个 GPU 有完整模型副本，处理不同数据             │
│                                                          │
│  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐   │
│  │   GPU 1     │   │   GPU 2     │   │   GPU 3     │   │
│  │ 模型副本 A   │   │ 模型副本 B   │   │ 模型副本 C   │   │
│  │             │   │             │   │             │   │
│  │ 数据批次 1   │   │ 数据批次 2   │   │ 数据批次 3   │   │
│  │ loss_1      │   │ loss_2      │   │ loss_3      │   │
│  │ grad_1      │   │ grad_2      │   │ grad_3      │   │
│  └──────┬──────┘   └──────┬──────┘   └──────┬──────┘   │
│         │                 │                 │           │
│         └────────┬─────────┴────────┬────────┘           │
│                  ▼                  ▼                    │
│           梯度聚合（Gradient Aggregation）               │
│           avg_grad = (grad_1 + grad_2 + grad_3) / 3      │
│                  │                                       │
│                  ▼                                       │
│           所有 GPU 用 avg_grad 更新模型                  │
│                                                          │
│  优点：                                                  │
│    • 实现简单                                           │
│    • 训练速度提升 N 倍（N = GPU 数量）                  │
│                                                          │
│  缺点：                                                  │
│    • 每个 GPU 需要存储完整模型（显存受限）              │
│    • 不适合超大模型（如 GPT-3）                         │
│                                                          │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│            模型并行（Model Parallelism）                  │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  原理：大模型分片到多个 GPU                              │
│                                                          │
│  模型结构：                                              │
│  ┌───────────────────┐                                  │
│  │   Layer 1-10      │ → GPU 1                          │
│  ├───────────────────┤                                  │
│  │   Layer 11-20     │ → GPU 2                          │
│  ├───────────────────┤                                  │
│  │   Layer 21-30     │ → GPU 3                          │
│  └───────────────────┘                                  │
│                                                          │
│  前向传播：                                              │
│    input → GPU1(Layer1-10) → 中间结果1                  │
│         → GPU2(Layer11-20) → 中间结果2                  │
│         → GPU3(Layer21-30) → output                     │
│                                                          │
│  优点：                                                  │
│    • 可以训练超大模型                                   │
│    • 突破单卡显存限制                                   │
│                                                          │
│  缺点：                                                  │
│    • GPU 间通信频繁（网络开销大）                       │
│    • 实现复杂                                           │
│    • 需要高速互联（InfiniBand、NVLink）                │
│                                                          │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│            混合并行（Hybrid Parallelism）                 │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  = 数据并行 + 模型并行                                   │
│                                                          │
│  用于：超大模型（如 GPT-3、GPT-4）                       │
│                                                          │
│  示例：8 个 GPU 训练 GPT-3                               │
│    • 模型分 2 片（模型并行）                            │
│    • 每片用 4 个 GPU（数据并行）                        │
│                                                          │
│  ┌─────────────┬─────────────┐                          │
│  │  GPU 1-4    │  GPU 5-8    │                          │
│  │  Layer 1-50 │  Layer 51-96│                          │
│  │  (数据并行) │  (数据并行) │                          │
│  └─────────────┴─────────────┘                          │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**0G Compute 的分布式训练协调**：

```
┌──────────────────────────────────────────────────────────┐
│            0G Compute 训练协调流程                        │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  场景：用户要微调一个 10B 参数模型，需要 4 个 A100 GPU  │
│                                                          │
│  Step 1: Broker 资源分配                                 │
│  ────────────────────                                    │
│    • 查询可用 Provider：                                │
│      - Provider A：2 个 A100                            │
│      - Provider B：2 个 A100                            │
│      - Provider C：4 个 A100 ✓（选这个）                │
│                                                          │
│    • 分配任务给 Provider C                               │
│                                                          │
│  Step 2: Provider C 初始化                               │
│  ─────────────────────────                               │
│    • 从 0G Storage 下载：                                │
│      - 基础模型（llama-3-10b）                          │
│      - 训练数据                                         │
│                                                          │
│    • 在 TEE 中加载推理代码                               │
│      - 验证 code_hash                                    │
│                                                          │
│  Step 3: 多 GPU 训练（数据并行）                         │
│  ──────────────────────────────                          │
│    • GPU 1-4 各加载模型副本                              │
│    • 数据分片：                                          │
│      - GPU 1：batch 1-250                               │
│      - GPU 2：batch 251-500                             │
│      - GPU 3：batch 501-750                             │
│      - GPU 4：batch 751-1000                            │
│                                                          │
│    • 训练循环：                                          │
│      for epoch in range(3):                              │
│          for batch in batches:                           │
│              # 前向传播（各 GPU 独立）                  │
│              loss = model(batch)                         │
│                                                          │
│              # 反向传播（计算梯度）                      │
│              loss.backward()                             │
│                                                          │
│              # 梯度聚合（All-Reduce）                   │
│              all_reduce(gradients)                       │
│                                                          │
│              # 更新模型                                  │
│              optimizer.step()                            │
│                                                          │
│  Step 4: 梯度聚合（关键）                                │
│  ───────────────────────                                 │
│    使用高速网络同步梯度：                                │
│      • NVLink（GPU 内部互联，最快）                     │
│      • InfiniBand（跨节点，次快）                       │
│      • 以太网（慢，但便宜）                              │
│                                                          │
│    (可选) 梯度通过 0G DA 层验证：                        │
│      • 每隔 N 步，将梯度提交到 DA 层                    │
│      • 链上记录训练进度                                 │
│      • 防止 Provider 作弊                               │
│                                                          │
│  Step 5: 保存 Checkpoint                                 │
│  ──────────────────────                                  │
│    每个 epoch 结束：                                     │
│      • 保存模型参数到 0G Storage                        │
│      • 记录训练指标（loss, accuracy）                   │
│      • 生成 TEE 证明（证明诚实训练）                    │
│                                                          │
│  Step 6: 训练完成                                        │
│  ─────────────────                                       │
│    • 微调后模型上传到 0G Storage                        │
│    • 返回模型 ID 给用户                                 │
│    • 链上结算费用                                       │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**通信优化**：

```
梯度聚合通信模式：

1. Parameter Server（参数服务器）：
   ┌────────┐
   │ Server │ ← 收集所有梯度，聚合后分发
   └────────┘
   ↑ ↓ ↑ ↓ ↑ ↓
   GPU1 GPU2 GPU3

   缺点：Server 成为瓶颈

2. All-Reduce（全归约，推荐）：
   GPU1 ↔ GPU2 ↔ GPU3
    ↕      ↕      ↕
   环状通信，无瓶颈

   实现：Ring All-Reduce
   • 每个 GPU 只和邻居通信
   • 通信复杂度：O(N)，N = GPU 数量

3. 0G 创新：DA 层验证（可选）：
   • 关键梯度步骤提交到 DA 层
   • 生成训练证明
   • 链上验证训练过程
   • 防止 Provider 伪造训练
```

## 5.5 安全与信任

### 5.5.1 TEE 的局限性

**已知攻击**：

```
侧信道攻击（Side-Channel Attacks）：
  • 通过功耗、时间分析推断数据
  • 防御：算法层面的恒定时间操作

Spectre/Meltdown：
  • CPU 漏洞，可能泄露 TEE 数据
  • 防御：硬件补丁、软件缓解

物理攻击：
  • 拆解硬件、探测内存
  • 防御：物理安全（但成本极高）
```

**增强方案**：

```
多重验证：
  • TEE + ZKP（零知识证明）
  • 双重保证

声誉系统：
  • 跟踪 Provider 历史行为
  • 异常行为降权

保险机制：
  • 用户可购买"计算保险"
  • 如果 Provider 作恶，保险赔偿
```
