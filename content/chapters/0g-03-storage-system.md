# 3. 0G 存储系统：分层设计支持多样化应用

## 本章学习目标

在阅读本章之前，假设你已经了解：
- 第二章中 0G 的基本架构（存储网络、共识网络）
- 数据如何上传和存储到 0G

本章将详细解释：
- **是什么**：0G Storage 的三层架构是什么
- **为什么**：为什么需要分层设计
- **怎么做**：每一层如何工作，如何协同

## 关键概念

- **Log 层**：追加写入的永久存储层
- **Key-Value 层**：支持可变数据的存储层
- **事务处理层**：支持并发控制的高级层
- **日志播放（Log Replay）**：通过重放日志条目构建状态
- **冲突检测**：检测并发操作是否冲突

## 3.1 为什么需要分层设计？

### 不同应用的不同需求

**问题**：不同的去中心化应用需要不同的存储特性。

**例子 1：Layer 2 网络**
- 需求：永久存储交易数据，不需要修改
- 特性：追加写入、永久存储、可验证

**例子 2：去中心化社交网络**
- 需求：用户可以发布、修改、删除消息
- 特性：可变数据、访问控制、所有权管理

**例子 3：协作编辑（如 Google Docs）**
- 需求：多个用户同时编辑，需要处理冲突
- 特性：并发控制、事务处理、ACID 特性

**解决方案**：0G Storage 提供三层架构，每层解决不同的问题。

### 三层架构概览

```
┌─────────────────────────────────────┐
│  事务处理层 (Transactional)         │  ← 最复杂：并发控制、ACID
│  - 乐观并发控制                      │
│  - 冲突检测与解决                    │
└─────────────────────────────────────┘
           ↓ 基于
┌─────────────────────────────────────┐
│  Key-Value 层                        │  ← 中等：可变数据、访问控制
│  - Put() / Get() API                 │
│  - 所有权管理                        │
└─────────────────────────────────────┘
           ↓ 基于
┌─────────────────────────────────────┐
│  Log 层 (基础层)                     │  ← 最简单：永久存储、追加写入
│  - 去中心化存储网络                  │
│  - 激励机制                          │
└─────────────────────────────────────┘
```

**设计原则**：
- 底层提供基础功能（Log 层）
- 上层基于底层构建更复杂的功能（Key-Value 层、事务处理层）
- 应用可以选择使用哪一层（根据需求）

## 3.2 Log 层：永久存储的基础

### Log 层是什么？

**是什么**：Log 层是一个追加写入（append-only）的永久存储系统，数据只能追加，不能修改或删除。

**类比**：就像图书馆的档案系统
- 新书只能添加到书架末尾
- 不能删除或修改已有记录
- 但可以基于这些记录构建索引和目录

### 为什么需要 Log 层？

**原因 1：永久性**
- 某些数据需要永久保存（如 Layer 2 的交易历史）
- 一旦写入，永远不能删除或修改
- 保证数据的完整性和可追溯性

**原因 2：简单性**
- 追加写入是最简单的存储操作
- 不需要复杂的更新和删除逻辑
- 降低系统复杂度

**原因 3：为上层提供基础**
- Key-Value 层和事务处理层都基于 Log 层
- Log 层提供顺序性和永久性保证

### Log 层如何工作？

#### 步骤 1：数据追加到日志

**是什么**：当用户上传数据时，数据被追加到日志的末尾。

**怎么做**：

1. **用户上传数据**
   - 用户调用智能合约，提交数据
   - 数据包含：实际数据内容、元数据（大小、时间戳等）

2. **智能合约创建日志条目**
   - 智能合约分配一个唯一的日志位置（offset）
   - 第一个条目的 offset = 0
   - 第二个条目的 offset = 第一个条目的大小
   - 以此类推

3. **数据存储到网络**
   - 数据块被分散到存储节点
   - 存储节点存储数据块

**具体例子**：
```
日志位置（offset）：
0: 数据条目 1 (1 MB) - 时间戳: 2024-01-01 10:00:00
1MB: 数据条目 2 (500 KB) - 时间戳: 2024-01-01 10:00:05
1.5MB: 数据条目 3 (2 MB) - 时间戳: 2024-01-01 10:00:10
...
```

#### 步骤 2：存储节点维护日志

**是什么**：存储节点存储日志中的数据块，并维护日志的顺序。

**怎么做**：

1. **节点同步日志**
   - 节点定期检查智能合约，获取新的日志条目
   - 节点下载对应的数据块
   - 节点验证数据块的完整性（使用 Merkle root）

2. **节点组织数据**
   - 节点按照 offset 顺序组织数据
   - 节点维护一个索引，记录每个 offset 对应的数据块位置

3. **节点服务查询**
   - 当用户查询数据时，节点根据 offset 快速定位数据
   - 节点返回对应的数据块

**具体例子**：
```
存储节点 A 的本地索引：
offset 0 → 数据块位置: /storage/chunk_001.bin
offset 1MB → 数据块位置: /storage/chunk_002.bin
offset 1.5MB → 数据块位置: /storage/chunk_003.bin
...
```

#### 步骤 3：激励机制

**是什么**：存储节点通过存储数据获得奖励。

**怎么做**：
- 存储节点参与 PoRA 挖矿（详见第五章）
- 证明自己存储了数据
- 获得挖矿奖励

**为什么需要**：
- 鼓励节点存储数据
- 确保数据有足够的副本
- 平衡数据副本数量

### Log 层的限制

**限制 1：只能追加，不能修改**
- 一旦数据写入，就不能修改或删除
- 适合归档，不适合需要更新的应用

**限制 2：非结构化数据**
- Log 层存储的是原始数据块
- 没有结构化的查询接口（如"根据 key 查询 value"）
- 需要上层系统提供结构化接口

**解决方案**：在 Log 层之上构建 Key-Value 层。

## 3.3 Key-Value 层：可变数据的管理

### 为什么需要 Key-Value 层？

**问题**：Log 层只能追加，不能修改，但很多应用需要更新数据。

**例子**：去中心化社交网络
- 用户发布消息（创建）
- 用户修改消息（更新）
- 用户删除消息（删除）

**解决方案**：Key-Value 层在 Log 层之上提供可变数据接口。

### Key-Value 层是什么？

**是什么**：Key-Value 层提供类似字典的接口，可以通过 key 查询和更新 value。

**基本操作**：
- `Put(key, value)`：创建或更新键值对
- `Get(key)`：获取键对应的值

**类比**：就像 Python 的字典
```python
# 创建或更新
kv_store["user_123"] = {"name": "Alice", "posts": 10}

# 查询
user = kv_store["user_123"]
```

### Key-Value 层如何工作？

#### 核心机制：日志播放（Log Replay）

**是什么**：Key-Value 层通过"播放"（重放）Log 层的日志条目，构建当前的键值状态。

**什么是"日志播放"（Log Replay）**：
- 就像播放录音带一样，按照顺序"播放"日志中的每个操作
- 从第一个操作开始，逐个执行，最终得到当前状态
- 类似于"回放历史"，通过重放所有操作来重建状态

**为什么这样设计**：
- Log 层保证顺序性和永久性
- 通过播放日志，可以重建任何时刻的状态
- 所有节点播放相同的日志，得到相同的状态
- **关键优势**：即使节点重启，也可以通过播放日志重建状态，不需要单独存储状态

**怎么做**：

**步骤 1：更新操作写入日志**

当用户调用 `Put(key, value)` 时：

1. **构造日志条目**
   - Key-Value 层将更新操作封装成日志条目
   - 日志条目包含：key、value、操作类型（PUT）、时间戳

2. **追加到 Log 层**
   - 日志条目被追加到 Log 层
   - 获得一个唯一的 offset

3. **本地更新（可选）**
   - Key-Value 节点可以在本地立即更新状态
   - 但最终状态由日志决定

**具体例子**：
```
用户调用: Put("user_123", {"name": "Alice"})
↓
日志条目（offset 1000）:
{
  operation: "PUT",
  key: "user_123",
  value: {"name": "Alice"},
  timestamp: 2024-01-01 10:00:00
}
↓
追加到 Log 层
```

**步骤 2：节点播放日志构建状态**

**是什么**：Key-Value 节点通过播放日志条目，构建当前的键值状态。

**怎么做**：

1. **节点启动时**
   - 节点连接到 Log 层
   - 从 offset 0 开始播放所有日志条目
   - **播放过程**：就像执行一个程序，从第一条指令开始，逐步执行所有操作
   - 构建初始状态

2. **播放过程**
   - 对于每个日志条目：
     - 如果是 PUT 操作：更新 key 对应的 value（如果 key 已存在，覆盖；如果不存在，创建）
     - 如果是 DELETE 操作：删除 key
   - **关键理解**：后写入的操作会覆盖先写入的操作，所以最终状态是最后一次写入的结果
   - 最终得到当前状态

3. **持续同步**
   - 节点持续监听新的日志条目
   - 播放新条目，更新状态
   - **为什么需要持续同步**：因为新的操作会不断追加到日志，节点需要实时更新状态

**具体例子**：
```
初始状态: {}
↓ 播放 offset 0 (PUT "user_123" = {"name": "Alice"})
状态: {"user_123": {"name": "Alice"}}
↓ 播放 offset 1 (PUT "user_456" = {"name": "Bob"})
状态: {"user_123": {"name": "Alice"}, "user_456": {"name": "Bob"}}
↓ 播放 offset 2 (PUT "user_123" = {"name": "Alice", "posts": 10})
状态: {"user_123": {"name": "Alice", "posts": 10}, "user_456": {"name": "Bob"}}
```

**步骤 3：查询操作**

当用户调用 `Get(key)` 时：

1. **节点查询本地状态**
   - 节点已经通过播放日志构建了当前状态
   - 直接查询本地状态，返回对应的 value

2. **如果 key 不存在**
   - 返回 null 或错误

**具体例子**：
```
用户调用: Get("user_123")
↓
节点查询本地状态
↓
返回: {"name": "Alice", "posts": 10}
```

### 为什么所有节点状态一致？

**原因**：所有节点播放相同的日志，按照相同的顺序处理相同的操作。

**保证**：
- Log 层保证日志条目的顺序（通过 offset）
- 所有节点从相同的 offset 开始播放
- 所有节点按照相同的顺序处理操作
- 因此，所有节点得到相同的最终状态

**具体例子**：
```
节点 A: 播放 offset 0, 1, 2 → 状态 S
节点 B: 播放 offset 0, 1, 2 → 状态 S
节点 C: 播放 offset 0, 1, 2 → 状态 S

→ 所有节点状态一致
```

### 访问控制：所有权管理

**是什么**：Key-Value 层支持为每个 key 关联所有权信息，只有所有者可以更新。

**为什么需要**：
- 去中心化社交网络：只有消息所有者可以修改/删除
- 文件系统：只有文件所有者可以更新文件

**怎么做**：

1. **创建 key 时指定所有者**
   - 日志条目包含：key、value、owner（所有者地址）

2. **更新时验证所有权**
   - 用户调用 `Put(key, value)` 时，需要签名证明自己是所有者
   - Key-Value 节点验证签名
   - 只有所有者可以更新

3. **所有权信息存储在日志中**
   - 所有权信息作为日志条目的一部分
   - 所有节点播放日志时，都知道每个 key 的所有者

**具体例子**：
```
用户 Alice (地址: 0x123...) 创建消息:
Put("msg_001", "Hello", owner: 0x123...)
↓
日志条目包含所有权信息
↓
只有 Alice 可以更新/删除这个消息
```

### 性能优化

**优化 1：批量更新**
- `Put()` 允许一次更新多个键值对
- 减少日志条目数量
- 提高效率

**优化 2：选择性同步**
- 新节点可以跳过不相关的日志条目
- 只播放包含特定 stream ID 的条目
- **什么是 stream ID**：stream ID 是数据流的标识符。0G 支持主流（所有数据在一个流中）和专用流（每个应用有独立的流）。通过 stream ID，节点可以选择只同步特定应用的数据，加快启动速度
- 加快启动速度

## 3.4 事务处理层：并发控制与 ACID

### 为什么需要事务处理？

**问题**：多个用户可能同时更新同一个 key，需要保证数据一致性。

**例子**：协作编辑（Google Docs）
- 用户 A 和用户 B 同时编辑同一段落
- 用户 A 添加文字，用户 B 删除文字
- 需要检测冲突，保证最终一致性

**解决方案**：事务处理层提供乐观并发控制（Optimistic Concurrency Control）。

### 事务处理层是什么？

**是什么**：事务处理层在 Key-Value 层之上提供事务语义，支持多个操作的原子执行和并发控制。

**什么是 ACID 特性**：
- **A（Atomicity，原子性）**：事务中的所有操作要么全部成功，要么全部失败
- **C（Consistency，一致性）**：事务执行后，数据保持一致状态
- **I（Isolation，隔离性）**：并发事务互不干扰
- **D（Durability，持久性）**：事务提交后，更改永久保存

0G 的事务处理层通过乐观并发控制实现这些特性。

**基本操作**：
- `BeginTx()`：开始事务
- `Put(key, value)`：在事务中更新
- `Get(key)`：在事务中查询
- `EndTx()`：提交事务

**类比**：就像数据库的事务
```python
tx = BeginTx()
tx.Put("balance", 100)
tx.Put("balance", 150)  # 更新
balance = tx.Get("balance")  # 查询
EndTx(tx)  # 提交
```

### 事务处理层如何工作？

#### 步骤 1：开始事务

**是什么**：用户调用 `BeginTx()` 开始一个新事务。

**怎么做**：

1. **记录当前日志位置**
   - 事务开始时，记录当前日志的 offset（比如 offset 1000）
   - 这个 offset 是事务的"起始点"

2. **构建状态快照**
   - 通过播放日志到 offset 1000，构建当前状态快照
   - **什么是状态快照**：状态快照是某个时刻（offset 1000）所有键值对的完整状态
   - **为什么需要快照**：事务需要基于一个确定的状态开始工作，不能基于正在变化的状态
   - **类比**：就像拍照，记录某个时刻的状态，然后基于这张"照片"工作
   - 事务基于这个快照工作

3. **创建本地缓冲区**
   - 事务的更新操作存储在本地缓冲区
   - 不立即写入日志

**具体例子**：
```
当前日志 offset: 1000
用户调用: BeginTx()
↓
事务开始位置: offset 1000
状态快照: 播放日志到 offset 1000
本地缓冲区: {}
```

#### 步骤 2：执行操作

**是什么**：在事务中执行更新和查询操作。

**怎么做**：

1. **更新操作（Put）**
   - 更新存储在本地缓冲区
   - 不立即写入日志
   - 其他节点看不到这些更新

2. **查询操作（Get）**
   - 先查询本地缓冲区（事务中的更新）
   - 如果不存在，查询状态快照
   - 返回结果

**具体例子**：
```
事务中:
tx.Put("balance", 100)  → 本地缓冲区: {"balance": 100}
tx.Put("balance", 150)  → 本地缓冲区: {"balance": 150}
balance = tx.Get("balance")  → 返回 150（从本地缓冲区）
```

#### 步骤 3：提交事务

**是什么**：用户调用 `EndTx()` 提交事务。

**怎么做**：

1. **构造提交记录**
   - 提交记录包含：
     - 事务开始位置（offset 1000）
     - 读写集（read-write set）：
       - **读集（read set）**：事务中读取了哪些 keys（如 "balance"）
       - **写集（write set）**：事务中写入了哪些 keys（如 "balance"）和对应的 values
     - **为什么需要读写集**：用于冲突检测。如果其他事务在冲突窗口中更新了读集中的 keys，就会发生冲突

2. **追加到日志**
   - 提交记录被追加到日志
   - 获得一个新的 offset（比如 offset 1001）

**具体例子**：
```
提交记录（offset 1001）:
{
  startOffset: 1000,
  readSet: ["balance"],
  writeSet: {"balance": 150}
}
↓
追加到日志
```

#### 步骤 4：冲突检测

**是什么**：其他节点播放日志时，检测事务是否与其他操作冲突。

**怎么做**：

1. **识别冲突窗口**
   - **什么是冲突窗口**：冲突窗口是事务开始位置到提交位置之间的所有日志条目
   - 在这个例子中：offset 1000 到 1001 之间的所有条目
   - **为什么叫"窗口"**：就像时间窗口，在这个时间段内的操作可能与事务冲突
   - **关键理解**：如果在这个窗口内有其他操作更新了事务读取的 keys，就会发生冲突

2. **检查冲突**
   - 检查冲突窗口中的操作是否更新了事务读取的 keys
   - 如果冲突窗口中有操作更新了 "balance"
   - 且事务也读取了 "balance"
   - → 冲突！

3. **处理冲突**
   - 如果有冲突：事务中止，不应用更新
   - 如果没有冲突：事务提交，应用更新

**具体例子**：

**场景 1：无冲突**
```
日志（每条占一个 offset；只有「提交记录」会写入日志，「事务开始」不占条目）:
offset 1000: 某条先前的日志（该事务的 startOffset = 1000）
offset 1001: 其他操作 PUT("name", "Alice")  ← 不冲突（不同的 key）
offset 1002: 本事务的提交记录（startOffset: 1000, readSet: ["balance"], writeSet: {"balance": 150}）

冲突窗口: offset 1000～1002 之间的所有条目
检查: 冲突窗口中没有操作更新 "balance"
结果: 事务提交成功，writeSet 被应用
```

**场景 2：有冲突**
```
日志:
offset 1000: 某条先前的日志（该事务的 startOffset = 1000）
offset 1001: 其他操作 PUT("balance", 200)  ← 冲突！（更新了事务读取的 key）
offset 1002: 本事务的提交记录（startOffset: 1000, readSet: ["balance"], writeSet: {"balance": 150}）

冲突窗口: offset 1000～1002
检查: 冲突窗口中有操作更新了 "balance"
结果: 事务中止，本事务的 writeSet 不应用（offset 1002 的这条记录仍留在日志中，但不对状态产生效果）
```

### 为什么叫"乐观"并发控制？

**是什么**：乐观并发控制假设冲突很少发生，先执行操作，再检测冲突。

**为什么"乐观"**：
- 假设大多数情况下不会有冲突
- 不需要锁定资源
- 只有在提交时检测冲突

**对比悲观并发控制**：
- 悲观：先锁定资源，再执行操作（如数据库的行锁）
- 乐观：先执行操作，再检测冲突（0G 的方式）

---

### 常见疑问：事务失败时是否「浪费」？实现上怎么做？冲突后谁重试？

**1. 事务失败（例如发生冲突）时，这次操作是不是就浪费了？**

- **从「是否生效」看**：是的。冲突时该事务的 **writeSet 不会应用到全局状态**，相当于这次提交在状态层面「白做了」。
- **从「日志与 offset」看**：不会丢一条「位置」。**提交记录**（startOffset、readSet、writeSet）仍会**追加到日志末尾**并占有一个新的 offset。这样：(a) 全网的顺序一致，(b) 人人都能看到「有一次以 startOffset/readSet/writeSet 的提交尝试」，(c) 冲突检测有统一依据。只是回放时大家一致决定**不应用**这份 writeSet，所以这次尝试在状态上等价于「 abort 」。
- **从「用户/应用」看**：这次尝试里在本地缓冲区算出的那些更新、基于「旧快照」做的逻辑，在冲突时都不会进入全局状态，相当于**这次尝试被浪费**；需要靠**重试**（见下）再走一遍。

**2. 实现细节两问**

**Q1：定义/提交事务时，是把事务追加到日志最后并单独占一个 offset 吗？**

- **BeginTx()** 时**不**往日志里写任何新条目，也不新占 offset；只是记录「当前日志尾巴」的 offset，作为该事务的 **startOffset**。
- **EndTx()** 时，把**提交记录**（startOffset、readSet、writeSet）**追加到当前日志末尾**，并得到**一个新的 offset**。这个 offset 就是「这条提交记录」在逻辑日志里的位置；事务在日志里的「身份」就是这一条提交记录 + 其 offset。
- 因此：**每个提交尝试**对应日志里**一条**提交记录、**一个** offset；不存在「事务开始」再单独占一个 offset。

**Q2：事务是不是先进缓冲区，到真正 commit 才发现冲突？**

- **是**。事务内的 Put/Get 只改本地缓冲、基于状态快照算，**不**立刻写进日志。
- 调用 **EndTx()** 时，本机把提交记录追加到日志（此时还没做冲突检测）。  
- **冲突是在「回放」时发现的**：任一节点（包括发起方）在**按顺序回放日志**、处理到这条提交记录时，根据 startOffset 与当前记录的 offset 划出冲突窗口，再根据 readSet 与窗口中其它操作做冲突检测。若冲突，则**不把该记录的 writeSet 应用到本机状态**。
- 所以：**先写提交记录、占 offset，后在各节点的回放过程中发现冲突并决定「不应用」**；冲突不是「在 EndTx 那一刻在提交方本地算出来的」，而是由**所有节点在重放同一份日志时一致地**按规则判定。

**3. 真的发生冲突时，系统怎么处理？**

- **提交记录**：保留在日志中，占用的 offset 不变；所有人看到的顺序和内容一致。
- **状态**：该条提交记录的 **writeSet 不应用**；全局状态如同「这笔事务没发生过」。
- **对用户/应用**：需要通过 API 或上层协议知道「该事务已 abort」（例如读到的状态里没有自己的写入，或收到显式 abort 返回值），然后自行决定是否重试。

**4. 在什么情况下会「重新执行」？谁负责重试？**

- **系统不会自动重新执行同一笔事务**。所谓「重试」是指**用户/应用**自己发起的一次**新**事务：
  - 再调一次 **BeginTx()**（会拿到新的 startOffset，基于**当前**日志尾巴），
  - 再按业务逻辑做一遍 Put/Get（在新的本地缓冲里），
  - 再 **EndTx()**，得到一条**新的**提交记录、**新的** offset、新的冲突窗口。
- 这次新事务基于的是「已经不包含上次失败写集」的当前状态，因此逻辑上等价于「在上次失败后重做一遍」。

**5. 重试时，之前在本地的状态会不会都浪费掉？**

- **会**。上一次尝试里在本地缓冲区做的更新、基于旧快照算出的中间结果，都**不会**进入全局状态；重试时也不会被复用。
- 重试 = 新事务、新快照、新缓冲区，**从当前全局状态重新来过**。乐观并发的代价就是：冲突时，**这一次尝试的计算与本地状态都浪费掉**，只能依赖应用层重试；因此适合「冲突较少」的场景（例如协作编辑里冲突概率不高）。

### 安全假设

**协作模型**：
- 假设事务参与者是协作的
- 会诚实地在提交记录中填写正确的读写集

**适用场景**：
- Google Docs：用户通常信任共享访问的其他人
- 协作应用：参与者通常是可信的

**不可信场景**：
- 可以将事务代码存储在 0G 日志中
- 使用可验证计算机制：
  - 零知识证明（ZKP）
  - 可信执行环境（TEE）
- 验证提交记录的有效性

## 3.5 三层架构如何协同工作？

### 数据流示例

让我们看一个完整的例子：用户在去中心化社交网络上发布并修改消息。

**步骤 1：发布消息（使用 Key-Value 层）**
```
用户调用: Put("msg_001", "Hello", owner: 0x123...)
↓
Key-Value 层构造日志条目
↓
追加到 Log 层（offset 5000）
↓
所有 Key-Value 节点播放日志，更新状态
```

**步骤 2：修改消息（使用 Key-Value 层）**
```
用户调用: Put("msg_001", "Hello World", owner: 0x123...)
↓
Key-Value 层验证所有权（只有所有者可以更新）
↓
构造日志条目
↓
追加到 Log 层（offset 5001）
↓
所有节点播放日志，更新状态
```

**步骤 3：协作编辑（使用事务处理层）**
```
用户 A: BeginTx()
用户 A: Put("doc_001", "Version 1")
用户 A: EndTx()  → 提交（offset 5002）

用户 B: BeginTx()  → 开始位置: offset 5002
用户 B: Put("doc_001", "Version 2")
用户 B: EndTx()  → 提交（offset 5003）

冲突检测:
- 用户 B 的事务读取了 "doc_001"
- 冲突窗口（offset 5002-5003）中没有其他操作更新 "doc_001"
- 结果: 用户 B 的事务提交成功
```

### 应用如何选择使用哪一层？

**简单存储**：只用 Log 层
- Layer 2 交易数据
- 数据归档
- 不需要更新

**可变数据**：使用 Key-Value 层
- 去中心化社交网络
- 文件系统
- 需要更新，但不需要并发控制

**复杂应用**：使用事务处理层
- 协作编辑
- 数据库工作负载
- 需要并发控制和事务语义

## 总结

0G Storage 的分层设计提供了：

1. **Log 层（基础层）**：
   - 永久存储，追加写入
   - 为上层提供顺序性和永久性保证

2. **Key-Value 层（中间层）**：
   - 可变数据，访问控制
   - 通过日志播放构建状态

3. **事务处理层（高级层）**：
   - 并发控制，ACID 特性
   - 通过乐观并发控制处理冲突

这种设计使得 0G Storage 能够支持从简单到复杂的各种去中心化应用，同时保持高性能和可扩展性。
